## 0. FOUNDATIONAL CONSTRAINTS

Before defining any mechanism, we must state the non-negotiable constraints that govern every genetic operation in the colony. These derive from the Stress Test Analysis and the Colony Thesis, and they are not subject to optimization or relaxation.

**Constraint 0.1 — The Hard Boundary.** Evolution modifies ONLY bytecode. The C firmware (HAL, VM, safety system, RTOS), the bootloader, and the partition table are signed with RSA-3072 (secure boot v2) and encrypted with AES-XTS-256 (flash encryption). No genetic operator can produce a variant that modifies the safety ISRs, the watchdog, the output clamping logic, or the state machine transitions. The fitness function, safety thresholds, and partition table are "below" the evolutionary process — they are Gye Nyame, the untouchable.

**Constraint 0.2 — The VM is the Sandbox.** All bytecodes execute inside the Reflex VM, which enforces: no out-of-bounds memory access, no infinite loops (cycle budget: 1000µs per tick at 100Hz, or 100µs per tick at 1kHz), no unsafe actuator outputs (post-execution clamping per channel), no runaway recursion (call depth limit = 16), and fail-safe semantics on any violation (VM halts, actuators go to configured safe positions). This means "lethal mutations" cost nothing — the validator catches them at load time before execution.

**Constraint 0.3 — Deterministic Reproduction.** The same bytecode binary, given the same sensor register inputs and the same initial PID/variable state, produces identical actuator register outputs in the same number of VM cycles, every tick, on every supported MCU. This is the definition of the Heraclitean Identity: a bytecode's identity IS its deterministic input-output behavior, not its source code.

**Constraint 0.4 — Flash Economics.** ESP32-S3 flash endurance is ~100K erase cycles per sector. The 1MB `reflex_bc` LittleFS partition stores bytecodes. At 4KB per erase sector and typical bytecode sizes of 2–20KB (average ~8KB), writing a new bytecode consumes 2–5 erase sectors. At 1 evolution/day per node, this is 365–1825 sector erases/year. The partition lasts 55–274 years. This is sufficient. Version history beyond the last 2–3 local copies lives on the Jetson NVMe.

**Constraint 0.5 — Seasonal Mandate.** The colony's evolutionary process MUST oscillate through Spring (exploration), Summer (exploitation), Autumn (consolidation), and Winter (rest). No variant is generated during Winter. This is non-overridable. Any attempt to disable Winter triggers the same safety override as disabling the hardware watchdog.

## 1. MUTATION OPERATORS

The colony defines four mutation levels, each operating on a different granularity of the bytecode genome. Each level has different safety requirements, different validation costs, and different approval pathways. These levels map directly to the temporal stratification of the promotion pipeline: Level 1 requires <100ms Lyapunov validation; Level 2 requires <10s SMT-based verification; Level 3 requires human review; Level 4 requires human physical work.

### 1.1 Level 1 — Parameter Mutation

**What changes:** The three PID gain constants (Kp, Ki, Kd) and the two output bounds (output_min, output_max) in the `pid_state_t` structure. Each PID controller is 32 bytes; the mutable fields are at offsets +0, +4, +8, +24, +28. The integral_limit at offset +20 may also be mutated. These are the ONLY fields in a Level 1 mutation.

**Mutation method — Hybrid Gradient Descent + Bayesian Optimization:**

Level 1 mutations are generated by a hybrid approach that combines the speed of gradient descent with the robustness of Bayesian optimization for non-convex landscapes.

```
Algorithm: generate_level1_variant(parent_bytecode, telemetry_history)
    Input: parent bytecode binary, last N ticks of telemetry (N=1000-10000)
    Output: child bytecode binary with modified PID gains

    1. IDENTIFY the active PID_COMPUTE syscalls in the bytecode.
       Scan for opcode=0x00 with flags=0x80, operand1=0x02.
       Extract pid_idx from operand2 & 0xFFFF.

    2. EXTRACT current gains from the parent bytecode's embedded
       configuration block (first 64 bytes, defined by reflex compiler).
       Current: Kp, Ki, Kd, integral_limit, output_min, output_max.

    3. COMPUTE gradient estimate from telemetry:
       error_signal = [setpoint_i - actual_i for i in telemetry_window]
       For each gain g in {Kp, Ki, Kd}:
           delta_g = -learning_rate * partial_derivative(
               sum(error_signal^2), g)
       learning_rate = 0.001 * (1.0 / (1.0 + generation_number * 0.01))
       This is a 10% gradient step, decaying over generations.

    4. SAMPLE from Bayesian posterior:
       If a Gaussian Process surrogate model is available (maintained
       on Jetson from previous evaluations):
           g_proposed = gp_sample_mean + gp_sample_std * N(0,1)
       Else:
           g_proposed = g_current + delta_g + N(0, sigma)
           sigma = 0.05 * abs(g_current)  // 5% relative noise

    5. CLAMP gains to safe bounds:
       Kp: [0.0, 100.0]
       Ki: [0.0, 50.0]
       Kd: [0.0, 10.0]
       integral_limit: [0.0, Kp * 100]
       output_min, output_max: [actuator_min, actuator_max]

    6. GENERATE child bytecode:
       Copy parent bytecode.
       Overwrite PID configuration block with new gains.
       Compute new SHA-256 content hash.
       Attach metadata header (see Section 2).

    7. VALIDATE: Run Lyapunov stability check (see Section 1.1.1).
       If unstable, reject and retry from step 4.
       Max retries: 3. If all fail, return parent unchanged.
```

**Step sizes:** The gradient step is 0.1% of the current gain value per generation, decaying as `0.001 / (1 + 0.01 * gen)`. The Bayesian noise term is 5% of the current gain value (one standard deviation). These are conservative: the colony prioritizes stability over speed of adaptation. In Spring, the noise term doubles (10%) to encourage exploration. In Summer, it halves (2.5%) to exploit known-good regions.

**Search space constraints:** Gains are bounded to physically meaningful ranges. The bounds are stored in the bytecode's configuration header and are themselves non-evolvable (they are part of the reflex definition authored by the human engineer or AI model). The colony can only tune within these bounds. Widening the bounds requires Level 2+ mutation.

**Lyapunov stability certificate (Level 1):** The Jetson computes a Lyapunov certificate for every Level 1 variant before deployment. The process:

1. Linearize the closed-loop system around the current operating point using the plant model (known from commissioning data).
2. Solve the continuous-time algebraic Riccati equation: `A'P + PA - PBR^{-1}B'P + Q = 0`.
3. Check that P is positive definite (all eigenvalues > 0).
4. Verify that `dV/dt = x'(PA + A'P)x < 0` for all x in the operating envelope.
5. If the check passes, attach the certificate (P matrix, eigenvalues, operating point) to the variant's metadata.

**Computation time:** <100ms on Jetson Orin NX for a SISO PID loop. For MIMO systems (up to 4 PID loops interacting), <500ms. This is well within the budget for Level 1 validation.

**Unsolved problem:** The Lyapunov certificate is computed for a linearized model. If the actual plant has significant nonlinearity (e.g., rudder stall beyond 35 degrees, bilge pump cavitation), the linearized certificate may not guarantee stability in the nonlinear regime. The colony mitigates this by maintaining separate certificates for different operating regimes (see Section 5), but a rigorous nonlinear certificate remains an open research problem.

### 1.2 Level 2 — Conditional Mutation

**What changes:** The bytecode's control flow structure — specifically, JUMP_IF_FALSE (0x1E) and JUMP_IF_TRUE (0x1F) instructions and the comparison operations (EQ_F 0x11, LT_F 0x12, GT_F 0x13, LTE_F 0x14, GTE_F 0x15) that feed them. A conditional mutation can: (a) add a new condition-action pair (insert comparison + branch + action block), (b) modify an existing condition (change the comparison opcode or its operands), (c) remove a condition-action pair (replace branch with NOP), or (d) change the branch target of an existing condition.

**How the AI proposes new condition-action pairs:**

The Jetson's code-synthesis model (the Demiourgos) proposes Level 2 mutations by analyzing telemetry patterns. The process:

1. **Anomaly detection.** The Jetson runs statistical process control on the incoming telemetry stream. When a metric deviates by >2σ from its rolling baseline for >10 consecutive ticks, an "anomaly event" is generated.
2. **Root cause hypothesis.** The AI model generates a natural-language hypothesis: "When wave_height > 2.5m AND heading_error > 15°, the current PID gains produce oscillation with period ~4s and amplitude ±5°."
3. **Condition-action synthesis.** The model proposes a bytecode modification: add a branch that, when wave_height > 2.5m AND heading_error > 15°, switches to a more conservative set of PID gains (reduced Kp, increased Kd). This is expressed as a JSON reflex fragment.
4. **Compilation to bytecode.** The reflex compiler on the Jetson compiles the JSON fragment into the 32-opcode ISA, producing a block of 8-byte instructions. The block is structurally validated (all jumps target valid 8-byte-aligned offsets within the bytecode, stack depth never exceeds 256, call depth never exceeds 16).
5. **SMT verification (Level 2).** Before deployment, the condition-action block is verified using Z3 (SMT solver) against a bounded model of the VM state space. The verification checks:
   - The condition is satisfiable (it CAN be true under some input combination).
   - The action block terminates within the cycle budget (worst-case execution time ≤ 800µs, leaving 200µs margin for the 1000µs tick).
   - No stack overflow or underflow is possible through any execution path.
   - Output values remain within the safety guard's clamping range.
   
   **Computation time:** <10s for simple conditions (1–2 comparisons, 1 branch). Up to 60s for complex conditions (3+ comparisons, nested branches, multiple state transitions). The Jetson runs this as a background task.

**How conditions are tested for safety before deployment:**

A Level 2 variant undergoes a three-stage safety pipeline:

1. **Static analysis (instant, <1s):** The bytecode validator checks structural integrity — valid opcodes, aligned jump targets, bounded stack depth, no cycles in the call graph, valid sensor/actuator indices, valid variable indices.
2. **Formal verification (seconds to minutes):** The Z3 bounded model checker verifies the condition-action logic against the VM's safety invariants. If the model checker finds a violation, the variant is rejected with a counterexample.
3. **Shadow execution (hours to days):** The variant runs in parallel with the production bytecode in a shadow VM instance. Both receive the same sensor inputs. The shadow's actuator outputs are compared to the production's. If the shadow ever produces an output that differs from production by more than a configurable threshold (default: 10% of the actuator's full range), an alert is generated. The shadow runs for a minimum of 1000 ticks under varied conditions before the variant is eligible for promotion.

### 1.3 Level 3 — Algorithm Mutation

**What changes:** An entire control strategy is replaced. For example, a PID controller is replaced by a state-machine-based controller, or a feedforward + feedback architecture replaces a pure feedback architecture. This is the most dangerous mutation level because it changes the fundamental structure of the control loop, not just its parameters or branching logic.

**How the system replaces an entire control strategy:**

Level 3 mutations are proposed by the AI model and undergo the most rigorous validation:

1. **AI proposal.** The model generates a complete replacement bytecode (2–20KB), justified by a natural-language explanation of why the new strategy is expected to outperform the current one. The explanation must reference specific telemetry patterns and quantitative performance metrics.
2. **Full simulation on Jetson.** The replacement bytecode is run in the Jetson's VM simulator against a plant model derived from commissioning data. The simulation runs for the equivalent of 24 hours of real-time operation (864,000 ticks at 100Hz), covering the full range of expected operating conditions.
3. **Cross-validation against historical data.** The bytecode is replayed against the last 30 days of historical telemetry. Performance metrics (RMS error, actuator effort, energy consumption) are computed for each day and compared to the production bytecode's actual performance on those same days.
4. **Monte Carlo stress testing.** The bytecode is run with 10,000 random perturbations of the plant model parameters (±20% variation in all coefficients). The percentage of perturbations that produce stable behavior must exceed 95%.
5. **Human review.** A human operator reviews: the AI's justification, the simulation results, the cross-validation report, and the Monte Carlo stress test. The human can approve, reject, or request modifications.

**Rollback mechanism for failed algorithms:**

Every Level 3 deployment is accompanied by a mandatory rollback plan:

- The parent bytecode is preserved in the `reflex_bc` LittleFS partition as `reflex_<name>_rollback.rbc`.
- A countdown timer is started (configurable, default: 72 hours).
- If the operator does not explicitly confirm the deployment within the countdown period, the system automatically rolls back to the parent bytecode.
- If ANY of the following occurs during the countdown: (a) the VM halts >3 times in 100 ticks, (b) any safety metric regresses by >20% relative to the parent, (c) the operator sends a manual rollback command, the system rolls back immediately.
- The rollback transfer takes <1 second: load parent bytecode from LittleFS, validate, reinitialize VM.

### 1.4 Level 4 — Architecture Mutation

**What changes:** The physical hardware configuration — new sensors, new actuators, modified wiring, changed pin mappings, new communication buses. This is not a bytecode mutation at all; it is a hardware modification that requires the bytecode genome to adapt to a new environment.

**Human involvement requirement:**

Level 4 mutations CANNOT be initiated by the colony. They can only be REQUESTED by the colony and APPROVED and EXECUTED by a human operator. The request mechanism:

1. **Deficiency detection.** The colony's fitness monitoring detects that the current architecture fundamentally limits performance. For example: "The bilge pump bytecode cannot distinguish between normal bilge accumulation and a leak because there is no flow-rate sensor, only a level sensor. Adding a flow-rate sensor would enable leak detection."
2. **Architectural change proposal.** The AI model generates a structured proposal: what sensor/actuator to add, where to connect it, what pin mapping to use, what reflex definition to create, and what fitness improvement is expected.
3. **Human evaluation.** The operator evaluates the proposal for: physical feasibility, cost, installation effort, and risk. The operator can approve, reject, or modify the proposal.
4. **Physical installation.** The operator (or a technician) physically installs the new hardware, updates the pin configuration in NVS, and reboots the node.
5. **Genome adaptation.** After the reboot, the node reports its new hardware configuration to the Jetson. The Jetson generates a new reflex definition for the modified hardware and begins a Level 3 mutation cycle to evolve bytecodes that exploit the new capability.

**What the system CANNOT do at Level 4:** The system cannot physically modify hardware. It cannot solder wires, install sensors, or reconfigure the PCB. It can only request and propose. This is the constitutional constraint in its most literal form: the colony's evolutionary freedom ends at the silicon boundary.

## 2. THE GENETIC REPRESENTATION

### 2.1 Genotype: The Bytecode Binary

The genotype is the raw bytecode binary stored in the LittleFS `reflex_bc` partition on the ESP32. It is a flat array of 8-byte instructions, ranging from 256 bytes (32 instructions) to 20,480 bytes (2560 instructions). The binary includes:

```
Bytecode Binary Layout:
    Offset 0x00:  Metadata Header (64 bytes, see Section 2.3)
    Offset 0x40:  Configuration Block (64 bytes)
                  - PID gain tuples (Kp, Ki, Kd per controller)
                  - Output bounds per controller
                  - Sensor/actuator mapping table
                  - Calibration constants
    Offset 0x80:  Instruction Array (variable length, 8-byte aligned)
                  - Sequence of 8-byte instructions
                  - Terminated by HALT sentinel (NOP with flags=0x80)
    Offset N:     Padding to 8-byte alignment (if needed)
```

The content hash is SHA-256 of the instruction array (excluding the metadata header and configuration block, which may change without altering the bytecode's control logic).

### 2.2 Phenotype: Runtime Behavior

The phenotype is NOT the bytecode. The phenotype is the bytecode's behavioral fingerprint — a compressed representation of its input-output behavior across a standardized test suite. This is the Heraclitean Identity from the Colony Thesis: identity is behavioral pattern, not code at a moment.

**Behavioral fingerprinting algorithm:**

```
Algorithm: compute_behavioral_fingerprint(bytecode)
    Input: bytecode binary
    Output: 128-byte fingerprint vector

    1. INITIALIZE the Jetson VM simulator with the bytecode.
    2. FOR each of 32 standardized test scenarios:
           (sine wave setpoint, step input, ramp, impulse,
            noise, multi-frequency, edge cases)
           a. Run the bytecode for 10,000 ticks at 100Hz.
           b. Record the output time series (actuator commands).
           c. Compute the following features from the output:
              - Mean output value (4 bytes)
              - RMS output value (4 bytes)
              - Peak output value (4 bytes)
              - Dominant frequency via FFT (4 bytes)
              - Phase margin estimate (4 bytes)
              - Rise time / settling time (4 bytes)
              - Number of state transitions (4 bytes)
           d. Append the 28-byte feature vector to the fingerprint.
    3. NORMALIZE the fingerprint to unit variance per feature dimension.
    4. RETURN the 128-byte vector (32 scenarios × 4 features each).
```

The fingerprint is stored in the variant's metadata on the Jetson. It is used for:
- **Similarity search:** "Find variants with fingerprints within Euclidean distance D of target F."
- **Anomaly detection:** "This variant's fingerprint deviates by >3σ from its parent's fingerprint. Investigate."
- **Cluster analysis:** "These 5 variants cluster together. They likely solve the same sub-problem."

### 2.3 Metadata Header

Every bytecode variant carries a 64-byte metadata header that travels with the binary from generation to generation:

```c
typedef struct {
    uint32_t magic;              // 0x4E45585A ("NEXU")
    uint8_t  version;            // Metadata format version (currently 1)
    uint8_t  mutation_level;     // 1=Parameter, 2=Conditional, 3=Algorithm
    uint8_t  generation;         // Generation number (0-255, wraps)
    uint8_t  season;             // 0=Spring, 1=Summer, 2=Autumn, 3=Winter
    uint32_t parent_hash[2];     // SHA-256 of parent (first 8 bytes)
    uint32_t parent2_hash[2];    // SHA-256 of second parent if crossover (0 if asexual)
    uint32_t content_hash[8];    // SHA-256 of instruction array (32 bytes)
    float    fitness_score;      // Most recent fitness evaluation
    float    kolmogorov_fitness; // behavioral_score / compressed_size
    float    stability_eigenvalue_min; // Minimum eigenvalue of Lyapunov P matrix
    uint16_t tick_count_tested;  // Number of ticks this variant has been tested
    uint16_t bytecode_size;      // Size of instruction array in bytes
    uint8_t  environment_hash[8];// Hash of environmental conditions during testing
    uint8_t  node_id;            // ESP32 node that generated/last ran this variant
    uint8_t  reserved[8];        // Future use
} bytecode_metadata_t;  // 64 bytes
```

The `environment_hash` is computed from a summary of the environmental conditions during testing: mean and std of key sensor values (wave height, wind speed, water temperature, heading error, vessel speed). This allows the system to compare variants that were tested under similar vs. different conditions.

## 3. SELECTION MECHANISMS

### 3.1 Tournament Selection with Epsilon-Greedy Exploration

The colony uses tournament selection with an epsilon-greedy exploration component to balance exploitation (testing variants likely to be good) with exploration (testing variants that might be good).

**The variant pool.** At any time, a node maintains a pool of up to 5 candidate bytecodes for a given functional niche (e.g., rudder control). One is the production variant (currently driving actuators). Up to 4 are candidates under evaluation. The pool size is limited by the ESP32's LittleFS storage: 5 bytecodes × 20KB max = 100KB, well within the 1MB partition.

**Tournament selection process:**

```
Algorithm: select_variants_for_testing(node_id, niche)
    Input: node ID, functional niche
    Output: list of variants to test

    pool = get_variant_pool(node_id, niche)  // up to 5 variants
    production = pool[0]  // currently active variant

    // Determine testing allocation
    IF seasonal_phase == SPRING:
        exploration_rate = 0.30  // 30% of traffic to exploration
        tournament_size = 2      // small tournaments = more diversity
    ELIF seasonal_phase == SUMMER:
        exploration_rate = 0.10  // 10% exploration
        tournament_size = 3      // larger tournaments = exploit known-good
    ELIF seasonal_phase == AUTUMN:
        exploration_rate = 0.05
        tournament_size = 4
    ELSE:  // WINTER
        exploration_rate = 0.0   // NO EXPLORATION DURING WINTER
        RETURN [production]      // Only production variant runs

    // Generate new candidate (if room in pool)
    IF len(pool) < 5 AND random() < exploration_rate:
        candidate = generate_variant(production, telemetry_history)
        pool.append(candidate)
        write_to_littlefs(candidate)

    // Tournament: pick best from random subset
    IF random() < exploration_rate:
        // Epsilon-greedy: pick a random candidate
        test_variant = random_choice(pool[1:])  // exclude production
    ELSE:
        // Tournament: pick best from k random candidates
        tournament = random_sample(pool, tournament_size)
        test_variant = max(tournament, key=lambda v: v.fitness_score)

    RETURN [production, test_variant]
```

### 3.2 Traffic Splitting and A/B Testing

**How many variants are tested simultaneously?** Up to 2: the production variant and one candidate. The candidate receives a configurable fraction of ticks (not time-based traffic splitting). At each tick:

- With probability `1 - test_fraction`: run production variant, drive actuators with its output.
- With probability `test_fraction`: run candidate variant in a shadow VM, compare its output to what production would have produced, but DO NOT drive actuators with the candidate's output. Log the comparison.

**Safety-critical splitting:** For safety-critical niches (rudder, throttle), `test_fraction = 0.05` (5% of ticks shadow-tested). The candidate never drives actuators until it has accumulated 10,000+ shadow ticks with no anomalies.

**Non-critical splitting:** For non-critical niches (bilge pump, lighting), `test_fraction = 0.20` (20% of ticks shadow-tested). After 5,000+ clean shadow ticks, the candidate may be promoted to live testing where it drives actuators 10% of the time (with production output as fallback if safety guard triggers).

### 3.3 Statistical Significance

**How long does each test run?** The minimum test duration is determined by a sequential probability ratio test (SPRT):

```
H0: candidate is NOT better than production (δ ≤ 0)
H1: candidate IS better than production (δ > δ_min)

δ_min = minimum detectable improvement (default: 5% reduction in RMS error)
α = type I error rate (default: 0.05)
β = type II error rate (default: 0.20)

Required sample size (approximate):
    N ≈ ((Z_α + Z_β)² × σ²) / δ_min²

For σ = 1.0 (normalized error), δ_min = 0.05:
    N ≈ ((1.645 + 0.842)² × 1.0) / 0.0025 ≈ 4,950 ticks

At 100Hz: 49.5 seconds. At 10Hz: 8.25 minutes.
```

In practice, tests run for at least 1,000 ticks (10 seconds at 100Hz) before any statistical evaluation. Tests for Level 2+ mutations run for at least 100,000 ticks (~16 minutes at 100Hz) under varied conditions.

**What triggers promotion to production?** A variant is promoted when:
1. Its fitness score exceeds the production variant's by >5% with p < 0.05 (two-tailed t-test on the fitness metric distribution).
2. It has accumulated ≥10,000 shadow ticks with zero safety violations.
3. Its Lyapunov stability certificate is valid.
4. It has been tested in at least 3 distinct environmental conditions (different sea states, loadings, or temperatures).

**What triggers retirement?** A variant is retired from the pool when:
1. It has been in the pool for >14 days without promotion.
2. Its fitness score falls below 80% of the pool's best variant.
3. A newer variant has been promoted that is its direct descendant AND scores >10% higher.

## 4. CROSSOVER AND INHERITANCE

### 4.1 Bytecode Crossover

Two successful bytecodes CAN be "crossed" to create a child. This is not biological sexual recombination (which combines two full genomes) but rather **subtree crossover** inspired by genetic programming. The process:

1. **Identify crossover points.** Both parent bytecodes are parsed into a control flow graph (CFG). The CFG nodes are basic blocks — sequences of instructions with a single entry point and a single exit point (a branch or fall-through).
2. **Select compatible blocks.** A block from Parent A is compatible with a block from Parent B if: (a) both blocks have the same stack effect (net push/pop count), (b) both blocks read the same sensor registers and write to the same actuator registers, (c) both blocks have the same cyclomatic complexity (±1).
3. **Swap blocks.** The selected blocks are swapped, producing a child bytecode that inherits block structure from both parents. Jump targets are remapped to maintain CFG validity.
4. **Validate.** The child bytecode undergoes the full validation pipeline (structural validation, Lyapunov check, shadow execution).

**When is crossover used?** Crossover is used during Spring (exploration phase) when the colony needs to generate novel combinations from its existing gene pool. During Summer, the colony primarily uses single-parent mutation (asexual reproduction), which is faster and more conservative. The crossover rate is controlled by the seasonal parameters:

| Season | Mutation Rate | Crossover Rate | Epsilon Exploration |
|--------|-------------|---------------|-------------------|
| Spring | 30%         | 15%           | 30%               |
| Summer | 10%         | 5%            | 10%               |
| Autumn | 5%          | 2%            | 5%                |
| Winter | 0%          | 0%            | 0%                |

### 4.2 Inheritance Model

**Default: Asexual (single-parent).** Most variants inherit from a single parent. The child's metadata records only `parent_hash`. This is the Lamarckian model: the parent's accumulated experience (encoded in its bytecode structure) is directly inherited and then modified.

**Crossover (two-parent).** When crossover occurs, the child records both `parent_hash` and `parent2_hash`. The Griot narrative describes which blocks came from which parent and why the combination was expected to be beneficial.

**No horizontal gene transfer between niches.** A rudder control bytecode cannot directly inherit from a bilge pump bytecode. Cross-niche knowledge transfer happens only through the AI model (the queen bee), which observes patterns across all niches and proposes mutations informed by cross-niche correlations. This prevents "genetic contamination" — a bilge pump optimization accidentally breaking rudder control logic.

### 4.3 Griot Layer Maintenance Across Generations

The Griot narrative chain is maintained by the Jetson's learning pipeline. Each generation appends to the narrative chain:

```
Generation 410: "Increased Kp by 2% to compensate for hull fouling."
  └── references: Generation 389 ("Reduced Kp when hull was clean")
Generation 411: "Added sea-state conditional: if wave > 2m, reduce Kp by 15%."
  └── references: Generation 410, stress_test_analysis section 3.1
Generation 412: "Increased Kp by 3.2%, decreased Kd by 1.8%."
  └── references: Generation 411, ancestor A2.3 from 127 days ago
```

The narrative chain is stored as a linked list of JSON records on the Jetson's NVMe, indexed by content hash. The chain is append-only — old records are never modified. The chain grows linearly with generation count: ~500 bytes/generation × 365 generations/year = ~180KB/year. Over 10 years: 1.8MB. This is trivially small on the Jetson's NVMe.

When a variant is retired, its Griot record is archived but retained. When a variant is promoted, its Griot record is flagged as "production lineage." The colony's Griot layer enables the "temporal rewind" capability: given any historical content hash, the Jetson can reconstruct the full narrative chain of how that variant came to be, what conditions it was adapted for, and what lessons it carries.

## 5. CONDITIONAL GENETICS (MULTI-GENOME PORTFOLIOS)

### 5.1 Portfolio Structure

Each node maintains a portfolio of bytecodes for different environmental conditions. This is the phenotypic plasticity mechanism from Phase 1D. The portfolio is stored in the LittleFS `reflex_bc` partition:

```
reflex_bc partition layout:

    /reflex_rudder_calm.rbc      (12KB, gen 847)
    /reflex_rudder_moderate.rbc  (14KB, gen 623)
    /reflex_rudder_rough.rbc     (18KB, gen 412)
    /reflex_rudder_dock.rbc      (8KB,  gen 234)
    /reflex_rudder_safe.rbc      (4KB,  gen 0, factory default)
    /reflex_rudder_candidate_A.rbc (varies, under evaluation)
    /reflex_rudder_candidate_B.rbc (varies, under evaluation)
    /portfolio_index.json        (256 bytes, see below)
```

The `portfolio_index.json` defines the switching conditions:

```json
{
  "niche": "rudder_control",
  "node_id": 1,
  "genomes": [
    {
      "file": "reflex_rudder_calm.rbc",
      "content_hash": "a3f2b8c1...",
      "condition": {
        "type": "threshold",
        "sensor": "wave_height_m",
        "max": 0.5,
        "fallback": true
      }
    },
    {
      "file": "reflex_rudder_moderate.rbc",
      "condition": {
        "type": "range",
        "sensor": "wave_height_m",
        "min": 0.5,
        "max": 2.5
      }
    },
    {
      "file": "reflex_rudder_rough.rbc",
      "condition": {
        "type": "threshold",
        "sensor": "wave_height_m",
        "min": 2.5
      }
    },
    {
      "file": "reflex_rudder_dock.rbc",
      "condition": {
        "type": "manual",
        "trigger": "operator_docking_mode"
      }
    }
  ],
  "emergency": "reflex_rudder_safe.rbc"
}
```

### 5.2 Switching Triggers

Genome switching is triggered by **sensor-driven environmental classification**, not by explicit conditional code within bytecodes. The switching logic lives in the HAL (the "cell membrane"), not in the bytecode (the "genome"). This is a deliberate architectural choice:

1. **Sensor classification.** At each tick, before invoking the VM, the HAL evaluates the portfolio index conditions against current sensor values. This is a simple threshold/range comparison, not a VM operation.
2. **Genome selection.** The HAL selects the appropriate bytecode file and loads it into the VM. If the selected genome differs from the currently loaded genome, a "genome switch" event is emitted (via the EMIT_EVENT syscall in the outgoing genome, and logged by the telemetry task).
3. **State continuity.** When switching genomes, the VM's variable space is preserved (variables persist across genome switches), but PID state (integral accumulator, previous error) is RESET. This prevents integral windup artifacts from contaminating the new genome's PID controller.

**Switching latency:** A genome switch takes <1ms: HAL evaluates conditions (<10µs), LittleFS read of new bytecode (<500µs for 20KB), VM validation (<100µs), VM reinitialization (<100µs). This is well within the tick budget.

### 5.3 Fair Fitness Comparison Across Conditions

Different genomes operate under different conditions, making direct fitness comparison unfair. A calm-weather genome tested in Sea State 0 will always score higher than a rough-weather genome tested in Sea State 4, even if the rough-weather genome is "better" relative to its conditions.

**Solution: Condition-normalized fitness.**

```
normalized_fitness = (raw_fitness - condition_baseline_mean) / condition_baseline_std
```

Each environmental condition has a baseline mean and standard deviation computed from the production variant's historical performance under that condition. The normalized fitness measures how much better (in standard deviations) a variant performs relative to the baseline for its specific conditions. This allows fair comparison:

- Calm genome with normalized_fitness = +2.1σ → exceptionally good for calm conditions
- Rough genome with normalized_fitness = +1.8σ → very good for rough conditions
- These ARE comparable: the calm genome is "more exceptional" relative to its conditions

### 5.4 Preventing Genome Explosion

The system caps the portfolio at **7 genomes per niche** (the Apeiron Index minimum). When a new condition is identified that warrants a dedicated genome:

1. **Condition clustering.** The Jetson runs hierarchical clustering (HDBSCAN) on the environmental condition vectors from the last 90 days. If a cluster emerges that is not covered by any existing genome, a new genome slot is allocated.
2. **Slot allocation.** If the portfolio has fewer than 7 genomes, a new slot is created. If the portfolio already has 7 genomes, the LEAST-FIT genome (lowest normalized_fitness averaged over the last 30 days) is retired, and its slot is reused.
3. **Genome merging.** If two conditions have become sufficiently similar (their environmental condition vectors have cosine similarity > 0.95), the Jetson proposes merging their genomes. The merged genome is a crossover of the two parent genomes, optimized for the combined condition range.

This mechanism ensures that the portfolio grows to cover genuinely distinct conditions but never exceeds the storage and cognitive budget of 7 genomes per niche.

## 6. THE COMPUTE REDUCTION PATHWAY

### 6.1 Mechanism: How Evolution Reduces Compute

The Compute Reduction Theorem (Phase 1D) states that evolutionary adaptation reduces the computational cost of achieving a given level of performance. The mechanisms:

**Mechanism 1 — Dead code elimination.** The Kolmogorov fitness function (`behavioral_score / compressed_binary_size`) explicitly penalizes large bytecodes. Over generations, instructions that do not contribute to behavioral fitness are pruned. The Jetson's post-evolution optimizer detects dead code by symbolic execution: it traces all possible execution paths and marks instructions that are never reached (unreachable blocks) or whose outputs are never consumed (dead computations). These instructions are replaced with NOPs and eventually removed, reducing bytecode size.

**Mechanism 2 — Arithmetic simplification.** Common sub-expression elimination: if the same computation (e.g., `READ_PIN 3; MUL_F; PUSH_F32 0.5`) appears in multiple execution paths, the Jetson's optimizer lifts it to a shared variable. Constant folding: `PUSH_F32 3.14; PUSH_F32 2.0; MUL_F` becomes `PUSH_F32 6.28`.

**Mechanism 3 — Precision reduction.** Over generations, the colony discovers which computations require full float32 precision and which can use smaller representations. For example, a threshold comparison (`wave_height > 2.5m`) can use a float16 representation of the threshold (2.5 → 0x4140 in float16 → PUSH_I16 0x4140) if the sensor's own precision doesn't justify float32. The compiler emits PUSH_I16 instead of PUSH_F32, saving 2 bytes and 0 cycles (both are 1-cycle instructions, but the reduced memory footprint improves cache behavior in the VM's instruction fetch).

**Mechanism 4 — Structural specialization.** A general-purpose bytecode that handles all possible inputs with reasonable performance will be larger than a specialized bytecode that handles only the inputs it actually encounters. After 847 generations of exposure to THIS specific rudder's response characteristics, the colony may discover that certain input combinations never occur (e.g., heading error > 90° AND speed > 0 — because the vessel never turns that sharply at speed). The bytecode can eliminate the branch handling that impossible case, reducing both size and execution time.

### 6.2 Concrete Example: PID Gains Optimization Reducing VM Tick Time

Consider a rudder control bytecode with 3 PID controllers (heading, rate, damping) and 2 conditional branches (sea-state switch, speed limit). The original bytecode has 120 instructions (960 bytes).

**Before optimization (generation 0):**
```
Instruction sequence (simplified):
  READ_PIN 0           ; heading_error_deg (sensor 0)
  READ_PIN 1           ; vessel_speed_knots (sensor 1)
  PUSH_F32 0.0         ; speed threshold
  LT_F                 ; speed < 0?
  JUMP_IF_FALSE slow_speed_handler  ; branch for low-speed regime
  PUSH_F32 2.0         ; speed limit for rough seas
  LT_F                 ; speed < 2?
  JUMP_IF_FALSE normal_speed_handler
  ; ... 3 PID_COMPUTE calls with full gain set ...
  ; ... 2 CLAMP_F operations ...
  ; ... output WRITE_PIN ...
  HALT
Total: 120 instructions × 1-4 cycles = ~280 cycles ≈ 280µs at 1 cycle/µs
```

**After 847 generations of optimization (generation 847):**
```
Instruction sequence (simplified):
  READ_PIN 0           ; heading_error_deg
  ; speed check removed — this vessel never operates below 3 knots
  ; sea-state conditional simplified — calibration showed only 2 regimes needed
  ; PID_COMPUTE 0 (heading) — gains tightened, integral limit reduced
  ; PID_COMPUTE 1 (rate) — removed entirely (redundant with heading PID)
  ; output WRITE_PIN
  HALT
Total: 48 instructions × 1-4 cycles = ~96 cycles ≈ 96µs
```

The tick time decreased from 280µs to 96µs — a 66% reduction. This is more aggressive than the fleet-average 35% because this vessel has particularly consistent operating conditions (always >3 knots, always Sea State 0-3), allowing more aggressive specialization.

### 6.3 Kolmogorov Lower Bound

The theoretical minimum compute for a given control task is the Kolmogorov complexity of the optimal controller — the length of the shortest bytecode program that produces the required behavior. This is incomputable in general (Kolmogorov complexity is uncomputable), but the colony approximates it through evolutionary compression.

In practice, the lower bound is determined by the minimum instruction count required to:
1. Read the relevant sensor registers.
2. Compute the control law.
3. Clamp the output.
4. Write the actuator register.
5. Halt.

For a single PID controller: `READ_PIN; READ_PIN; PID_COMPUTE; CLAMP_F; WRITE_PIN; HALT` = 6 instructions = 48 bytes. The minimum is 6 instructions, but the actual minimum depends on the control law's complexity. A linear state feedback controller can be implemented in ~10 instructions. A nonlinear controller (e.g., with gain scheduling) requires ~20-40 instructions.

The colony's bytecodes typically converge to within 2-3× of this theoretical minimum after 500+ generations. The remaining overhead is structural — state machine transitions, logging (RECORD_SNAPSHOT, EMIT_EVENT), and safety margins (wider CLAMP_F ranges than strictly necessary).

## 7. ROLLBACK AND GENETIC TIME TRAVEL

### 7.1 Version History Storage

**On the ESP32 (local, fast):** The LittleFS `reflex_bc` partition (1MB) stores the current production bytecode and up to 6 additional genomes (the portfolio, see Section 5). This is sufficient for immediate switching and the last-known-good rollback. Storage per variant: 2-20KB. Total: 14-140KB for a 7-genome portfolio. Well within 1MB.

**On the Jetson (archive, deep):** The Jetson's NVMe stores the COMPLETE version history for all nodes. Storage per variant: bytecode (2-20KB) + Griot narrative (~500 bytes) + metadata (64 bytes) + fitness history (~200 bytes) ≈ 3-21KB. With 10 nodes evolving daily: 30-210KB/day. Over 2 years: 22-153GB. With LZ4 compression (~2:1 for repetitive bytecode): 11-77GB. The Jetson's 500GB NVMe holds 6-45 years of history.

**Version history lifecycle:**
- **0-90 days:** Full fidelity. Every variant stored with complete metadata, telemetry samples, and Griot narrative.
- **90 days - 2 years:** Compressed. Bytecode stored; telemetry downsampled to 1Hz summaries; Griot narrative retained.
- **2+ years:** Aggregated. Only bytecode hash, fitness score summary, lineage chain, and final Griot narrative retained. The actual bytecode binary is deleted (but can be reconstructed from the lineage chain if the parent binaries are still available).

### 7.2 "Rewind to Any Stable Point in Under a Minute"

**For locally stored versions (ESP32 → ESP32):**
1. HAL receives rollback command with target content hash.
2. HAL checks LittleFS for a bytecode with that hash. If found: validate (<100µs), load into VM (<500µs), reinitialize PID state (<10µs). Total: <1ms.
3. If not found locally, request from Jetson.

**For Jetson-archived versions (Jetson → ESP32):**
1. ESP32 sends rollback request with target content hash over RS-422.
2. Jetson searches its version history (SQLite index on NVMe). Lookup: <10ms.
3. Jetson compresses the bytecode with LZ4 (if not already compressed) and transmits over RS-422 at 921,600 baud.
4. Transfer time for 20KB bytecode: 20,000 bytes / 92,160 bytes/s ≈ 217ms. With COBS framing overhead (worst case +1 byte/254 bytes): ~220ms.
5. ESP32 receives, validates, writes to LittleFS, loads into VM.
6. Flash write time for 20KB (3 sectors × 4KB): ~60ms.
7. Total: ~300ms. Well under the 1-minute target.

**For deeply archived versions (>2 years, bytecode deleted):**
1. Jetson reconstructs the bytecode by replaying the lineage chain from the nearest available ancestor.
2. For each generation in the chain: look up the mutation description, apply it to the parent bytecode, validate.
3. Reconstruction time: ~10ms per generation. For a chain of 200 generations: ~2 seconds.
4. Then transfer to ESP32 as above: ~300ms.
5. Total: ~2.3 seconds. Still well under 1 minute.

### 7.3 What Defines a "Stable Point"

A "stable point" is a version that meets ALL of the following criteria:

1. **Structural validity.** The bytecode passes the validator's full check (valid opcodes, aligned jumps, bounded stack, no call cycles).
2. **Lyapunov stability.** The variant has a valid Lyapunov stability certificate with `min_eigenvalue > 0.001`. (Eigenvalues below 0.001 indicate marginal stability — technically stable but dangerously close to the boundary.)
3. **Operational validation.** The variant has been tested for ≥10,000 ticks (100 seconds at 100Hz) with zero VM halts and zero safety guard interventions.
4. **Human implicit approval.** The variant has been in production for ≥24 hours without a manual rollback. (This is "implicit" approval — the human did not intervene.)
5. **Fitness floor.** The variant's fitness score is ≥80% of the pool's best fitness at the time of its promotion.

Only versions meeting all 5 criteria are marked as "stable points" in the Jetson's version database. These are the versions that appear in the rollback menu and are preserved indefinitely (they are never subject to the 2-year aggregation lifecycle).

### 7.4 Reconstructing Environmental Context

When rolling back to a historical version, the colony must also reconstruct the environmental context in which that version was optimized. This is critical because a bytecode optimized for Sea State 2 at 10°C may perform poorly at Sea State 5 at 25°C.

**Context reconstruction:**
1. The variant's `environment_hash` identifies the environmental conditions during its testing.
2. The Jetson retrieves the Griot narrative for the variant, which describes the specific conditions in natural language.
3. The Jetson compares the current environmental conditions (from live telemetry) to the variant's historical conditions using cosine similarity on the environment feature vector.
4. If the similarity is >0.90, the rollback proceeds normally.
5. If the similarity is 0.70-0.90, the Jetson issues a WARNING: "This variant was optimized for different conditions. It may perform suboptimally." The rollback proceeds with the warning.
6. If the similarity is <0.70, the Jetson issues a CAUTION and recommends finding a more appropriate variant from the portfolio. The rollback still proceeds (the operator has the final say), but the system switches to the portfolio's emergency genome if any safety violation occurs within the first 1,000 ticks.

## 8. UNSOLVED PROBLEMS AND OPEN QUESTIONS

This section honestly catalogs the problems that the current specification does NOT solve.

**Problem 8.1 — Emergent Behavior Detection.** When multiple independently-evolved bytecodes interact through shared physical systems, emergent behaviors can arise that no single bytecode encodes. The current specification has no mechanism for detecting or characterizing these behaviors at the colony level. The observability layer is node-centric, not colony-centric.

**Problem 8.2 — Nonlinear Lyapunov Certificates.** The Lyapunov stability certificate is computed for linearized models. For strongly nonlinear plants (rudder stall, pump cavitation, actuator saturation), the linearized certificate may not guarantee stability. Sum-of-squares (SOS) optimization can provide nonlinear certificates but is NP-hard in the worst case and computationally infeasible on the Jetson for real-time verification.

**Problem 8.3 — Inter-Colony Knowledge Transfer.** When two vessels with independent colonies encounter each other, there is no mechanism for sharing evolved bytecodes. The current architecture treats each colony as isolated. A marketplace or peer-to-peer sharing protocol for colony genomes would dramatically accelerate adaptation for new deployments.

**Problem 8.4 — Fitness Function Drift.** The fitness function is defined as a constitutional document that cannot be modified by evolution. But the environment changes, and a fitness function optimized for one set of conditions may produce pathological behavior in another. The mechanism for updating the fitness function (human-driven, seasonal) is defined but not the criteria for when an update is necessary.

**Problem 8.5 — The AI Model's Creative Horizon.** The colony's Lamarckian evolution is bounded by the AI model's latent space. If the optimal solution for a novel condition was not represented in the model's training data, the colony cannot discover it. The epsilon-exploration mechanism (10% random variants) partially addresses this but is limited to the bytecode mutation space, not the algorithmic design space. Truly novel algorithms require Level 3 mutation (AI-proposed), which is constrained by the model's training distribution.

**Problem 8.6 — Deterministic Replay Failures.** When replaying a historical bytecode against historical telemetry, the results may not match the original execution if the PID state (integral accumulator, previous error) at the start of the replay differs from the original. The current specification does not mandate recording PID state with each telemetry sample. Adding this would increase telemetry size by 24 bytes per PID controller per sample (8 controllers × 2 floats each = 64 bytes), which may exceed the RS-422 bandwidth budget at high sample rates.

## APPENDIX B: FITNESS FUNCTION EQUATIONS

The extended fitness function (from the Colony Thesis, Layer 1):

```
F_total(v) = α·F_immediate(v) + β·F_heritability(v) + γ·F_adaptability(v) + δ·F_reversible(v) - ε·Debt(v)
```

Where:
- `F_immediate(v) = behavioral_score(v) / compressed_size(v)` — Kolmogorov fitness, the ratio of behavioral quality to bytecode efficiency. Behavioral score is computed from the 128-byte fingerprint vector's distance to the ideal fingerprint (defined by the reflex specification).
- `F_heritability(v) = mean(fitness(children of v))` — How well this variant's innovations are preserved in its descendants. Computed retrospectively after children have been evaluated.
- `F_adaptability(v) = min(F_normalized(v, c) for c in condition_set)` — The worst-case normalized fitness across all tested conditions. Penalizes variants that excel in one condition but fail in another.
- `F_reversible(v) = 1.0 / (1.0 + Hamming_distance(v, parent_v))` — How easily the variant can be reverted to its parent. Small mutations score higher than large ones.
- `Debt(v) = flash_consumed(v) - flash_budget` + `max(0, variable_count(v) - 32)` — Generational debt: resources consumed by this variant that reduce the colony's future optionality.
- `α = 0.5, β = 0.15, γ = 0.20, δ = 0.10, ε = 0.05` — Default weights. These are constitutional constants, NOT tuneable by evolution. Changes require human approval with philosophical justification.
