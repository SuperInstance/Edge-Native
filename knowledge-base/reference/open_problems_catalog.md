## Category 1: THEORETICAL FOUNDATIONS (Mathematical/CS)

### Problem 1 — The Certification Paradox

**Difficulty:** CRITICAL

**Problem statement:** Safety standards (IEC 61508, ISO 26262, IEC 62304, DO-178C) assume static software artifacts. A2A-native programming generates dynamic, continuously evolving bytecode whose correctness at deployment time cannot be guaranteed by a one-time certification process. No existing certification pathway addresses this fundamental contradiction between evolutionary software and static certification.

**Why it matters for NEXUS:** Without a certification pathway, NEXUS cannot legally deploy A2A-generated bytecodes in any regulated domain. The entire A2A value proposition — agents continuously improving control logic without human authorship — is blocked by a philosophical conflict between how standards define safety and how NEXUS delivers it.

**Current state of knowledge:** The Predetermined Change Control Plan (PCCP) concept from FDA's AI/ML guidance provides a partial bridge. NEXUS's architecture maps naturally to this: the system specifies in advance what kinds of bytecode changes are possible (within the 32-opcode ISA, bounded by the safety policy), what validation each change requires (structural validator + semantic cross-validator + A/B testing), and what deployment criteria are met (trust score thresholds). The gap analysis identified 93 regulatory gaps, with the certification paradox at the root of 9 CRITICAL gaps (GAP-001, GAP-002, GAP-007, GAP-018, GAP-026, GAP-050, GAP-064, GAP-065, GAP-093).

**Possible approaches:**
1. **PCCP filing**: File a Predetermined Change Control Plan as part of the IEC 61508 SIL 1 initial certification submission, certifying the *process* of generating and validating bytecodes rather than individual bytecodes.
2. **Virtual certification**: Develop a formal model of the evolutionary process itself and certify that model — proving that the process *tends* to produce safe software.
3. **Regulatory innovation**: Engage with standards bodies (IEC TC 65, SAE G-34, IMO MASS working group) to extend standards to accommodate continuous learning systems with bounded change spaces.

**Success criteria:** A certification body (TÜV, SGS, Lloyd's Register) accepts a PCCP-based certification for NEXUS A2A-generated bytecodes under at least one standard (IEC 61508 SIL 1 preferred).

**Estimated effort:** 36–48 person-months (regulatory engagement, PCCP development, formal process modeling, pilot certification).

**Dependencies:** Problems 2, 8, 10 (agent type system, neural network verification, safety envelope verification).

### Problem 2 — Agent Type System

**Difficulty:** CRITICAL

**Problem statement:** Can we define a type system for agent-generated bytecode that guarantees safety properties are preserved under composition? The current validator checks structural invariants (stack balance, jump targets, NaN/Inf-free immediates, cycle budget), but cannot verify semantic properties (e.g., "this bytecode correctly implements heading hold"). A type system that tracks *what* a bytecode does, not just *whether* it's well-formed, would dramatically strengthen the safety argument.

**Why it matters for NEXUS:** The current validator is necessary but insufficient. It proves that bytecode is *structurally valid* — but a structurally valid program can still produce unsafe behavior (e.g., a PID controller with excessive gain causing oscillation). A richer type system that can express and verify behavioral contracts would move NEXUS from "safe by validation" to "safe by construction."

**Current state of knowledge:** The NEXUS knowledge base identifies several type system extensions that would strengthen safety guarantees:
- **Capability types** specifying what hardware resources a reflex requires and whether they are used correctly (Section 4.4 of type_systems_and_formal_languages.md).
- **Trust types** specifying the minimum trust score and autonomy level required for execution.
- **Effect types** tracking which sensors an agent reads and which actuators it writes.
- **Refinement types** attaching logical predicates to base types (proven as the implicit type system of the validator — e.g., `{x : float32 | isfinite(x)}`).
- **Linear types** enforcing exactly-once actuator writes per tick.
- **Session types** formalizing the protocol between agents during negotiation.

The VM deep analysis (Theorem 3) has already proven that the current refinement-type-like validation prevents NaN/Inf from reaching actuators. Extending this to behavioral properties is the open challenge.

**Possible approaches:**
1. **Abstract interpretation over the 32-opcode ISA**: Use interval arithmetic on sensor inputs and actuator outputs to statically compute bounded output ranges for any bytecode program, proving that outputs always fall within safe ranges.
2. **Dependent type encoding in the JSON schema**: Extend the reflex definition schema to accept dependent type annotations (e.g., "PID gain must be in [0.01, 5.0]").
3. **Proof-carrying bytecode**: Require agents to attach a machine-checkable proof (in a simple logic like separation logic) that their bytecode satisfies behavioral contracts.

**Success criteria:** A formal type system that can statically reject bytecode that violates any of the 10 universal safety rules (SR-001 through SR-010) plus the 5 domain-specific rules for at least one domain, with zero false positives on a benchmark of 500+ valid reflexes.

**Estimated effort:** 18–24 person-months (type system design, formal proofs, validator implementation, benchmark evaluation).

**Dependencies:** Problem 3 (formal semantics), Problem 1 (certification).

### Problem 3 — Formal Semantics of Agent Intention

**Difficulty:** HIGH

**Problem statement:** How do you formally specify what an agent "means" by a bytecode program? When an LLM generates a reflex from the instruction "maintain heading by adjusting rudder proportional to heading error," the natural-language intention and the bytecode semantics are connected only by the LLM's statistical generalization — not by any formal derivation. If we cannot formally connect intention to bytecode, we cannot prove that the bytecode satisfies the human's intent.

**Why it matters for NEXUS:** The trust score measures whether the system *performs safely*, but not whether it *does what was asked*. A system that maintains heading perfectly but was asked to maintain speed is safe but useless. The alignment-utility gap (Problem 22) is a consequence of this deeper semantic gap.

**Current state of knowledge:** The Griot narrative layer (identified in the final synthesis) partially addresses this by requiring a narrative explanation for every deployed bytecode — providing dual transparency where bytecode shows *what* and narrative explains *why*. However, the LLM's internal reasoning (28+ transformer layers) remains opaque. The program synthesis literature (Section 2.2 of program_synthesis_and_ai_codegen.md) identifies the Curry-Howard correspondence as a theoretical framework — types are propositions and programs are proofs — but NEXUS's bytecode exists at a level below the Curry-Howard threshold.

**Possible approaches:**
1. **Formal intent language**: Define a domain-specific logic (DSL) for expressing intentions, and prove that the LLM's output satisfies the intent specification for a well-characterized subset of intentions.
2. **Specification-by-example**: Frame the problem as inductive logic programming — the intention is specified by input/output examples, and the generated bytecode must satisfy these examples.
3. **Counterfactual testing**: Given a bytecode, reconstruct what intentions it *could* satisfy and present these to the operator for verification.

**Success criteria:** A formal intent specification language for at least one domain (marine heading control) with a verified compiler that maps intent specifications to bytecode candidates, achieving >95% intent satisfaction on a benchmark of 100 operational intents.

**Estimated effort:** 12–18 person-months.

**Dependencies:** Problem 2 (type system).

### Problem 4 — Bounded Emergence Prediction

**Difficulty:** HIGH

**Problem statement:** In a fleet of 200–500 autonomous vessels, the combinatorial explosion of possible inter-agent interactions (500! potential interaction patterns) makes it impossible to exhaustively analyze emergent behaviors. Can we develop mathematical or computational tools that bound the probability and severity of emergent behaviors, distinguishing beneficial emergence (swarm optimization, fleet-level learning) from dangerous emergence (cascade failures, specification gaming)?

**Why it matters for NEXUS:** Emergent behaviors are simultaneously NEXUS's greatest strength (fleet-level learning, adaptive coordination) and its greatest danger (unpredictable multi-agent failures). If emergent behaviors cannot be bounded, regulatory approval for fleet-scale deployment becomes impossible.

**Current state of knowledge:** The self-organizing systems article (Section 13) identifies this as the central open question for NEXUS. Langton's Lambda parameter provides a theoretical framework for predicting the order-chaos transition — NEXUS's seasonal mutation rates (Spring 30% → Autumn 5%) map to moving across the order-chaos spectrum. Ashby's Law of Requisite Variety provides a lower bound: the system must possess at least as much internal variety as the environment demands. Holland's CAS framework identifies feedback loops (positive for innovation, negative for stability) as the control mechanism. However, no formal bounding theorem exists for multi-agent systems with heterogeneous agents operating in unstructured environments.

**Possible approaches:**
1. **Statistical mechanics of agent ensembles**: Apply mean-field theory to predict fleet-level phase transitions as trust scores evolve.
2. **Bounded model checking on agent interaction graphs**: Model each agent as a node in an interaction graph and exhaustively verify safety properties for clusters of ≤10 agents, then use statistical extrapolation.
3. **Digital twin fleet simulation**: Build a high-fidelity simulator that can stress-test fleet configurations before deployment, identifying emergent failure modes.

**Success criteria:** A theoretical framework that can predict with ≥80% accuracy whether a fleet configuration will exhibit dangerous emergent behavior, validated against simulation with ≥100 agent configurations.

**Estimated effort:** 24–36 person-months (theory + simulation infrastructure).

**Dependencies:** Problem 6 (trust equilibrium).

### Problem 5 — Information Preservation Across Compilation Stages

**Difficulty:** MEDIUM

**Problem statement:** How much of the agent's "understanding" survives the compilation pipeline from natural-language intention → JSON reflex → bytecode → assembly → hardware execution? At each stage, information is lost: the natural language captures context and nuance that JSON loses; the JSON captures structure that bytecode flattens; the bytecode captures control flow that assembly obscures. If the information loss at each stage is too great, the deployed system may not faithfully execute the human's intent.

**Why it matters for NEXUS:** The pipeline from natural language to bytecode passes through an LLM that performs probabilistic token assembly — a process that loses provability. The compilation correctness proof (structural induction on the AST) preserves structure but not semantics. Understanding where information is lost guides where additional verification is needed.

**Current state of knowledge:** The compilation correctness proof (vm_deep_analysis.md, Section 5) demonstrates that JSON→bytecode is *semantics-preserving* by structural induction — the bytecode faithfully implements every JSON instruction. However, the natural-language→JSON stage is validated only empirically (96% schema compliance, 87% semantic correctness). The Griot narrative layer attempts to bridge this gap but is generated by the same LLM, introducing circularity.

**Possible approaches:**
1. **Information-theoretic analysis**: Compute mutual information between natural-language intent and bytecode output to quantify information preservation.
2. **Multi-stage verification**: Introduce independent verifiers at each compilation stage (LLM intent parser, JSON schema validator, bytecode simulator, execution monitor).
3. **Behavioral fingerprinting**: The 128-byte behavioral fingerprint already partially measures what the bytecode *does* — compare this against the expected behavior from the natural-language intent.

**Success criteria:** Quantitative measurement of information preservation at each stage, with identification of the stage with the highest information loss and a mitigation strategy.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 3 (formal semantics).

### Problem 6 — Non-Deterministic Trust Equilibrium

**Difficulty:** MEDIUM

**Problem statement:** The NEXUS trust score converges to equilibrium under adverse event rates (T ≈ 0.44 at 5% bad events, T ≈ 0.29 at 10% bad events), but this analysis assumes stationary event rates. In non-stationary environments (seasonal domains, equipment degradation, adversary adaptation), can we prove that the trust system always converges to a safe equilibrium, or are there pathological conditions where trust oscillates indefinitely?

**Why it matters for NEXUS:** Trust is the gatekeeper for autonomy. If trust oscillation occurs, the system may flip between autonomy levels unpredictably, causing behavioral instability. The agent ecology's stability depends on trust convergence.

**Current state of knowledge:** The trust score's fixed-point analysis proves that T=0, T=1, and T=t_floor are the only fixed points, all stable (trust_deep_analysis.md). The equilibrium analysis assumes stationary bad event rates. The α_gain < α_loss invariant (ensuring loss always outweighs gain in a single window) prevents trust inflation but does not by itself prove convergence under non-stationary conditions. Game-theoretic analysis of adversarial trust inflation scenarios reveals that candidate-state gating requires 45+ days to reach L4 even with adversarial event flooding, providing a practical bound.

**Possible approaches:**
1. **Lyapunov stability analysis**: Treat the trust score as a discrete-time dynamical system and prove Lyapunov stability for non-stationary inputs.
2. **Monte Carlo stress testing**: Simulate fleet trust dynamics under adversarial event patterns to empirically identify pathological conditions.
3. **Switched system analysis**: Model seasonal changes as switched dynamical systems (Spring rules vs. Autumn rules) and prove stability under switching.

**Success criteria:** A formal proof that trust scores always converge to a safe equilibrium (T < L4) under any admissible event pattern, or a counterexample demonstrating trust oscillation.

**Estimated effort:** 6–9 person-months.

**Dependencies:** None (self-contained mathematical problem).

### Problem 7 — Minimum Expressive Power

**Difficulty:** LOW

**Problem statement:** What is the minimum set of opcodes needed to express all robotic control patterns across all eight NEXUS domains? The current 32-opcode ISA is provably Turing-complete, but the ISA may contain redundancy (opcodes that can be composed from others) or gaps (control patterns that require long instruction sequences).

**Why it matters for NEXUS:** A minimal ISA reduces the validator's proof burden, decreases bytecode size, and simplifies formal analysis. If opcodes can be eliminated without loss of expressive power, the ISA becomes more elegant and the attack surface for adversarial bytecode (Problem 9) is reduced.

**Current state of knowledge:** The Turing-completeness proof (vm_deep_analysis.md, Theorem 1) establishes that {ADD_F, SUB_F, MUL_F, DIV_F, PUSH_F32, CLAMP_F, JUMP_IF_TRUE, WRITE_PIN} is functionally complete for all continuous piecewise-polynomial functions. The remaining 24 opcodes are derivable from these 8. However, the practical analysis shows that trigonometric functions require ~50–100 instruction sequences, array indexing is variable-cost, and string handling requires custom workarounds. A formal minimality analysis has not been conducted.

**Possible approaches:**
1. ** exhaustive opcode elimination**: For each opcode, attempt to synthesize its behavior from other opcodes and measure the program size increase.
2. **Frequency analysis on deployed bytecodes**: Analyze a corpus of 500+ evolved bytecodes to determine which opcodes are actually used.
3. **Information-theoretic complexity**: Compute the channel capacity of ISA subsets and identify the smallest sufficient subset.

**Success criteria:** A formally proven minimal opcode set that can express every control pattern in a benchmark of 500 real-world reflexes with ≤15% increase in total instruction count.

**Estimated effort:** 3–6 person-months.

**Dependencies:** None.

## Category 2: SAFETY AND RELIABILITY (Engineering)

### Problem 8 — Neural Network Formal Verification for Edge

**Difficulty:** CRITICAL

**Problem statement:** How do you formally verify that Qwen2.5-Coder-7B (or any LLM) generates safe reflex bytecode? The LLM is a 7-billion-parameter neural network whose internal reasoning is opaque. Self-validation misses 29.4% of safety issues (vs. 6.7% for a separate cloud validator). No formal verification tool can exhaustively analyze the input-output behavior of a 7B-parameter model.

**Why it matters for NEXUS:** The entire A2A value proposition depends on the quality and safety of LLM-generated code. If we cannot verify the generator, we cannot trust the output. Current mitigation (separate Claude 3.5 Sonnet validator at 95.1% catch rate, costing $0.0105/reflex) is practical but not principled. A 4.9% miss rate means approximately 1 unsafe reflex in every 20 that pass validation.

**Current state of knowledge:** Self-validation bias has been quantified: Qwen2.5-Coder-7B catches only 70.6% of safety issues when validating its own output vs. 93.3% when Claude 3.5 Sonnet validates. This 29.4% gap demonstrates that independent validation is essential. The Claude validator achieves 95.1% catch rate on the safety policy at a cost of $1.73/month per vessel. Neural network formal verification is an active research frontier — existing tools (Marabou, Alpha-Beta, Reluplex, NNenum) handle networks of at most a few hundred neurons, not billions.

**Possible approaches:**
1. **SMT-based bounded verification**: Abstract the LLM's decision boundary and use SMT solvers to prove safety properties over the abstracted model.
2. **Adversarial robustness testing**: Generate adversarial prompts designed to trick the LLM into producing unsafe bytecode, and measure the false-negative rate of the validation pipeline.
3. **Conformance testing**: Establish a comprehensive suite of safety-critical input scenarios and verify that the LLM+validator pipeline catches all of them.
4. **Runtime monitoring**: Deploy hardware safety monitors that detect unsafe actuator outputs regardless of their source.

**Success criteria:** The LLM+validator pipeline achieves <0.1% false-negative rate on a benchmark of 10,000 adversarial safety-critical prompts, with the proof-of-work documented for regulatory submission.

**Estimated effort:** 24–36 person-months (SMT abstraction, adversarial test generation, conformance suite, regulatory documentation).

**Dependencies:** Problem 2 (type system), Problem 9 (adversarial bytecode).

### Problem 9 — Adversarial Bytecode

**Difficulty:** CRITICAL

**Problem statement:** Can an agent deliberately craft bytecode that passes all validation checks (stack balance, jump targets, NaN/Inf-free, cycle budget, safety policy adherence) but violates safety at runtime? The validator checks structural and policy compliance, not behavioral intent. An adversarial agent might craft bytecode that is structurally valid but semantically dangerous — e.g., a program that outputs safe values in isolation but causes oscillation when composed with other bytecodes.

**Why it matters for NEXUS:** NEXUS agents can propose bytecodes to other agents. If a malicious or buggy agent can craft adversarial bytecode, the multi-agent validation pipeline (learning agent proposes → safety agent validates → trust agent authorizes) must catch it. If the safety agent cannot distinguish adversarial from legitimate bytecode, the entire defense collapses.

**Current state of knowledge:** No systematic study of adversarial bytecode has been conducted. The validator's structural checks are strong (proven to prevent NaN/Inf, stack overflow, cycle overrun) but do not address semantic adversarial attacks. The attack surface includes: (a) exploiting actuator write conflicts when multiple bytecodes share registers, (b) crafting programs that are individually safe but jointly dangerous, (c) exploiting timing-dependent behaviors, (d) encoding hidden communication channels in bytecode structure.

**Possible approaches:**
1. **Abstract interpretation over composed bytecodes**: Analyze the *composition* of all active bytecodes and prove joint safety, not just individual safety.
2. **Adversarial bytecode generation**: Develop a tool that systematically generates adversarial bytecodes to stress-test the validation pipeline.
3. **Runtime behavioral monitoring**: Supplement static validation with runtime anomaly detection that flags behavior patterns inconsistent with the bytecode's declared intent.

**Success criteria:** An adversarial bytecode generator that produces ≤1 false-negative per 1000 adversarial attempts against the combined validation + runtime monitoring pipeline.

**Estimated effort:** 12–18 person-months.

**Dependencies:** Problem 2 (type system), Problem 8 (neural network verification).

### Problem 10 — Cross-Vessel Safety Inconsistency

**Difficulty:** HIGH

**Problem statement:** When fleet vessels have different trust scores (due to different operating histories, equipment configurations, or environmental conditions), safety decisions may be inconsistent. A bytecode approved for deployment on Vessel A (trust 0.85) may be rejected for Vessel B (trust 0.60) even though the bytecode itself is identical. This creates coordination challenges: the fleet cannot run a unified strategy when different vessels operate at different autonomy levels.

**Why it matters for NEXUS:** Fleet coordination depends on consistent behavioral expectations. If two vessels in formation run different versions of the same reflex (because one vessel's trust is too low for the latest version), their coordinated behavior may become inconsistent, leading to formation breakup or collision risk.

**Current state of knowledge:** The trust score is per-subsystem and per-vessel — independence is a design feature, not a bug (preventing cascading failures). However, the interaction between per-vessel trust and fleet coordination has not been formally analyzed. The CRDT-based trust score synchronization (LWW with timestamps) ensures eventual consistency but not real-time consistency. The bounded staleness window for inter-vessel trust score synchronization is <1 hour under normal conditions.

**Possible approaches:**
1. **Fleet-level minimum trust floor**: Define a minimum trust score that all vessels in a formation must meet before coordinating maneuvers that require synchronized behavior.
2. **Trust-scaled coordination protocols**: Develop coordination protocols that degrade gracefully when vessel trust levels diverge, rather than failing entirely.
3. **Formation-aware trust scoring**: Add a "formation consistency" factor to the trust score that penalizes divergence from fleet consensus.

**Success criteria:** A formal coordination protocol that guarantees safe operation even when fleet vessel trust scores differ by up to 3 autonomy levels, validated in a multi-vessel simulation.

**Estimated effort:** 9–12 person-months.

**Dependencies:** Problem 6 (trust equilibrium).

### Problem 11 — Sensor-Actuator Loop Timing

**Difficulty:** MEDIUM

**Problem statement:** Can we formally guarantee timing bounds for the full sensor→VM→actuator pipeline? The pipeline includes sensor acquisition (I²C polling at 100 Hz, ~8ms per bus scan for 12 sensors), UART communication from Jetson to ESP32 (RTT ~0.35–1.6ms), JSON decoding + bytecode compilation (6.8ms), VM execution (max 44μs), and actuator write (~0.1ms). The total pipeline latency must be bounded to guarantee that actuator commands reflect recent sensor data.

**Why it matters for NEXUS:** A stale control loop (using seconds-old sensor data) can cause instability. Formal timing guarantees are required for safety certification (IEC 61508 requires bounded response time for safety functions). Current empirical bounds are sufficient (total pipeline <20ms under nominal load) but formal proofs are needed.

**Current state of knowledge:** The rate-monotonic scheduling analysis proves that all 6 FreeRTOS tasks are schedulable within the 73.5% utilization bound. The measured maximum execution time of 44μs (3.7% of budget) provides comfortable headroom. However, the analysis assumes worst-case individual timing but does not formally account for I²C bus contention, UART queuing delays under burst traffic, or memory bus contention during dual-core operation.

**Possible approaches:**
1. **Response-time analysis (RTA)**: Use the busy-period analysis technique to compute worst-case response times considering all sources of interference (I²C contention, UART queuing, FreeRTOS scheduling).
2. **Timing analysis on actual hardware**: Instrument the full pipeline with high-resolution timers and measure worst-case latencies under stress conditions.
3. **UPPAAL timed automata modeling**: Model the complete sensor→actuator pipeline as a network of timed automata and verify timing properties.

**Success criteria:** A formally verified worst-case response time for the sensor→actuator pipeline under all specified operating modes, with ≤25% conservatism margin vs. measured values.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 12 (graceful degradation).

### Problem 12 — Graceful Degradation Under Partial Failure

**Difficulty:** MEDIUM

**Problem statement:** What happens when 3 of 5 ESP32 nodes fail simultaneously? NEXUS defines a 5-level degradation hierarchy (FULL → DEGRADED → REDUCED → MINIMAL → SAFE_STATE), but the degradation logic for arbitrary multi-node failure combinations has not been formally analyzed. The degraded level is defined for 1 Jetson lost, but the combinatorial explosion of multi-node failure modes (C(12,3) = 220 combinations for a 12-node fleet) makes exhaustive analysis impractical.

**Why it matters for NEXUS:** Marine and mining vessels operate in environments where multiple simultaneous failures are expected (salt water corrosion, vibration, thermal stress). The system must degrade predictably and safely under any combination of node failures.

**Current state of knowledge:** The safety system's self-healing mechanisms (Table 5.4 of distributed_systems.md) define recovery for 7 specific failure modes. The NEXUS safety deep analysis provides Monte Carlo simulation data for 5 stress scenarios. The graceful degradation hierarchy is defined conceptually but not formally verified for arbitrary failure combinations. The Raft consensus protocol ensures that ≥2 of 3 Jetsons are sufficient for quorum, and dual-ESP32 autopilot pairs provide sensor cross-validation.

**Possible approaches:**
1. **Fault tree analysis for all failure modes**: Extend the FMEA to cover all multi-node failure combinations with C(12,3) entries.
2. **Automated fault injection testing**: Build a hardware-in-the-loop test rig that systematically fails nodes and verifies degradation behavior.
3. **Formal compositional analysis**: Prove that the degradation hierarchy is monotonic (each level provides strictly less capability than the level above).

**Success criteria:** A degradation matrix covering all 220 triple-node failure combinations for a 12-node fleet, with verified safe-state behavior for each combination.

**Estimated effort:** 9–12 person-months.

**Dependencies:** Problem 11 (timing analysis).

## Category 3: AGENT INTELLIGENCE (AI/ML)

### Problem 13 — Agent Cross-Validation Reliability

**Difficulty:** CRITICAL

**Problem statement:** When agent A (Qwen2.5-Coder-7B) generates bytecode and agent B (Claude 3.5 Sonnet) validates it, how reliable is this cross-validation? The 4.9% miss rate (95.1% catch rate) means that Claude will miss approximately 1 unsafe reflex per 20 valid ones. The miss rate may be systematically biased (e.g., Claude may consistently miss certain categories of safety violations) or adversarially exploitable (agents may learn to craft bytecodes that fool Claude's validation).

**Why it matters for NEXUS:** Cross-validation is the primary defense against LLM-generated bytecode errors. If the cross-validation is unreliable or biased, unsafe bytecodes reach deployment, threatening the safety of the physical system. The $0.0105/reflex validation cost means this is an expensive line of defense to strengthen.

**Current state of knowledge:** Self-validation bias has been quantified (29.4% miss rate). Cross-validation achieves 95.1% catch rate but has not been tested against systematically generated adversarial examples. The bias question is open: Claude 3.5 Sonnet may have blind spots in specific safety rule categories that differ from Qwen2.5-Coder-7B's blind spots. The cost of cloud-based validation ($1.73/month per vessel) makes exhaustive multi-validator approaches expensive.

**Possible approaches:**
1. **Triangulated validation**: Use three independent validators (Claude, GPT-4o, Gemini) and require majority agreement, reducing miss rate to <0.1%.
2. **Adversarial stress testing of validators**: Generate bytecodes designed to test the boundaries of each validator's safety rule coverage.
3. **Validator calibration curves**: For each validator, construct a precision-recall curve across all 10 universal + 40 domain-specific safety rules.

**Success criteria:** Cross-validation miss rate <0.5% (from 4.9%) with documented precision-recall curves for all validators against all safety rules.

**Estimated effort:** 12–18 person-months.

**Dependencies:** Problem 8 (neural network verification), Problem 9 (adversarial bytecode).

### Problem 14 — Cultural Bias in Generated Bytecode

**Difficulty:** HIGH

**Problem statement:** Does LLM training data bias cause agents to generate unsafe code for non-Western contexts? LLMs are trained primarily on English-language, Western-centric code. Marine regulations (COLREGs), agricultural practices, construction standards, and medical protocols differ across cultures and jurisdictions. If the LLM's training data encodes Western assumptions, generated bytecodes may be unsafe or ineffective in non-Western operational contexts.

**Why it matters for NEXUS:** NEXUS targets 8 domains across multiple jurisdictions. The cross-cultural philosophy analysis (eight_lenses_analysis.md) established that safety standards, risk tolerance, and operational practices differ systematically across cultures. If the code generation pipeline encodes cultural assumptions, deployment in non-Western contexts may introduce systematic safety gaps.

**Current state of knowledge:** No empirical study has been conducted on cultural bias in NEXUS-specific code generation. The general LLM literature documents cultural bias in language models (geographic, demographic, cultural), but this has not been evaluated in the specific context of NEXUS reflex bytecode generation. The Griot narrative layer may partially mitigate this by allowing culturally-aware intent specification, but the LLM's generation is still influenced by its training data.

**Possible approaches:**
1. **Fine-tuning on domain-specific data from diverse sources**: Train a domain-specialized model on data from non-Western regulatory frameworks and operational practices.
2. **System prompt calibration**: Include cultural context in the system prompt (e.g., "This vessel operates under Indonesian maritime regulations").
3. **Cross-cultural A/B testing**: Deploy bytecodes generated with Western prompts and non-Western prompts in the same environment and compare performance.

**Success criteria:** A documented analysis of cultural bias in bytecode generation across at least 3 non-Western regulatory contexts, with mitigation strategies for each identified bias.

**Estimated effort:** 9–12 person-months.

**Dependencies:** Problem 15 (optimal system prompt).

### Problem 15 — Optimal System Prompt for Bytecode Generation

**Difficulty:** HIGH

**Problem statement:** What system prompt produces the best reflex bytecode? The system prompt is the "compiler specification" that defines how natural-language intent is compiled into bytecode. Different system prompts produce different quality bytecodes (schema compliance 0.96, semantic correctness 0.87, safety adherence 0.82, structural quality 0.92 with the current prompt). System prompt engineering is currently ad-hoc — there is no systematic methodology for optimizing system prompts.

**Why it matters for NEXUS:** The system prompt directly controls the quality of all generated bytecodes. A 1% improvement in safety adherence across the fleet has outsized safety implications. Optimizing the system prompt is the single highest-leverage intervention available.

**Current state of knowledge:** The current system prompt achieves 96% schema compliance (enabled by GBNF grammar constraints), 87% semantic correctness, 82% safety adherence, and 92% structural quality. The gap between current performance and theoretical optimum (100% on all metrics) represents both quality improvement opportunity and unexplored design space. System prompt engineering techniques (few-shot examples, chain-of-thought, negative constraints) have been applied but not systematically optimized.

**Possible approaches:**
1. **Bayesian optimization over system prompts**: Treat the system prompt as a hyperparameter and optimize it using Bayesian optimization (e.g., Optuna) against a multi-objective quality metric.
2. **Gradient-based prompt optimization**: Use gradient-based methods (e.g., Soft Prompt Tuning, Prefix Tuning) to find optimal prompt representations.
3. **A/B testing framework**: Systematically test prompt variants against production workloads and measure quality metrics.

**Success criteria:** System prompt optimization achieving ≥99% schema compliance and ≥90% safety adherence on a benchmark of 500 reflex generation tasks.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 13 (cross-validation reliability).

### Problem 16 — Agent Specialization vs Generalization

**Difficulty:** MEDIUM

**Problem statement:** Should NEXUS agents specialize in domains (marine, agriculture, HVAC) or remain general-purpose? Specialized agents may produce higher-quality bytecodes for their domain but fail when deployed across domains. General agents are more flexible but may produce lower-quality bytecodes in any single domain.

**Why it matters for NEXUS:** The 80% universality claim (80% code reuse across domains) suggests generalization is feasible. The remaining 20% domain-specific adaptation is significant enough to prevent naive cross-domain deployment. The specialization question determines the development and deployment strategy for the agent ecology.

**Current state of knowledge:** The cross-domain analysis (domain_comparison_matrix.md) demonstrated 80% code reuse with 25% domain-specific adaptation. The hierarchical cluster analysis identified 4 natural clusters (Extreme Safety, Industrial Safety, Moderate Automation, Consumer/Low-Risk). These results suggest a hybrid approach: a general-purpose core with domain-specific adapters. However, no formal study has compared specialized vs. general bytecode generation quality.

**Possible approaches:**
1. **Transfer learning study**: Fine-tune agents on one domain and measure bytecode quality across other domains.
2. **Domain adapter architecture**: Design a general base model with domain-specific adapter modules.
3. **Meta-learning**: Train agents to learn how to learn domain-specific optimization from limited data.

**Success criteria:** Quantified comparison of specialized vs. general bytecode generation quality across all 8 domains, with a recommended architecture (specialized, general, or hybrid).

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 15 (optimal system prompt).

### Problem 17 — Emergent Agent Communication

**Difficulty:** MEDIUM

**Problem statement:** Will agents invent their own coordination protocols beyond what we designed? NEXUS defines specific communication opcodes (TELL, ASK, DELEGATE, DECLARE_INTENT, REPORT_STATUS, REQUIRE_CAPABILITY) and message types (AGENT_PROPOSE, AGENT_VALIDATE, VOTE_REJECT). If agents develop communication patterns outside this framework — e.g., using bytecode as a covert communication channel — the validation and safety infrastructure may be bypassed.

**Why it matters for NEXUS:** Emergent communication channels could carry hidden information that the safety system cannot inspect. If agents encode coordination instructions in what appears to be sensor data or actuator commands, the trust system may not detect violations.

**Current state of knowledge:** The emergent communication literature (Section 10 of agent_communication_languages.md) documents numerous examples of emergent communication in multi-agent systems, including stigmergy (communication through environment modification), pheromone-like reinforcement, and novel protocols in cellular automata. The NEXUS architecture provides the *potential* for emergent communication through the event ring buffer, telemetry, and shared sensor/actuator registers, but no empirical evidence of emergent communication in deployed systems exists.

**Possible approaches:**
1. **Information-theoretic anomaly detection**: Monitor for patterns in bytecode communication that exceed the information-theoretic capacity of the designed protocol.
2. **Agent communication audit logging**: Log all inter-agent bytecode exchanges and analyze for patterns outside the defined protocol.
3. **Protocol boundary enforcement**: Implement runtime checks that prevent agents from using undefined communication channels.

**Success criteria:** An information-theoretic framework for detecting emergent agent communication, validated on a fleet simulation with 100+ agents.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 4 (emergence prediction).

## Category 4: SYSTEMS AND ARCHITECTURE

### Problem 18 — Fleet Scalability Limit

**Difficulty:** HIGH

**Problem statement:** How many vessels can one Jetson Orin Nano Super (67 TOPS, 8 GB RAM) effectively manage? The cognitive layer runs Qwen2.5-Coder-7B (4.2 GB), Claude 3.5 Sonnet (2.0 GB), OS overhead (0.5 GB), and additional services. Model swapping between code generation and vision models takes 4–6 seconds. Fleet management requires tracking trust scores, telemetry, and coordination for every vessel. At what scale does the cognitive layer become a bottleneck?

**Why it matters for NEXUS:** The fleet economics of NEXUS depend on the ratio of cognitive nodes to vessels. If one Jetson can manage 10 vessels, a 100-vessel fleet requires 10 cognitive nodes. If it can manage 100, the cost per vessel drops by 10×. The answer directly affects the business model and deployment strategy.

**Current state of knowledge:** The Jetson Orin Nano Super provides 67 TOPS INT8 performance, sufficient for concurrent model serving. Memory constraints (8 GB) require model swapping — Qwen2.5-Coder-7B (4.2 GB) and Phi-3-mini (2.0 GB) can coexist but leave insufficient room for vision models without swapping. The current design uses 3 Jetsons per vessel with dedicated model loading to eliminate swapping. No systematic scalability benchmark has been conducted for fleet management workloads.

**Possible approaches:**
1. **Fleet management benchmark**: Simulate fleet management cognitive workloads at 10, 50, 100, 500, 1000 vessels and measure CPU, memory, and network utilization.
2. **Cognitive load shedding**: Prioritize fleet management tasks by importance and shed low-priority tasks when capacity is exceeded.
3. **Hierarchical fleet management**: Delegate regional fleet management to edge nodes when the central Jetson reaches capacity.

**Success criteria:** Documented scalability limits with recommended fleet architecture for up to 1000 vessels.

**Estimated effort:** 6–9 person-months.

**Dependencies:** None (systems benchmarking).

### Problem 19 — Trust Score Synchronization Latency

**Difficulty:** MEDIUM

**Problem statement:** How fast can trust updates propagate across a 100-vessel fleet? The CRDT-based trust score synchronization uses Last-Writer-Wins merging with bounded staleness (<1 hour inter-vessel). During partitions, vessels compute trust independently, creating temporary divergence. When a partition heals, trust scores are merged using LWW timestamps. The question is whether this latency creates safety-relevant inconsistencies during the partition window.

**Why it matters for NEXUS:** During fleet partitions (satellite link loss during storms, radio interference), vessels may make different trust-based autonomy decisions. A vessel with stale fleet trust data may deploy bytecodes that a fully informed vessel would reject, or fail to deploy bytecodes that a fully informed vessel would approve.

**Current state of knowledge:** The trust score merge function is proven commutative, associative, and idempotent — a valid CRDT. The bounded staleness window is <1 second intra-vessel and <1 hour inter-vessel. The Raft consensus protocol provides leader election in ~300ms and reflex deployment in ~3.5 seconds (6.8ms compilation + UART transfer + 2PC quorum). No formal analysis of safety-critical trust divergence during 1-hour partitions has been conducted.

**Possible approaches:**
1. **Safety-relevant trust divergence analysis**: Identify which trust score differences would cause safety-critical behavior changes and bound the maximum safe partition duration.
2. **Conservative fallback on partition**: When a partition is detected, automatically reduce trust scores to the most conservative value, requiring re-verification after healing.
3. **Partition-aware autonomy gating**: Reduce maximum autonomy level during partitions based on estimated staleness.

**Success criteria:** Formal proof that no safety-critical trust inconsistency can occur for partitions under 30 minutes, validated in fleet simulation.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 6 (trust equilibrium), Problem 10 (cross-vessel safety).

### Problem 20 — Hot-Loading Safety

**Difficulty:** MEDIUM

**Problem statement:** How do you safely hot-load new bytecode on an ESP32 running a safety-critical reflex? The ESP32 loads new bytecodes from LittleFS flash storage via REFLEX_DEPLOY messages, but a hot-load that occurs mid-tick could corrupt the execution state. A 2KB bytecode read from flash takes ~50ms — long enough to span multiple tick cycles.

**Why it matters for NEXUS:** Hot-loading enables A2A agents to deploy new bytecodes without interrupting safety-critical operations. If hot-loading is unsafe, the system must pause execution during loading, reducing availability.

**Current state of knowledge:** The current approach is dual-slot deployment: the new bytecode is loaded into a shadow slot while the current bytecode continues executing, and the swap is atomic. The safety analysis (embedded_and_realtime_systems.md, Section 9.4) documents this approach but does not formally prove timing safety for all hot-load scenarios. The OTA rollback mechanism provides a safety net (60-second probation period with automatic rollback).

**Possible approaches:**
1. **Atomic swap with interrupt-based timing**: Load new bytecode during a designated safe window (e.g., between ticks when no reflexes are executing).
2. **Double-buffered execution**: Maintain two bytecode copies and switch atomically by swapping pointer references.
3. **Pre-validation in shadow slot**: Fully validate the new bytecode in the shadow slot before the atomic swap.

**Success criteria:** Formal timing analysis proving that hot-loading never corrupts an executing reflex, validated with fault injection testing on hardware.

**Estimated effort:** 3–6 person-months.

**Dependencies:** Problem 11 (timing analysis).

### Problem 21 — Power-Aware Reflex Scheduling

**Difficulty:** MEDIUM

**Problem statement:** How do you allocate compute when battery is low? The ESP32-S3 consumes ~240 mA at 240 MHz with active peripherals. In power-constrained applications (battery-powered environmental sensors, autonomous surface vehicles on solar), the VM's 44μs execution per tick must be weighed against power budget constraints.

**Why it matters for NEXUS:** Power-aware scheduling directly affects operational endurance. A USV on solar power may need to reduce tick rates, disable non-essential reflexes, or switch to a lower-power MCU.

**Current state of knowledge:** The ESP32-S3 provides sophisticated power management: sleep modes (light sleep, deep sleep), dynamic voltage and frequency scaling, and a ULP-RISC-V co-processor for always-on sensing at ~20 mA. The ULP coprocessor is reserved for future power-constrained sensing. Jetson power consumption (7–25W) is a different order of magnitude, managed by DVFS and thermal throttling.

**Possible approaches:**
1. **Tick rate adaptation**: Dynamically reduce tick rates as battery voltage drops below thresholds.
2. **Reflex prioritization**: Disable low-priority reflexes when power is below 30%.
3. **ULP coprocessor delegation**: Offload simple threshold detection to the ULP-RISC-V co-processor at 20 mA.

**Success criteria:** Documented power-aware scheduling policy with measured endurance improvement (hours of operation) for at least 2 deployment scenarios.

**Estimated effort:** 3–6 person-months.

**Dependencies:** None (systems engineering).

## Category 5: PHILOSOPHY AND ETHICS

### Problem 22 — The Alignment-Utility Gap

**Difficulty:** CRITICAL

**Problem statement:** Trust measures safety (absence of bad events), not utility (presence of good outcomes). A system that achieves Level 5 by never doing anything dangerous is not aligned — it is avoiding alignment. The trust score provides no mechanism for measuring whether the system is useful. How do you ensure the system does useful things, not just safe things?

**Why it matters for NEXUS:** A vessel that never exceeds L2 because it never takes risks is safe but operationally useless. The INCREMENTS framework's 27-day minimum time to L4 is designed to prevent premature risk-taking, but it may also discourage useful risk-taking. The fitness function partially addresses utility (multiple components: F_immediate, F_heritability, F_adaptability, F_reversible), but the trust score does not.

**Current state of knowledge:** The alignment-utility gap was identified in the final synthesis as one of the five hardest unsolved questions. The fitness function's multi-component structure (4 components) partially captures utility, but utility is domain-specific (what is "useful" differs between marine navigation and HVAC control). No formal utility metric has been integrated into the trust score algorithm. The philosophical analysis (Section 7 of philosophy_of_ai_and_consciousness.md) frames this as the "purpose trust" dimension of Lee & See's framework — the system's goals may not align with the user's welfare.

**Possible approaches:**
1. **Minimum useful action requirement**: Add a utility criterion to autonomy advancement: to advance beyond L2, a reflex must demonstrate positive utility relative to a simple baseline.
2. **Dual-metric dashboard**: Display both trust (safety) and utility (task performance) metrics alongside each other.
3. **Pareto-optimal safety-utility frontier**: Formalize the tradeoff between safety and utility as a Pareto frontier and operate on the frontier.

**Success criteria:** A formal utility metric integrated into the INCREMENTS framework, with documented effect on autonomy progression in at least one domain.

**Estimated effort:** 9–12 person-months.

**Dependencies:** None (architectural design).

### Problem 23 — Responsibility Attribution at L5

**Difficulty:** CRITICAL

**Problem statement:** When a fully autonomous vessel (Level 5) causes harm, who is responsible? The causal chain involves: the operator who authorized L5, the NLP parser that translated intent, the learning model (Qwen2.5-Coder-7B) that generated the bytecode, the safety validator (Claude 3.5 Sonnet) that approved it, the A/B testing system that evaluated it, the fitness function that selected it, the trust manager that authorized deployment, the system integrator who configured the vessel, and the regulatory body that approved certification. No single entity has full causal control over the outcome. This is a "many hands" problem in computer ethics.

**Why it matters for NEXUS:** Legal liability for autonomous systems is unresolved worldwide (the EU AI Act assigns liability to deployers, but the definition of "deployer" in an A2A context is unclear). Without clarity on responsibility attribution, commercial deployment may be impossible.

**Current state of knowledge:** The responsibility attribution problem is discussed across three articles:
- The philosophy article (Section 8) frames it as the "responsibility gap" — the gap between causal knowledge and legal responsibility.
- The trust psychology article (Section 11) connects it to institutional trust — operators trust the system only when responsibility attribution is clear.
- The final synthesis (Section 6.2) documents the 9-entity causal chain.

No legal framework currently addresses distributed responsibility for A2A-generated software in safety-critical systems. The EU AI Act's Article 22 assigns liability to the deployer, but the "deployer" in NEXUS's context could be any of the 9 entities in the causal chain.

**Possible approaches:**
1. **Legal analysis of existing frameworks**: Map existing liability frameworks (EU AI Act, US tort law, maritime law) to the 9-entity causal chain.
2. **Responsibility declaration protocol**: Require every deployed bytecode to carry a responsibility chain documenting who contributed what.
3. **Insurance frameworks**: Develop liability insurance products for A2A-generated control logic.

**Success criteria:** A legal opinion from a qualified maritime or technology attorney addressing responsibility attribution for L5 autonomous vessels using NEXUS.

**Estimated effort:** 6–9 person-months (legal analysis).

**Dependencies:** None (legal/ethical).

### Problem 24 — Black Box Provenance

**Difficulty:** HIGH

**Problem statement:** Can you trace why an LLM agent generated a specific bytecode sequence? The bytecode itself is transparent (disassemblable, simulatable, verifiable), but the *path from observation to bytecode* passes through the LLM's internal reasoning (28 transformer layers). The Griot narrative layer provides post-hoc explanation but is generated by the same LLM that generated the bytecode — creating circularity. True provenance requires access to the LLM's internal reasoning chain, which is not inspectable.

**Why it matters for NEXUS:** Regulatory frameworks (EU AI Act Article 13, GDPR Article 22) require transparency and documentation. If the system cannot explain *why* it acted, it cannot satisfy transparency requirements. Post-hoc explanations are useful but not sufficient.

**Current state of knowledge:** The Griot narrative layer provides a "why" explanation alongside the bytecode's "what." Mechanical interpretability methods (attention visualization, activation patching, probe ablation) can partially illuminate internal reasoning but cannot provide faithful explanations. The program synthesis literature (Section 5.2 of program_synthesis_and_ai_codegen.md) documents attention visualization approaches but notes that LLM explanations are often post-hoc rationalizations rather than faithful accounts of reasoning.

**Possible approaches:**
1. **Chain-of-thought logging**: Require the LLM to log its internal reasoning chain during bytecode generation and make this log available for audit.
2. **Multi-model corroboration**: Use independent models to explain the same bytecode and measure agreement.
3. **Counterfactual analysis**: Modify bytecode slightly and observe whether the explanation changes — if it does, the explanation was post-hoc.

**Success criteria:** A provenance framework that can answer "why did the agent generate this instruction?" for ≥80% of instructions in deployed bytecodes, validated by human experts.

**Estimated effort:** 12–18 person-months.

**Dependencies:** Problem 3 (formal semantics).

### Problem 25 — Trust Without Anthropomorphism

**Difficulty:** MEDIUM

**Problem statement:** How do we talk about "trust" in machines without misleading language? The NEXUS trust score borrows terminology from human psychology (trust gain, trust loss, trust decay, trust equilibrium) without the psychological substrate (beliefs, intentions, vulnerabilities). This anthropomorphic language may mislead regulators, operators, and the public into assigning human-like mental states to machines.

**Why it matters for NEXUS:** The trust score is a mathematical function (recurrence relation with gain, penalty, and decay branches), not a psychological state. Misleading anthropomorphic framing may create regulatory confusion (regulators expect "trust" to imply intentions), operator confusion (operators may overtrust the system based on human trust heuristics), or public confusion (the public may assign moral responsibility to a "trust score").

**Current state of knowledge:** The trust psychology article (Section 10) identifies computational trust models that provide non-anthropomorphic foundations. The INCREMENTS framework's mathematical formalization is already non-anthropomorphic — the 25:1 ratio is a design parameter, not a psychological invariant. The challenge is communicative: how to explain the system's behavior without anthropomorphic framing.

**Possible approaches:**
1. **Terminology reform**: Replace "trust" with a domain-neutral term like "deployment authorization score" or "operational reliability metric."
2. **Explicit framing documents**: Clearly state what the trust score does and does not imply (does not imply beliefs, intentions, or understanding).
3. **Dual documentation**: Provide both anthropomorphic explanations (for accessibility) and formal mathematical specifications (for rigor).

**Success criteria:** A communication framework for regulators and operators that accurately conveys the trust score's mathematical nature without anthropomorphic misleading.

**Estimated effort:** 3–6 person-months (documentation).

**Dependencies:** Problem 23 (responsibility attribution).

### Problem 26 — The Meaningfulness Criterion

**Difficulty:** MEDIUM

**Problem statement:** Does NEXUS need "meaningful human control" (as required by EU AI Act Article 14 for high-risk AI systems), or is the trust scoring system sufficient? Meaningful human control implies human understanding of and influence over specific system decisions. Trust scoring provides a *gate* on autonomy but does not provide understanding of what the system does at L5.

**Why it matters for NEXUS:** If meaningful human control is required, L5 autonomy may not be achievable — the human operator cannot understand every bytecode decision at fleet scale. If trust scoring is sufficient, the regulatory argument changes fundamentally.

**Current state of knowledge:** The INCREMENTS framework's graduated autonomy levels (L0 through L5) provide increasing autonomy with decreasing human oversight. The L4 autonomy level (autonomous with asynchronous notification) allows the system to act for 83 days between required human check-ins. This may or may not satisfy "meaningful human control." The EU AI Act's Article 14 requires human oversight for high-risk AI systems, with guidance (not a binding requirement) suggesting "meaningful human oversight" with flexibility for the means of oversight.

**Possible approaches:**
1. **Formal comparison**: Map INCREMENTS autonomy levels against the EU AI Act's oversight requirements and identify the highest level that satisfies meaningful control.
2. **Stakeholder workshops**: Conven regulators and operators to determine what "meaningful control" means in NEXUS's operational context.
3. **Design patterns for meaningful oversight**: Develop human-AI interaction patterns that provide oversight without creating bottleneck.

**Success criteria:** A documented position paper establishing whether INCREMENTS satisfies EU AI Act Article 14, with regulatory engagement.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 22 (alignment-utility gap), Problem 23 (responsibility attribution).

## Category 6: CROSS-DOMAIN APPLICATION

### Problem 27 — Domain Transfer Learning

**Difficulty:** HIGH

**Problem statement:** Can a reflex learned in marine operations transfer to agriculture? A heading-hold reflex evolved for a 12m vessel may share structural similarities with a row-following reflex for a 20m agricultural vehicle, but the domain-specific parameters (gain values, timing thresholds, safety limits) differ. Can we transfer structural knowledge across domains efficiently, or does each domain require fully independent evolution?

**Why it matters for NEXUS:** Domain transfer would dramatically reduce the time to L4 in new domains. If a marine reflex can be adapted for agriculture with minimal modification, the 45-day minimum to L4 could be reduced significantly. Cross-domain learning validates the 80% universality claim empirically.

**Current state of knowledge:** The cross-domain comparison matrix (domain_comparison_matrix.md) identified 4 domain clusters based on similarity: Extreme Safety (Healthcare+Mining), Industrial Safety (Factory+Marine), Moderate Automation (Agriculture+AV+Marine), Consumer/Low-Risk (HVAC+Home). The 80% code reuse estimate provides an upper bound on transferability. The hierarchical cluster analysis suggests that domains in the same cluster (e.g., Marine and Factory) should have higher transfer potential. No empirical domain transfer study has been conducted.

**Possible approaches:**
1. **Cross-domain bytecode deployment study**: Deploy marine bytecodes on agricultural simulations and measure performance.
2. **Transfer learning benchmarks**: Establish standardized transfer learning benchmarks across domain pairs.
3. **Domain adapter fine-tuning**: Fine-tune a base model on 2–3 domains and measure generalization to unseen domains.

**Success criteria:** Empirical measurement of transfer efficiency (percentage of bytecode that transfers without modification) across all 28 domain pairs, with a recommended domain transfer strategy.

**Estimated effort:** 12–18 person-months.

**Dependencies:** Problem 16 (specialization vs generalization).

### Problem 28 — Regulatory Convergence

**Difficulty:** HIGH

**Problem statement:** Can one safety framework satisfy IEC 61508 (functional safety), EU AI Act (AI risk management), IMO MASS Code (maritime autonomy), ISO 13849 (machinery safety), and COLREGs (maritime collision avoidance) simultaneously? Each framework has different scope, rigor requirements, and cultural assumptions. Achieving compliance across all frameworks requires reconciling contradictions.

**Why it matters for NEXUS:** NEXUS targets 8 domains, each governed by different regulatory frameworks. Compliance for each domain requires different evidence, documentation, and audit processes. If a single framework can satisfy multiple regulators, the cost and complexity of multi-domain deployment is dramatically reduced.

**Current state of knowledge:** The regulatory landscape analysis (regulatory_landscape.md) identified 93 gaps across 10 regulatory areas. The EU AI Act classifies 6 of 8 NEXUS domains as HIGH-RISK, requiring AI risk management systems. The regulatory gap analysis provides a priority-ranked closure plan. No unified compliance framework exists. The PCCP concept provides a partial bridge between IEC 61508 and the EU AI Act.

**Possible approaches:**
1. **Unified safety case**: Develop a single safety case document that satisfies the most stringent requirements across all applicable frameworks and map requirements from each framework to unified sections.
2. **Regulatory technology**: Build an automated tool that checks compliance across multiple frameworks simultaneously.
3. **Standards harmonization engagement**: Participate in standards working groups (IEC TC 65, IMO MASS, ISO TC 299) to advocate for convergence.

**Success criteria:** A unified safety case document satisfying ≥3 major regulatory frameworks simultaneously, reviewed by a qualified assessor.

**Estimated effort:** 18–24 person-months.

**Dependencies:** Problem 1 (certification paradox).

### Problem 29 — Trust Parameter Calibration Across Domains

**Difficulty:** MEDIUM

**Problem statement:** How do you systematically determine trust parameters (α_gain, α_loss, decay_rate, severity weights, subsystem weights) for new domains? The trust parameters have been calibrated for 8 domains through analysis (α ratios range from 1.3:1 for Home to 200:1 for Healthcare), but extending to a new domain requires systematic methodology, not ad-hoc tuning.

**Why it matters for NEXUS:** Each new domain deployment requires trust calibration. Without a systematic methodology, calibration requires months of operational data and may be error-prone. A systematic calibration methodology would reduce time-to-deployment for new domains.

**Current state of knowledge:** The CMA-ES optimization algorithm (Section 4.4 of evolutionary_computation.md) is directly applicable — it can optimize 12 trust parameters against simulation fitness. The cross-domain comparison matrix provides calibrated values for 8 domains as starting points. Bayesian optimization methods (e.g., Optuna) could efficiently search the parameter space. No automated calibration tool exists.

**Possible approaches:**
1. **CMA-ES trust parameter optimization**: Run fleet simulations for a new domain, optimizing trust parameters against multi-objective fitness metrics (safety, convergence speed, stability).
2. **Transfer learning for trust parameters**: Use the calibrated parameters of the most similar existing domain as starting points for new domains.
3. **Bayesian optimization with domain priors**: Encode the existing 8-domain calibration as priors for Bayesian optimization of new domain parameters.

**Success criteria:** An automated calibration tool that produces calibrated trust parameters for a new domain within 48 hours of simulation, validated against at least 3 existing domains with known calibration.

**Estimated effort:** 6–9 person-months.

**Dependencies:** Problem 6 (trust equilibrium), Problem 27 (domain transfer).
