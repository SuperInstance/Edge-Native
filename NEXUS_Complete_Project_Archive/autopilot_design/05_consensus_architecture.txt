RESOLUTION DETAIL FOR #1 (PID Sample Rate):
  Brief 01 (Marine PID Engineering) provides rigorous analysis showing:
  - Hydraulic systems have 1-5 Hz bandwidth
  - NMEA compass data updates at 5-10 Hz
  - 10 Hz provides negligible phase lag (<10° at vessel natural frequency)
  - >20 Hz provides no benefit
  Brief 02 (ESP32 Architecture) assumed 100 Hz for oversampling, but this is
   wasteful given the physical plant bandwidth. The hydraulic solenoid valves
  cannot respond at 100 Hz (minimum pulse width 100ms). Running PID at 100 Hz
  would compute output values 9 out of 10 times with no new sensor data.
  DECISION: PID loop runs at 10 Hz. Rudder feedback reads at 50 Hz (oversampled
  for smooth rate limiting). Compass reads at 10 Hz.

RESOLUTION DETAIL FOR #2 (MOSFET vs Relay):
  Brief 03 (Hydraulic Control) provides exhaustive comparison:
  - MOSFET: <1μs switching, no contact wear, no bounce, silent, PWM-capable
  - Relay: 5-15ms switching, 100K-1M cycle life, contact bounce causes EMI
  - MOSFET IRLZ44N is logic-level, handles 47A (solenoids draw 2-3A), $1.50 each
  - MOSFET enables future proportional valve upgrade without hardware change
  DECISION: IRLZ44N N-channel MOSFET per channel (PORT, STBD, CLUTCH) with
  100Ω series gate resistor, 10KΩ gate pull-down, UF4007 flyback diode.

RESOLUTION DETAIL FOR #3 (Rudder Feedback):
  Brief 03 strongly recommends AS5048A (non-contact, infinite life, 0.05° accuracy)
  for marine use. Potentiometers are the #1 failure mode in marine autopilots.
  The marginal cost ($8-12 vs $5-15) is negligible for the reliability gain.
  Brief 02's potentiometer path (ADC1_CH5, GPIO6) is RETAINED as a fallback
  for vessels with existing potentiometer installations.
  DECISION: AS5048A on I2C0 at address 0x40 (primary). Potentiometer on
  ADC1_CH5 GPIO6 (secondary, selectable via config).

2. FINAL COMPONENT SELECTIONS
================================================================================

  MCU:        ESP32-S3-WROOM-1-N8R8 (512KB SRAM, 8MB PSRAM, 8MB Flash)
  RTOS:       FreeRTOS 10.5.1 (ESP-IDF v5.2+)

  COMPASS:    BNO085 (CEVA Hillcrest SH-2 Sensor Hub)
              I2C Address: 0x28, Bus: I2C0, Speed: 400 kHz
              Fused 9-DOF heading output, ±2° typical accuracy, <1°/hr drift
              Alternative (if vessel has existing unit): NMEA fluxgate on UART1

  RUDDER:     AS5048A Magnetic Rotary Encoder (PRIMARY)
              I2C Address: 0x40, Bus: I2C0, 14-bit, ±0.05° accuracy
              Alternative: 10KΩ linear potentiometer on ADC1_CH5 (GPIO6)

  SOLENOID DRIVER: IRLZ44N N-channel Logic-Level MOSFET (×3: PORT, STBD, CLUTCH)
              Vgs(th) 1-2V, Id(max) 47A, Rds(on) 22mΩ @ 5V, TO-220 package
              Gate drive: 100Ω series resistor from ESP32 GPIO
              Gate pull-down: 10KΩ to GND on each MOSFET
              Flyback diode: UF4007 ultra-fast recovery (1A, 1000V) per solenoid
              EMI suppression: 100nF ceramic cap across each solenoid terminal

  CURRENT SENSE: ACS712-5A Hall Effect Current Sensor (×2: drive, clutch)
              Analog output → ESP32 ADC1 channels
              Sensitivity: 185 mV/A, measures up to 5A DC
              Used for: overcurrent detection, stall detection, clutch engagement verify

  DISPLAY:    SSD1306 OLED 128×64, I2C Address: 0x3C, Bus: I2C0
              3.3V, shares I2C0 with compass and rudder encoder (mutex protected)

  POWER:
              Input: 10-16V DC (boat 12V system, with alternator charging)
              ESP32: LM1117-3.3 LDO (3.3V, 800mA max)
              5V rail: LM2596 buck converter (12V→5V, for solenoid MOSFET Vgs)
              Protection: 5A automotive fuse on main input, TVS diode for load dump
              Decoupling: 100µF + 10µF + 100nF on ESP32 3.3V rail

  RS-422:     SN75176BP differential transceiver (×2: NMEA input, Jetson link)
              UART1: NMEA 0183 input (compass/GPS) — 4800 baud
              UART2: Jetson communication — 115200 baud

  WATCHDOG:   MAX6818 external hardware watchdog (1.6s timeout)
              Kicked by ESP32 GPIO1 every 100ms
              Resets ESP32 on timeout (complete power cycle)

  BUZZER:     Piezo buzzer via MOSFET driver on GPIO2 (active HIGH)
              PWM-capable for beep patterns (alarm, confirmation, warning)

  Note: task_gps_nmea_reader is merged into task_nmea_parser. GPS sentences
  (RMC, GGA) are parsed there directly. GPS data also arrives via Jetson.

  task_current_monitor is NEW (not in Brief 02). Required by Brief 03 for

  PRIORITY ASSIGNMENT RATIONALE:
    24  PID control      — Never blocks, never misses deadline. WCET ~2.5ms.
    23  Watchdog         — Detects PID failure in <50ms. Kicks external WDT.
    20  NMEA parser      — Handles 4800 baud input, highest throughput task.
    19  Serial Jetson    — Bidirectional command/telemetry with AI interface.
    18  Compass/Rudder   — Sensor acquisition feeds PID loop.
    17  Current monitor — Safety monitoring, feeds watchdog.
    15  Button handler   — Human input, 50ms debounce masks jitter.
    12  OLED display     — 5 Hz refresh, no timing criticality.
     8  MQTT client      — Background telemetry, lowest priority.
     1  Idle task        — FreeRTOS minimum.

5. JSON COMMUNICATION PROTOCOL (ESP32 <-> JETSON)
================================================================================

  Physical: UART2, RS-422, 115200 baud, 8N1
  Framing:  JSON object + '\n' + '*XXXX' (CRC-16/CCITT-FALSE hex)
  Max frame: 256 bytes

  --- JETSON -> ESP32 COMMANDS ---

  1. handshake
     { "msg_id":N, "ts":N, "cmd":"handshake",
       "params":{"jetson_version":"x.y.z","protocol_version":"2.0",
       "capabilities":["set_heading","engage","track","gps"]} }
     Response: handshake_ack

  2. set_heading
     { "msg_id":N, "ts":N, "cmd":"set_heading",
       "params":{"heading":127.5,"source":"jetson","reason":"user_command"} }
     Validation: heading 0-359.9, must be engaged or include "engage":true

  3. engage
     { "msg_id":N, "ts":N, "cmd":"engage",
       "params":{"mode":"auto","heading":127.5} }
     Pre-conditions: compass valid, rudder valid, no critical alarms
     Actions: energize clutch, wait 500ms, verify engagement, capture heading

  4. standby
     { "msg_id":N, "ts":N, "cmd":"standby", "params":{} }
     Actions: PID off, integral=0, solenoids off, de-energize clutch

  5. set_mode
     { "msg_id":N, "ts":N, "cmd":"set_mode",
       "params":{"mode":"track"} }
     Modes: "standby", "auto", "track", "wind"

  6. set_gains
     { "msg_id":N, "ts":N, "cmd":"set_gains",
       "params":{"P":0.003,"I":0.0,"D":0.09,"DD":0.075,"PR":0.005,"FF":0.6} }
     All gains >=0, bounded ranges per firmware_config.json

  7. get_status
     { "msg_id":N, "ts":N, "cmd":"get_status", "params":{} }
     Response: full status message

  8. get_compass
     { "msg_id":N, "ts":N, "cmd":"get_compass", "params":{} }

  9. get_rudder
     { "msg_id":N, "ts":N, "cmd":"get_rudder", "params":{} }

  10. calibrate
      { "msg_id":N, "ts":N, "cmd":"calibrate",
        "params":{"type":"compass","action":"start"} }
      Types: "compass", "rudder_center", "rudder_limits"
      Actions: "start", "stop", "status"

  11. heartbeat
      { "msg_id":N, "ts":N, "cmd":"heartbeat", "params":{} }
      ESP32 resets Jetson heartbeat timer on receipt.

  --- ESP32 -> JETSON TELEMETRY ---

  1. heading (10 Hz)
     { "msg_id":N, "ts":N, "cmd":"telemetry",
       "params":{"heading":127.5,"heading_raw":128.1,"rate_of_turn":0.3,
       "source":"bno085","quality":"high","cal_status":"calibrated"} }

  2. rudder (10 Hz)
     { "msg_id":N, "ts":N, "cmd":"telemetry",
       "params":{"rudder_angle":5.2,"rudder_commanded":5.0,"sensor":"as5048a"} }

  3. pid_state (10 Hz when engaged)
     { "msg_id":N, "ts":N, "cmd":"telemetry",
       "params":{"mode":"auto","engaged":true,"heading_error":-2.3,
       "p_term":3.45,"i_term":0.12,"d_term":-0.8,"dd_term":0.05,
       "pr_term":0.11,"ff_term":0.0,"output":3.02} }

  4. status (1 Hz)
     { "msg_id":N, "ts":N, "cmd":"status",
       "params":{"mode":"auto","engaged":true,"heading":127.5,
       "target_heading":130.0,"rudder":5.2,"speed_stw":5.3,
       "clutch_engaged":true,"compass_valid":true,"link_up":true,
       "gps_fix":true,"alarms":[],"uptime_s":3600,"voltage":12.6} }

  5. alarm (event-driven)
     { "msg_id":N, "ts":N, "cmd":"alarm",
       "params":{"type":"COMPASS_FAIL","severity":"critical",
       "message":"No compass data for 500ms","action":"disengage"} }

  6. ack (response to all commands)
     { "msg_id":N, "ts":N, "cmd":"ack", "params":{"status":"ok"},
       "ref_id":<original_msg_id> }
     Or error:
     { "msg_id":N, "ts":N, "cmd":"ack", "params":{"status":"error",
       "error_code":"INVALID_HEADING","message":"heading must be 0-359.9"},
       "ref_id":<original_msg_id> }

  ERROR CODES:
    INVALID_HEADING, NOT_ENGAGED, COMPASS_FAIL, RUDDER_FAIL,
    CLUTCH_FAIL, OVERCURRENT, INVALID_MODE, GAINS_OUT_OF_RANGE,
    CALIBRATION_IN_PROGRESS, UNKNOWN_COMMAND

6. COMPLETE SAFETY SYSTEM
================================================================================

  LAYER 1: HARDWARE FAIL-SAFE (always active, no firmware dependency)
    - Solenoid valve spring-center: power loss → valve centers → rudder holds
    - MOSFET gate pull-down resistors: ESP32 boot/reset → solenoids OFF
    - External watchdog (MAX6818): 1.6s timeout → ESP32 hard reset
    - Automotive fuse (5A) on main 12V input
    - TVS diode on 12V input for load dump protection
    - Flyback diodes (UF4007) on all three solenoid coils

  LAYER 2: FIRMWARE SAFETY (managed by task_watchdog, priority 23)
    - PID loop heartbeat: checked every 100ms, timeout 50ms
      → CRITICAL ALARM, disengage, buzzer
    - Compass freshness: checked every 100ms, timeout 500ms
      → WARNING, "COMPASS FAIL", disengage if engaged
    - Jetson heartbeat: checked every 100ms, timeout 2000ms
      → WARNING, "LINK LOST" (autopilot continues on last heading)
    - Rudder feedback freshness: checked every 100ms, timeout 200ms
      → WARNING, "RUDDER FAIL"
    - Clutch current: checked every 50ms by task_current_monitor
      → If commanded ON but current < 0.5A: CLUTCH FAULT alarm
      → If current > 4.0A: OVERCURRENT alarm, disengage

  LAYER 3: SOFTWARE INTERLOCKS
    - NEVER energize PORT and STBD simultaneously (hardware impossible with
      spring-center valve, but software enforces too)
    - NEVER drive PORT/STBD unless clutch confirmed engaged (current > 0.5A)
    - If clutch disengages during operation: immediately kill PORT/STBD
    - Integral accumulator reset on mode change, engage, standby
    - Heading error clamped to ±60° before PID computation
    - Rudder command rate-limited to configurable max (default 6°/sec)
    - Solenoid minimum off-time: 200ms (prevent overheating)
    - Solenoid maximum on-time: 15 seconds (configurable)
    - If solenoid on > max_on_time: force OFF for cooldown period

  LAYER 4: ALARM HIERARCHY
    Severity: CRITICAL → WARNING → INFO

    CRITICAL (auto-disengage, buzzer, red LED):
      - PID loop missed deadline (>50ms)
      - Compass data lost for >500ms
      - Overcurrent detected (>4.0A drive, >3.0A clutch)
      - Supply voltage <10.0V or >16.0V
      - Rudder stall (current high, no rudder movement for >2s)

    WARNING (display alert, amber LED):
      - Jetson link lost (>2s no heartbeat)
      - Compass data stale (>300ms)
      - Heading error >20° while engaged
      - GPS fix lost (in track mode)
      - Supply voltage <11.0V or >15.0V
      - WiFi disconnected (if MQTT enabled)
      - Rudder approaching end stop (>30°)

    INFO (display note, green LED):
      - Mode changed
      - Heading adjusted
      - Calibration started/completed
      - Profile switched

  LAYER 5: FAIL-SAFE BEHAVIORS
    On CRITICAL alarm:
      1. Set engaged = false
      2. Set PORT = OFF, STBD = OFF (immediate)
      3. Wait 100ms
      4. Set CLUTCH = OFF (de-energize)
      5. Reset PID integral to 0
      6. Sound 3-beep alarm pattern (500ms on, 200ms off, repeat)
      7. Flash "ALARM" + alarm type on OLED
      8. Send alarm message to Jetson
      9. Red LED solid ON

    On power loss (total):
      - All MOSFET gates pulled LOW by 10KΩ resistors
      - Spring-center valve returns to center
      - Rudder holds position via hydraulic lock
      - Clutch de-energizes → helm free for manual steering

7. MODES OF OPERATION AND STATE MACHINE
================================================================================

  STATES:
    STANDBY  — Autopilot disengaged, clutch off, manual steering
    ENGAGING — Transition state, clutch energizing, 500ms wait
    AUTO     — Heading hold mode, PID active
    TRACK    — Course hold mode, heading command from APB/Jetson
    WIND     — Apparent wind angle hold, PID on AWA error
    ALARM    — Fault state, autopilot disengaged, alarm active

  STATE TRANSITIONS:

    STANDBY -> ENGAGING:
      Trigger: engage command (Jetson or button), OR AUTO button press
      Conditions: compass valid, rudder valid, no critical alarms
      Actions: energize clutch, start 500ms timer
      Reject if: compass fail, rudder fail, critical alarm active

    ENGAGING -> AUTO:
      Trigger: 500ms timer expired + clutch current > 0.5A
      Actions: capture current heading as target_heading, enable PID
      Fallback: If clutch current < 0.5A -> ALARM (CLUTCH_FAULT)

    ENGAGING -> STANDBY:
      Trigger: clutch engagement failed, standby command, or 2s timeout

    AUTO -> STANDBY:
      Trigger: standby command (Jetson or button), critical alarm
      Actions: PID off, integral=0, solenoids off, wait 100ms, clutch off

    AUTO -> TRACK:
      Trigger: set_mode "track", valid $APB sentence received, or Jetson
               sends track command with waypoint data
      Actions: switch PID input from manual heading to APB heading-to-steer
               field 12 (or Jetson-provided heading command)

    TRACK -> AUTO:
      Trigger: set_mode "auto", $APB sentences stop, or manual heading command
      Actions: switch PID input to last heading target

    AUTO -> WIND:
      Trigger: set_mode "wind", valid $MWV sentence received
      Actions: PID now controls on AWA error, not heading error

    WIND -> AUTO:
      Trigger: set_mode "auto", or wind sensor data lost
      Actions: return to heading hold with current heading as target

    ANY -> ALARM:
      Trigger: critical alarm condition detected
      Actions: immediate disengage (see Layer 5 above)

    ALARM -> STANDBY:
      Trigger: alarm acknowledged by user (long press MODE button),
               or alarm condition cleared and timeout expired

  BUTTON BEHAVIOR IN EACH STATE:
    STANDBY:
      AUTO/STBY -> ENGAGING (if conditions met)
      PORT/STBD ±1/±10 -> No effect (not engaged)
      MODE short -> No effect
      MODE long -> Enter calibration menu on OLED

    AUTO:
      AUTO/STBY -> STANDBY
      PORT -1° -> target_heading -= 1
      PORT -10° -> target_heading -= 10
      STBD +1° -> target_heading += 1
      STBD +10° -> target_heading += 10
      MODE short -> Cycle: AUTO -> TRACK -> WIND -> AUTO
      MODE long -> Enter settings menu on OLED

    ALARM:
      Any button -> ALARM -> STANDBY (acknowledge alarm)

8. NMEA 0183 SENTENCE HANDLING
================================================================================

  INPUT SENTENCES (parsed from UART1 at 4800 baud):

  $HDM — Heading Magnetic (PRIMARY heading source if no I2C compass)
    Fields: heading(0-359.9), M
    Action: Update shared_heading_raw
    Validation: heading 0-359.9, checksum valid
    Rate: 5-10 Hz

  $HDT — Heading True
    Fields: heading(0-359.9), T
    Action: Convert to magnetic (subtract variation), update shared_heading_raw
    Note: Requires magnetic variation value (from $RMC or config)

  $HDG — Heading, Deviation, Variation
    Fields: heading, deviation, variation (all in degrees)
    Action: Apply deviation correction, use variation for true<->magnetic
    This is the PREFERRED sentence — contains deviation and variation

  $RMC — Recommended Minimum Navigation
    Fields: status, lat, lon, SOG, COG, date, mag_variation
    Action: Extract SOG, COG, position, magnetic variation
    Used for: GPS fix validation, speed data, variation update

  $GGA — GPS Fix Data
    Fields: fix_quality, lat, lon, HDOP, alt, geoid_sep
    Action: Extract fix quality, position, HDOP
    Used for: GPS fix validation in track mode

  $APB — Autopilot Sentence B (CRITICAL for track mode)
    Fields: 14 fields including heading-to-steer (field 12), XTE, waypoint ID
    Action: Extract field 12 (heading to steer, magnetic) as target heading
             when in TRACK mode
    Switch to TRACK mode when valid $APB received with status 'A'
    Validation: Both status fields must be 'A' (data valid)

  $RMB — Recommended Minimum Navigation (LEGACY, for older chart plotters)
    Fields: status, XTE, direction, origin_wp, dest_wp, bearing, range
    Action: Used as fallback if $APB not available
    Compute heading command from bearing to destination

  $VHW — Water Speed and Heading
    Fields: water_speed_kts, heading_true, heading_mag
    Action: Extract water speed for gain scheduling
    CRITICAL: Water speed (not GPS SOG) determines rudder authority

  $MWV — Wind Speed and Angle
    Fields: wind_angle, R/T, wind_speed, K/M/N, status
    Action: Extract apparent wind angle and speed for WIND mode
    Required for wind mode operation

  OUTPUT SENTENCES (transmitted on UART1 TX at 4800 baud):

  $AGHDM — Heading Magnetic (talker ID: AG = Autopilot)
    $AGHDM,127.5,M*2A
    Purpose: Provide heading data to other instruments (chart plotter, repeaters)

  $AGRSA — Rudder Sensor Angle
    $AGRSA,5.2,A,5.2,A*3F
    Fields: starboard_rudder_angle, valid, port_rudder_angle, valid
    Purpose: Provide rudder position to instruments

  Note: Output is optional and configurable. Disable if not needed to
  reduce bus traffic.

  CHECKSUM VALIDATION:
    All input sentences validated against NMEA checksum (*XX format)
    XOR of all characters between $ and * (exclusive) must match hex value
    Invalid checksum: sentence discarded, error counter incremented
    >10 invalid sentences in 5 seconds: NMEA_BUS_ERROR warning

9. PID CONTROLLER DESIGN (DEFINITIVE)
================================================================================

  GAIN STRUCTURE (Extended PID from pypilot — proven in >2500 installations):

    Gain | Name            | Default | Range    | Description
    -----|-----------------|---------|----------|---------------------------
    P    | Proportional    | 0.003   | 0-0.03   | deg rudder per deg heading error
    I    | Integral        | 0.000   | 0-0.05   | Accumulated heading error (OFF by default)
    D    | Derivative      | 0.090   | 0-0.24   | Rate of turn (gyro-derived, primary gain)
    DD   | Derivative'     | 0.075   | 0-0.24   | Rate of rate of turn (reaction time)
    PR   | Proportional Root| 0.005  | 0-0.02   | sqrt(|error|)*sign — reduces overshoot
    FF   | Feed Forward    | 0.600   | 0-2.40   | Response to heading command changes

  FORMULA:
    command = P*e + I*INT(e) + D*(-dh/dt) + DD*(d²h/dt²) + PR*sqrt(|e|)*sign(e) + FF*Delta_heading_cmd

  WHERE:
    e = heading_error = normalize(target - actual) to [-180, +180]
    dh/dt = rate of turn from gyro (LOW-PASS FILTERED, cutoff ~0.3 Hz)
    d²h/dt² = rate of rate of turn (LOW-PASS FILTERED)
    Delta_heading_cmd = change in heading command (non-zero only during course changes)

  ANTI-WINDUP (5 layers):
    1. Heading error CLAMPED to ±60° before PID computation
    2. Integral accumulator CLAMPED to ±5 degree-seconds
    3. Conditional integration: only accumulate when |e| < 15° AND engaged > 5s
    4. Integral RESET on mode change, engage, standby
    5. Integral DECAY: INT *= (1 - 0.001 * dt) prevents indefinite accumulation

  DERIVATIVE ON MEASUREMENT:
    u_D = -D * d(heading_actual)/dt  (not d(error)/dt)
    This prevents "derivative kick" when heading command changes.

  DEADBAND (Linear Ramp):
    gain_factor = 0.0                          if |e| < 0.5°
    gain_factor = (|e| - 0.5) / (2.0 - 0.5)   if 0.5° <= |e| < 2.0°
    gain_factor = 1.0                          if |e| >= 2.0°
    u = gain_factor * PID(e)

  GAIN SCHEDULING (Speed-Based):
    When water speed (STW from $VHW) is available:
      speed_factor = (STW / V_ref)²  where V_ref = 5.0 kts
      Apply to P gain: P_eff = P * speed_factor
      Apply to D gain: D_eff = D * speed_factor
      Apply to PR gain: PR_eff = PR * speed_factor
      Do NOT schedule I or DD (maintains damping and steady-state)

  RATE LIMITING:
    Heading command rate: max 5.0°/sec (configurable for vessel type)
    Rudder command rate: max 6.0°/sec (configurable, protects hydraulics)
    Applied AFTER PID output, BEFORE solenoid command

  SOLENOID COMMAND FROM PID OUTPUT:
    PID output range: approximately -1.0 to +1.0 (normalized rudder command)
    Map to solenoid state:
      if output > deadband_threshold:
          STARBOARD solenoid ON
          PORT solenoid OFF
      elif output < -deadband_threshold:
          PORT solenoid ON
          STARBOARD solenoid OFF
      else:
          Both OFF (within deadband)

    Minimum pulse width: 100ms
    Minimum off time: 200ms
    Maximum continuous on: 15000ms

  SAMPLE RATE: 10 Hz (100ms period)
    Compass reads at 10 Hz. PID runs on each new compass reading.
    Rudder feedback reads at 50 Hz (used for rate limiting, not PID input).
    Current monitoring at 50 Hz (used for safety, not PID input).

10. COMPLETE WIRING SCHEMATIC DESCRIPTION
================================================================================

  POWER WIRING:
    Boat 12V Bus (+) -> 5A Automotive Fuse -> TVS Diode (SM8S36A, 36V clamping)
                       -> LM2596 12V->5V Buck (for MOSFET gate drive)
                       -> LM1117-3.3 LDO (for ESP32 and 3.3V sensors)
    Boat 12V Bus (-) -> Common Ground -> ESP32 GND

  ESP32-S3 WROOM MODULE:
    Pin 1 (3V3)  -> Decoupling: 100µF + 10µF + 100nF to GND
    Pin 2 (GND)  -> Common Ground
    Pin 3 (EN)   -> 10KΩ pull-up to 3.3V
    Pin 8 (IO7)  -> I2C0 SCL -> 4.7KΩ pull-up to 3.3V
    Pin 9 (IO8)  -> I2C0 SDA -> 4.7KΩ pull-up to 3.3V
    Pin 10(IO6) -> ADC1_CH5 -> Rudder pot wiper (via voltage divider)
                    OR unused if AS5048A primary
    Pin -- (IO1) -> WDG_KICK -> MAX6818 WDI input
    Pin -- (IO2) -> BUZZER -> 2N2222 base via 1KΩ -> Piezo buzzer
    Pin -- (IO3) -> CLUTCH MOSFET gate -> [100Ω] -> IRLZ44N gate
    Pin -- (IO4) -> PORT MOSFET gate -> [100Ω] -> IRLZ44N gate
    Pin -- (IO5) -> STBD MOSFET gate -> [100Ω] -> IRLZ44N gate
    Pin 17 (IO9) -> UART1 RXD -> SN75176 R output (NMEA input)
    Pin 18(IO10) -> UART1 TXD -> SN75176 D input (NMEA output, optional)
    Pin -- (IO15)-> PWM_OUT (reserved for proportional valve, unused by default)
    Pin -- (IO17)-> UART2 RXD -> SN75176 R output (Jetson RX)
    Pin -- (IO18)-> UART2 TXD -> SN75176 D input (Jetson TX)
    Pin 35 (IO35)-> BTN_AUTO_STBY -> 10KΩ pull-up to 3.3V, 100nF to GND
    Pin 36 (IO36)-> BTN_PORT_1    -> 10KΩ pull-up to 3.3V, 100nF to GND
    Pin 37 (IO37)-> BTN_PORT_10   -> 10KΩ pull-up to 3.3V, 100nF to GND
    Pin 38 (IO38)-> BTN_STBD_1    -> 10KΩ pull-up to 3.3V, 100nF to GND
    Pin 39 (IO39)-> BTN_STBD_10   -> 10KΩ pull-up to 3.3V, 100nF to GND
    Pin 40 (IO40)-> BTN_MODE      -> 10KΩ pull-up to 3.3V, 100nF to GND
    Pin 45 (IO45)-> LED_GREEN     -> 330Ω -> Green LED -> GND
    Pin 46 (IO46)-> LED_AMBER     -> 330Ω -> Amber LED -> GND
    Pin 47 (IO47)-> LED_RED       -> 330Ω -> Red LED -> GND
    Pin 48 (IO48)-> LED_BLUE      -> 330Ω -> Blue LED -> GND

  I2C0 BUS (GPIO8=SCL, GPIO8=SDA):
    Devices on bus:
      1. BNO085 compass    @ 0x28 (VDD=3.3V, 2x 4.7KΩ pull-ups shared)
      2. AS5048A encoder   @ 0x40 (VDD=3.3V, pull-ups shared)
      3. SSD1306 display   @ 0x3C (VDD=3.3V, pull-ups shared)
    Total bus capacitance: ~200pF (within 400pF limit for 400kHz I2C)
    Bus arbitration: i2c0_mutex held during each transaction (~2ms max)

  MOSFET DRIVER BOARD (×3 channels):
    Per channel schematic:
      ESP32 GPIO -> [100Ω] -> IRLZ44N Gate
      IRLZ44N Gate -> [10KΩ] -> GND
      IRLZ44N Drain -> Solenoid (-) terminal
      Solenoid (+) terminal -> +12V (from fused bus, via 5A fuse)
      UF4007: Cathode -> Solenoid(+), Anode -> Solenoid(-)/Drain
      [100nF] across solenoid terminals

    Channel 1: GPIO4 -> PORT solenoid
    Channel 2: GPIO5 -> STBD solenoid
    Channel 3: GPIO3 -> CLUTCH solenoid

  CURRENT SENSING:
    ACS712-5A #1: In series with PORT/STBD common supply
      Output -> ESP32 ADC1_CH6 (GPIO7, if available) or ADC1_CH4 (GPIO5 —
      conflicts! Use GPIO1 reserved for WDG... Use external ADC via I2C:
      ADS1115 I2C ADC @ 0x48 for current sensing)
    ACS712-5A #2: In series with CLUTCH solenoid supply
      Output -> ADS1115 channel 1

    NOTE: If ADC pin conflict exists, use ADS1115 I2C ADC (4 channels,
    16-bit, address 0x48/0x49) for current sensing. This keeps all current
    measurement off the ESP32 ADC1 which is reserved for rudder feedback.

  RS-422 INTERFACES:
    NMEA Input (UART1):
      NMEA bus TX+ -> SN75176 #1 A (RO) -> ESP32 GPIO9 (UART1 RXD)
      NMEA bus TX- -> SN75176 #1 B (RO-) -> GND
      SN75176 #1 VCC -> 5V, SN75176 #1 GND -> GND
      SN75176 #1 RE -> GND (always enabled)
      SN75176 #1 DE -> GND (receive only on this bus)
      120Ω termination resistor across A-B at ESP32 end

    Jetson Link (UART2):
      Jetson TX+ -> SN75176 #2 A (RO) -> ESP32 GPIO17 (UART2 RXD)
      Jetson TX- -> SN75176 #2 B (RO-)
      ESP32 GPIO18 (UART2 TXD) -> SN75176 #2 D (DI)
      ESP32 GPIO18 complement -> SN75176 #2 D- (via inverter or use
        SN75176 in half-duplex with DE control — alternatively, use
        separate TX/RX transceivers or TTL with optoisolation)
      SIMPLIFIED: Use ADuM1201 dual optoisolator for Jetson link
        (cleaner galvanic isolation than RS-422 for short cables)
      Jetson side: Jetson TX -> ADuM1201 VI1 -> ESP32 GPIO17
                  ESP32 GPIO18 -> ADuM1201 VO1 -> Jetson RX

  EXTERNAL WATCHDOG:
    ESP32 GPIO1 -> MAX6818 WDI
    MAX6818 VCC -> 3.3V
    MAX6818 GND -> GND
    MAX6818 WDO -> ESP32 EN pin (via RC reset circuit)
      WDO is normally HIGH, goes LOW on timeout
      Connect: MAX6818 WDO -> 10KΩ -> 3.3V, and WDO -> ESP32 EN
      When WDO goes LOW: ESP32 EN pulled LOW -> reset

  CONNECTOR (marine-rated, panel mount):
    M12 8-pin or DB9:
      Pin 1: +12V (fused)
      Pin 2: GND
      Pin 3: NMEA RX+ (from compass/GPS bus)
      Pin 4: NMEA RX- (from compass/GPS bus)
      Pin 5: Jetson TX (via opto) -> ESP32 RX
      Pin 6: Jetson RX <- ESP32 TX (via opto)
      Pin 7: Buzzer (+) (if external piezo)
      Pin 8: Shield / Cable screen
