RESOLUTION DETAIL FOR #1 (Jetson Model Selection):
  Agent B (Brief 11) provides a thorough cost-benefit analysis showing three Orin Nano
  Super units ($750 total, 201 TOPS) outperform a single AGX Orin ($2,000, 275 TOPS) in
  resilience and parallelism. Agent E (Brief 14) requests higher-end hardware for
  training and vision, but this is a Phase 3 enhancement. The baseline platform uses the
  Orin Nano Super. VRAM budgets in Brief 11 §4.4 show all models fit within 8GB when
  properly allocated. The architecture is designed to allow Bravo to be upgraded to an
  Orin NX 16GB as a drop-in replacement with no software architecture changes.

RESOLUTION DETAIL FOR #4 (Communication Bus):
  Brief 12 §2.4 provides the definitive analysis: RS-422 at 115200 baud provides 93 KB/s
  effective throughput, while our worst-case load is 37.5 KB/s per Jetson — a 2.5x headroom.
  CAN bus would require custom HATs on the Jetson, complex termination, and binary tooling.
  RS-422 uses standard USB serial adapters (FTDI FT4232H, kernel driver ftdi_sio) and is
  debuggable with any terminal program. This is the single most important architectural
  decision for serviceability at sea.

RESOLUTION DETAIL FOR #5 (Compass I2C Address):
  The BNO085 (CEVA Hillcrest SH-2) defaults to I2C address 0x28 when configured for
  I2C mode. The address 0x4A mentioned in Briefs 10 and 15 corresponds to a different
  operating mode or a different device variant. The original consensus (Brief 05) correctly
  specifies 0x28. DECISION: 0x28 for all BNO085 compass units.

RESOLUTION DETAIL FOR #6 (Rudder Encoder Bus):
  SPI eliminates I2C bus contention with the compass. When both BNO085 (0x28) and
  AS5048A (0x40) share an I2C bus, the 400 kHz bus with 200pF capacitance limit is
  tight. Agent F's SPI assignment (Brief 15) is the correct approach for dual-ESP32
  autopilot systems where each ESP32 has its own sensor suite. DECISION: SPI (HSPI) for
  AS5048A. Potentiometer fallback on ADC1_CH5 (GPIO6) retained.

8. DOCKSIDE ONBOARDING PROCEDURE (8 STEPS)
================================================================================

  The setup wizard is a finite state machine with 8 persistent states. Each
  state is individually saveable — power cycle during commissioning does NOT
  lose progress. Can be operated in three modes: (a) Standalone OLED+buttons,
  (b) Enhanced Jetson HDMI display, (c) Voice-guided via LLM interface.

  STEP 1 — VESSEL CONFIGURATION
    Collect vessel physical parameters: vessel type (displacement/planing/semi-displacement/
    sail), length (15–300 ft), displacement (1–2000 tons), max speed (5–45 knots), number
    of engines, propulsion type, steering type, rudder configuration. These select
    initial PID gain presets (P/I/D/DD/PR/FF) from a vessel-type lookup table.

  STEP 2 — COMPASS CALIBRATION
    "Swing ship" calibration: rotate vessel (or handheld compass) 360°, collecting
    BNO085 raw magnetometer samples at 50 Hz. Perform hard iron offset subtraction and
    soft iron ellipsoid fit. Build 36-entry deviation table (10° bins, int8 per entry).
    Validate against known bearing. Minimum confidence score: 70% acceptance, 85% open-ocean.

  STEP 3 — RUDDER CALIBRATION (3 sub-steps)
    3a: CENTER FINDING — Captain centers wheel, system captures AS5048A median of 100
        samples at 100 Hz. Stored as rudder_center_raw in NVS.
    3b: LIMITS — Captain turns wheel to port/starboard hard stop. System captures
        each limit, applies 2° software margin from mechanical stop. Asymmetry detection
        (warns if |port| ≠ |stbd| > 3°).
    3c: COUNTER-RUDDER TEST (at sea) — Engage autopilot, perform 3 heading changes
        (20°, 30°, 40°). Measure overshoot and settling time. Adjust counter-rudder
        gain: overshoot > 5° → increase gain 0.1; overshoot < 1° → decrease gain 0.05.

  STEP 4 — YAW DAMPING SETUP
    Engage autopilot at cruise speed, record 60-second heading trace. Perform FFT to
    identify dominant oscillation frequency. If amplitude > 2°, increment D gain in 0.15
    steps. Target: oscillation amplitude <1°. Maximum D gain: 3.0.

  STEP 5 — RESPONSE SPEED TUNING
    Full 30° heading change test. Measure: response latency (<500ms), rise time,
    overshoot (<5°), settling time (<30s). Adjust P gain if needed. Performance targets
    by vessel type defined.

  STEP 6 — SYSTEM VERIFICATION (CIRCLE TEST)
    Full 360° circle at cruise speed. Verify compass linearity across all headings.
    Flag any deviation > 5° in any 10° segment. Validates entire sensing + control chain.

  STEP 7 — ALARM & SAFETY TESTS
    Kill switch test (100ms solenoid de-energize, buzzer, no auto-re-engage on reset).
    Power cycle test (system boots to STANDBY, never moves solenoids during boot).
    Off-course alarm test (10° deviation triggers alarm, 15s timeout → auto-disengage).
    Watch alarm test (5-min interval, disengage if no response).

  STEP 8 — SAVE & FINALIZE
    Increment settings version. Write all parameters to NVS. Compute CRC-32 of entire
    calibration block. If dual-redundant: sync to BACKUP ESP32. Jetson backup to SQLite.
    Display summary: vessel name, type, compass confidence, rudder limits, PID gains,
    settings version, CRC. System enters STANDBY mode. Ready for normal operation.

9. DUAL-REDUNDANT ESP32 ARCHITECTURE FOR AUTOPILOT
================================================================================

  CRITICAL SYSTEMS (dual-ESP32 redundancy):

    Subsystem              Slot     Redundancy Pair
    -------------------- ------   -------------------
    Autopilot (PID steer)   SLOT-AP  AP-PRIMARY + AP-BACKUP
    Engine Monitoring       SLOT-EM  EM-PRIMARY + EM-BACKUP
    Thruster Control        SLOT-TC  TC-PRIMARY + TC-BACKUP (if vessel equipped)

  NON-CRITICAL (single ESP32, hot-swappable):
    Lighting, Bilge Pump, Tank Level, Deck Equipment, Camera Interface,
    Environmental Sensor, Navigation Aid

  PHYSICAL PLACEMENT:
    PRIMARY and BACKUP mounted on OPPOSITE sides of the wheelhouse (port/starboard).
    Ensures localized damage cannot destroy both units.

  TRUE SENSOR REDUNDANCY:
    Each ESP32 owns independent sensor hardware (NO shared I2C/SPI buses):
      - Own BNO085 compass (own I2C bus, 0x28)
      - Own AS5048A rudder encoder (own SPI bus)
      - Own SSD1306 OLED display (own I2C bus)
      - Own TMP117 temperature sensor
    This eliminates I2C bus as a single point of failure under maritime EMI.

  CROSS-COMMUNICATION:
    - 3-wire UART (TX, RX, GND) at 115200 baud between pair
    - GPIO17: heartbeat line (ACTIVE pulls LOW 10ms every 100ms; STANDBY monitors)
    - GPIO18: role-assert line (HIGH = asserting ACTIVE, enables relay)
    - GPIO21: jumper (HIGH = PRIMARY preference on equal health scores)

  RELAY SELECTOR BOARD:
    Two electromechanical relays (K1=PRIMARY, K2=BACKUP) with hardware interlock
    preventing both from closing simultaneously. Only the ACTIVE ESP32's output
    driver can reach the solenoids. If STANDBY firmware malfunctions, it physically
    CANNOT drive steering.

  POWER DISTRIBUTION (dual-ESP32 pair):
    12V bus → 5A fuse → LDO #1 (3.3V) → AP-PRIMARY
    12V bus → 5A fuse → LDO #2 (3.3V) → AP-BACKUP
    12V bus → 5A fuse → LDO #3 (5.0V) → Relay Board
    Independent regulators prevent cross-contamination of power faults.

  STARTUP SELF-TEST (POST) — ~800ms total:

    T+0ms    Both ESP32s begin boot simultaneously
    T+50ms   HARDWARE SELF-TEST: Flash CRC, SRAM march test, CPU registers, WDT arm,
             temperature check (reject if >85°C or <-10°C). Score: 0-100.
    T+100ms  BUS SELF-TEST: I2C scan (BNO085 at 0x28, OLED at 0x3C), SPI test
             (AS5048A read). Score: 0-40.
    T+200ms  SENSOR SELF-TEST: BNO085 calibration status, AS5048A range check.
             Score: 0-40.
    T+300ms  NVS INTEGRITY: Partition CRC, calibration blob, config blob, firmware
             version string. Score: 0-20. Total: 0-200.
    T+400ms  SCORE ANNOUNCEMENT: Each ESP32 broadcasts 5-byte binary frame on cross-link.
    T+500ms  PRIMARY ELECTION:
      Rule 1: Higher score (>100 vs ≤100) → higher becomes ACTIVE.
      Rule 2: Both >100, scores differ >20 → higher score ACTIVE.
      Rule 3: Scores within 20 → GPIO21 jumper decides.
      Rule 4: Both ≤100 → ALARM (neither drives outputs).
    T+600ms  ROLE ASSERTION:
      ACTIVE: GPIO18 HIGH → relay closes → PID loop, drives solenoids,
               heartbeat output, green LED solid.
      STANDBY: GPIO18 LOW → relay open → silent monitoring, yellow LED blink.
    T+800ms  JETSON HANDSHAKE: ACTIVE sends "AUTOPILOT_READY" to Jetson.
    SYSTEM OPERATIONAL.

  RUNTIME MONITORING (every 1 second on STANDBY):

    1. Heartbeat check: ACTIVE pulls GPIO17 LOW 10ms every 100ms.
       >500ms HIGH → HEARTBEAT_LOST → failover.
       >250ms HIGH → WARNING logged.
    2. UART command check: STANDBY sends QUERY, ACTIVE responds within 50ms with STATUS
       frame (heading, rudder, mode, PID integral, uptime). 2 retries then failover.
    3. Compass cross-check: |standby_heading - active_heading|
       <3° NOMINAL, 3-5° WARNING, 5-10° SENSOR_DISAGREE, >10° CRITICAL_DIVERGENCE
    4. Rudder cross-check: <3° NOMINAL, >5° encoder failure suspected.
    5. Health score degradation: STANDBY may request graceful handover if ACTIVE's
       score declines below its own.

  FAILOVER TIMING:
    Heartbeat detection: 500ms worst case
    Debounce: 20ms
    GPIO toggle: 1ms
    Relay closure: 10ms (electromechanical)
    PID state init: 2ms (load target from NVS, reset integral)
    Sensor read: 5ms
    First PID output: 3ms
    Subtotal (after detection): ~41ms
    Heading wobble: 1-2°, self-corrects in 1-2 seconds
    END-TO-END: ~540ms

10. HOT-SWAP WORKFLOW — CAPTAIN'S PROCEDURE
================================================================================

  SCENARIO: Autopilot Primary ESP32 fails during a trip at sea.

  STEP 1 — Acknowledge Failure
    Observe OLED on BACKUP ESP32 showing "ACTIVE (FAILOVER)" and single beep.
    Red LED on PRIMARY indicates fault. Buzzer beeps once (200ms).

  STEP 2 — Retrieve Spare
    Open waterproof spare parts kit, retrieve pre-programmed ESP32-S3 board.
    Kit contains: 3× ESP32-S3 boards (with VesselNode base firmware, no role-specific
    code), 3× M12 cables, 1× 4-pin jumper header.

  STEP 3 — Disconnect Failed Unit
    Unplug M12 connector from failed PRIMARY ESP32 enclosure.
    Label failed board for later diagnosis. Do NOT discard — can be reflashed and reused.

  STEP 4 — Install Replacement
    Plug spare ESP32 into the same cable run. M12 connector is keyed —
    can only be inserted one way. Ensure connector is fully seated (click).

  STEP 5 — Auto-Provisioning
    Spare ESP32 boots → reads MAC → checks NVS (empty) → enters ANNOUNCE state.
    Sends {"cmd":"announce","params":{"mac":"...","role":"unknown"}} on RS-422.
    Jetson provisioning server receives announce → looks up physical port mapping.
    If port /dev/rs422_port_0 is configured for auto-assignment to "autopilot":
      → Jetson automatically sends provision_new with role="autopilot" + stored calibration.
    No captain interaction required.

  STEP 6 — OTA Firmware Transfer (~2 minutes)
    Spare receives autopilot firmware binary over RS-422 (4KB chunks, CRC per chunk,
    SHA-256 verification). Transfer speed: ~12.5 KB/s at 115200 baud for 1.5MB binary.

  STEP 7 — Reboot and Calibration Load
    ESP32 reboots into autopilot role → sends announce as "AP-001".
    Jetson sends provision_existing with full calibration data from SQLite backup.
    ESP32 writes settings to NVS, verifies CRC.

  STEP 8 — Verify Operation
    - BACKUP ESP32 OLED shows: "ACTIVE | HDG: xxx | MODE: STANDBY"
    - Green LED solid on replacement unit.
    - Jetson-Alpha helm display shows "AP-001 ONLINE, settings synced ✓"
    - Press AUTO → engage autopilot. Verify heading hold.
    - Perform quick circle test (Section 8, Step 6) if sea conditions allow.

  STEP 9 — Log and Continue
    Jetson automatically logs failover event: timestamp, failed MAC, replacement
    MAC, total downtime. Record in voyage log.

  Total downtime: ~3 minutes (2 minutes OTA + 1 minute verification).

  NOTE FOR CRITICAL SCENARIO (both ESP32s in a pair fail):
    This is extremely unlikely (requires simultaneous destruction of port+starboard
    nodes). If it occurs:
    - Captain steers manually via mechanical helm (solenoids spring-center).
    - Replace PRIMARY first (Steps 1-8 above).
    - STANDBY (now only unit) automatically becomes ACTIVE.
    - Verify, then install replacement for the second unit.

APPENDIX A: PID CONTROLLER DESIGN SUMMARY
================================================================================

  GAIN STRUCTURE (Extended PID from pypilot, proven in 2500+ installations):

    Gain  Name            Default  Range       Description
    ----  ----------------  -------  ----------  --------------------------------
    P     Proportional    0.003   0-0.03      deg rudder per deg heading error
    I     Integral          0.000   0-0.05      Accumulated heading error (OFF by default)
    D     Derivative        0.090   0-0.24      Rate of turn (gyro-derived, primary)
    DD    Derivative'       0.075   0-0.24      Rate of rate of turn (reaction time)
    PR    Proportional Root  0.005   0-0.02      sqrt(|error|)×sign — reduces overshoot
    FF    Feed Forward      0.600   0-2.40      Response to heading command changes

  FORMULA:
    command = P*e + I*INT(e) + D*(-dh/dt) + DD*(d²h/dt²)
             + PR*sqrt(|e|)*sign(e) + FF*Delta_heading_cmd

  WHERE:
    e = heading_error = normalize(target - actual) to [-180, +180]
    dh/dt = rate of turn from gyro (low-pass filtered, ~0.3 Hz cutoff)
    Delta_heading_cmd = change in heading command (non-zero only during course changes)

  ANTI-WINDUP (5 layers):
    1. Heading error clamped to ±60° before PID
    2. Integral accumulator clamped to ±5 degree-seconds
    3. Conditional integration: only when |e| < 15° AND engaged > 5s
    4. Integral reset on mode change, engage, standby
    5. Integral decay: INT *= (1 - 0.001 * dt)

  DERIVATIVE ON MEASUREMENT:
    u_D = -D * d(heading_actual)/dt  (prevents derivative kick)

  DEADBAND (linear ramp):
    gain_factor = 0.0 if |e| < 0.5°
    gain_factor = (|e| - 0.5) / 1.5 if 0.5° ≤ |e| < 2.0°
    gain_factor = 1.0 if |e| ≥ 2.0°

  GAIN SCHEDULING (speed-based):
    speed_factor = (STW / V_ref)² where V_ref = 5.0 kts
    Applied to: P, D, PR gains. NOT applied to I or DD.

  SOLENOID COMMAND:
    output > threshold → STARBOARD ON, PORT OFF
    output < -threshold → PORT ON, STARBOARD OFF
    otherwise → both OFF
    Minimum pulse: 100ms. Minimum off-time: 200ms. Max continuous: 15 seconds.

  SAMPLE RATE: 10 Hz (compass reads at 10 Hz). Rudder feedback at 50 Hz for
  rate limiting. Current monitoring at 50 Hz. PID output at 10 Hz.

APPENDIX B: SAFETY LAYER HIERARCHY
================================================================================

  LAYER 1: HARDWARE FAIL-SAFE (always active, no firmware dependency)
    - Solenoid valve spring-center: power loss → valve centers
    - MOSFET gate pull-down (10KΩ to GND): boot/reset → solenoids OFF
    - External watchdog MAX6818 (1.6s timeout): ESP32 reset on timeout
    - Automotive fuse (5A) on main 12V input
    - TVS diode on 12V input for load dump protection
    - Flyback diodes (UF4007) on all solenoid coils
    - Relay board hardware interlock: both relays cannot close simultaneously

  LAYER 2: FIRMWARE SAFETY (task_watchdog, priority 23):
    - PID heartbeat: checked every 100ms, timeout 50ms → disengage
    - Compass freshness: <500ms timeout → disengage, WARNING
    - Jetson heartbeat: <2000ms → WARNING (autopilot continues on last heading)
    - Rudder feedback: <200ms timeout → WARNING
    - Clutch current: commanded ON but <0.5A → CLUTCH FAULT, disengage
    - Supply voltage: <10V or >16V → CRITICAL, disengage

  LAYER 3: SOFTWARE INTERLOCKS:
    - NEVER energize PORT and STBD simultaneously (hardware enforced + software check)
    - NEVER drive PORT/STBD unless clutch confirmed engaged (current > 0.5A)
    - Clutch disengagement → immediate PORT/STBD kill
    - Integral accumulator reset on mode change, engage, standby
    - Heading error clamped to ±60°
    - Rudder command rate-limited to 6°/sec
    - Solenoid minimum off-time: 200ms; max on-time: 15 seconds

  LAYER 4: ALARM HIERARCHY:
    CRITICAL (auto-disengage, buzzer, red LED):
      PID missed deadline, compass lost >500ms, overcurrent >4A, voltage
      <10V/>16V, rudder stall, supply voltage <11V/>15V
    WARNING (display, amber LED):
      Jetson link lost >2s, compass stale >300ms, heading error >20°,
      GPS fix lost (track mode), rudder near end stop, WiFi lost
    INFO (display note, green LED):
      Mode changed, heading adjusted, calibration complete

  LAYER 5: FAIL-SAFE BEHAVIORS:
    On CRITICAL alarm:
      1. engaged = false
      2. PORT = OFF, STBD = OFF (immediate)
      3. Wait 100ms
      4. CLUTCH = OFF
      5. Reset integral to 0
      6. 3-beep alarm pattern (500ms on, 200ms off)
      7. Flash "ALARM" on OLED
      8. Send alarm to Jetson
      9. Red LED solid ON

    On power loss (total):
      - All MOSFET gates pulled LOW by pull-down resistors
      - Spring-center valve returns to center
      - Clutch de-energizes → helm free for manual steering
      - Rudder holds position via hydraulic lock

APPENDIX C: NMEA SENTENCE REFERENCE
================================================================================

  INPUT SENTENCES (parsed by autopilot ESP32):

  $HDM — Heading Magnetic (primary heading source when I2C compass fails)
  $HDT — Heading True (convert to magnetic with variation from $RMC)
  $HDG — Heading + Deviation + Variation (PREFERRED — contains all correction)
  $RMC — Recommended Minimum Navigation (SOG, COG, position, variation)
  $GGA — GPS Fix Data (fix quality, position, HDOP)
  $APB — Autopilot Sentence B (heading-to-steer for track mode)
  $VHW — Water Speed and Heading (gain scheduling source)
  $MWV — Wind Speed and Angle (wind mode input)

  OUTPUT SENTENCES (transmitted by autopilot ESP32 on dedicated bus):
  $AGHDM — Heading Magnetic (talker ID "AG")
  $AGRSA — Rudder Sensor Angle
  $AGAPB — Autopilot Status (bearing, cross-track error, arrival)

  CHECKSUM: XOR of all bytes between $ and * (hex) must match.
