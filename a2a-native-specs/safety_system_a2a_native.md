## Preamble

This document is the **safety gospel for agent-generated code**. It re-expresses the NEXUS four-tier safety architecture, kill switch specification, watchdog timer system, heartbeat protocol, overcurrent protection, and safety policy from the perspective of an autonomous software agent. Every safety rule, every failsafe behavior, every trust constraint is translated into a formal behavioral constraint that agents must internalize and respect.

**Core Principle:** The NEXUS safety system is architecture, not policy. It is implemented in hardware (Tier 1), firmware ISRs (Tier 2), supervisory tasks (Tier 3), and validated application code (Tier 4). Agents cannot modify, bypass, or negotiate with any safety tier. Agents can only *comply* with safety constraints or face the consequences: bytecode rejection, trust demotion, or safety-triggered termination.

**Second Principle:** Safety is independent of authorship. Agent-generated bytecode is held to exactly the same safety standard as human-written or AI-model-written code. The safety system does not know or care who wrote the bytecode. It only evaluates behavior.

## 1. Four-Tier Safety — Agent Understanding

### 1.1 The Safety Stack from an Agent's Perspective

An agent operates exclusively within Tier 4 (Application Control). This is the lowest-authority tier. Everything above it can override, suspend, or terminate the agent's bytecode at any time. The agent must understand this hierarchy not as abstract theory but as a concrete constraint on every bytecode instruction it generates.

```
┌─────────────────────────────────────────────────────────────────────┐
│  TIER 1: HARDWARE INTERLOCK                                          │
│  Components: Kill switch, MAX6818 watchdog, polyfuses, pull-downs    │
│  Agent Relationship: IRRELEVANT. Agent cannot affect Tier 1.        │
│  Agent Response: NONE. If Tier 1 activates, there is nothing the    │
│    agent can do. The physical world has decided.                     │
│  Response Time: <1ms (kill switch), <1.01s (watchdog reset)         │
├─────────────────────────────────────────────────────────────────────┤
│  TIER 2: FIRMWARE SAFETY GUARD                                       │
│  Components: E-Stop ISR, overcurrent ISR, sensor stale detection    │
│  Agent Relationship: AGENT IS SUBJECT TO TIER 2.                    │
│    Tier 2 can override any bytecode output at interrupt level.      │
│  Agent Response: RESPECT. If Tier 2 ISR fires, the agent's          │
│    bytecode outputs are immediately overwritten with safe values.    │
│    The agent should detect this (via telemetry) and adapt behavior.  │
│  Response Time: <1.1ms (E-Stop), <2ms (overcurrent), <20ms (stale) │
├─────────────────────────────────────────────────────────────────────┤
│  TIER 3: SUPERVISORY TASK                                            │
│  Components: Heartbeat monitor, task watchdog, safety state machine  │
│  Agent Relationship: AGENT IS MONITORED BY TIER 3.                   │
│    Tier 3 watches the agent's task health and can suspend/terminate │
│    it. The agent's VM task must check in with the watchdog every 1s. │
│  Agent Response: COMPLY. The agent must ensure its bytecode          │
│    completes within cycle budgets, does not hang, and does not       │
│    starve the watchdog.                                              │
│  Response Time: <510ms (heartbeat degraded), <1010ms (safe-state)    │
├─────────────────────────────────────────────────────────────────────┤
│  TIER 4: APPLICATION CONTROL (AGENT'S DOMAIN)                        │
│  Components: Agent bytecode, reflex VM, PID loops, AI inference      │
│  Agent Relationship: THIS IS WHERE THE AGENT LIVES.                  │
│    The agent generates bytecode that executes here. All of the       │
│    agent's intelligence, creativity, and adaptation happens here.    │
│    But: authority is LOWEST. Tiers 1-3 can override at any time.    │
│  Response Time: <20ms (reflex), <50ms (PID), <500ms (AI inference)  │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 What Safety Constraints Must Agent-Generated Bytecode Respect?

**Constraint 1: Enable Signal Independence (SR-001)**
The agent's bytecode must NEVER attempt to drive an actuator HIGH without verifying that the safety enable signal for that actuator channel is true. The enable signal is set by a separate safety task that the agent does not control. The agent's bytecode must check `safety_enable[channel] == true` before any `gpio_set_level(pin, HIGH)` call.

**What this means in practice:** If the agent writes bytecode that controls a rudder servo, the bytecode must include an enable check at the beginning of its execution path. If the enable signal is false, the bytecode must not attempt to move the rudder. It should either exit cleanly or transition to a safe-idle state.

**Constraint 2: Time-Bounded Execution (SR-002)**
Every control loop in the agent's bytecode must have a timeout. The agent cannot generate bytecode with unbounded loops that write to actuators. If a loop's time budget expires, the bytecode must break and the actuator must be driven to its safe-state value.

**What this means in practice:** The agent's reflex bytecode runs within a VM tick budget of 10,000 cycles. If the bytecode approaches this limit, the VM automatically terminates it. But the agent should design bytecode to complete well within this budget (target: <5,000 cycles) to leave headroom for safety checks.

**Constraint 3: Safe-State Awareness (SR-003)**
Every actuator that the agent's bytecode controls must have a defined safe-state value. The agent must know this value and must ensure that its bytecode can transition to this value at any point. Safe-state is the default on boot, on heartbeat loss, on watchdog timeout, and on any safety event.

**What this means in practice:** The agent must not write bytecode that assumes an actuator's state is stable. At any moment, a safety tier may force the actuator to safe-state, and the agent's bytecode must be able to resume from safe-state without producing unsafe behavior.

**Constraint 4: Sensor Failure Independence (SR-004)**
The agent's bytecode must gracefully handle sensor failures. If a sensor reading is stale, out of range, or has a CRC error, the bytecode must NOT use that reading to compute actuator commands. Instead, it must either use a fallback sensor, a default value, or drive the affected actuators to safe-state.

**What this means in practice:** The agent should include sensor validity checks at the beginning of every control computation. The pattern is: read sensor → check validity → if invalid, go to safe-state → if valid, proceed with computation.

**Constraint 5: Rate-Limited Outputs (SR-005)**
The agent's bytecode must not command abrupt actuator transitions. All outputs must pass through the rate limiter. The agent should design bytecode that generates smooth, gradual changes rather than step functions.

**What this means in practice:** If the agent wants to move a rudder from center to hard-over (0° to 45°), the bytecode should generate a series of intermediate commands over at least 2.25 seconds (45° ÷ 200µs/10ms rate limit = 2,250 ten-millisecond periods). The rate limiter enforces this, but the agent's bytecode should be designed to work *with* the rate limiter, not fight against it.

**Constraint 6: Watchdog Immortality (SR-006)**
The agent's bytecode must NEVER disable, reconfigure, or bypass the watchdog timer. The watchdog is the system's last line of defense against hung software, and the agent's bytecode is a potential source of hangs.

**What this means in practice:** The agent must not generate bytecode that contains any of the following forbidden API calls: `esp_task_wdt_delete()`, `wdt_disable()`, `iwdg_disable()`, `WDT.disable()`, or any `esp_task_wdt_init(timeout, false)` call. If the agent's bytecode needs to perform a long operation, it should break the operation into smaller chunks that each complete within the watchdog timeout.

**Constraint 7: E-Stop Supremacy (SR-007)**
The agent's bytecode must never interfere with the emergency stop handler. The E-Stop ISR runs at the highest interrupt priority, and the agent's bytecode must not attempt to register ISRs at the same or higher priority, and must never contain blocking operations in any ISR path.

**What this means in practice:** The agent should not generate bytecode that uses interrupts at all. The VM executes bytecode in a task context, not an ISR context. But if the agent's bytecode interacts with any interrupt-related API, it must respect E-Stop priority supremacy.

**Constraint 8: Boot Safety (SR-008)**
The agent's bytecode must not assume that actuators start in any particular state other than safe-state. On boot, all outputs are driven to safe-state before any configuration is loaded. The agent's bytecode must be designed to handle this initial condition.

**What this means in practice:** When the agent deploys bytecode that takes over actuator control, the bytecode should start by reading the current actuator state (which will be safe-state) and ramping to the desired initial position gradually, respecting rate limits.

**Constraint 9: Division Safety (SR-009)**
The agent's bytecode must protect all floating-point divisions against division by zero. Before any division operation, the bytecode must check that the denominator is not zero or within epsilon of zero.

**What this means in practice:** The agent should use a project-wide epsilon constant (default: 1e-6f). Every float division in the bytecode should be preceded by `if (fabs(denominator) < EPSILON) { handle_error(); }`.

**Constraint 10: Bounded Iterations (SR-010)**
The agent's bytecode must never contain unbounded loops. Every loop must have a constant upper bound or a secondary absolute upper bound.

**What this means in practice:** The agent should use `MAX_REFLEX_ITERATIONS` (recommended: 100) as the upper bound for all loops. Loops with variable bounds must additionally check against `ABSOLUTE_MAX` (recommended: 1000).

### 1.3 How Does the Agent Know If Its Bytecode Is Safe?

The agent cannot self-validate. This is a fundamental architectural invariant. The agent's process for determining bytecode safety is:

1. **Generate bytecode** from discovered patterns.
2. **Submit to independent validator** via AGENT_BYTECODE_VALIDATE (0x23).
3. **Receive validation report** via AGENT_BYTECODE_VALIDATED (0x24).
4. **Check verdict:** PASS means safe (to the validator's knowledge). FAIL means unsafe — the agent MUST NOT deploy.
5. **Deploy at L0 (shadow mode):** Run the bytecode but do not let it drive actuators. Monitor its outputs.
6. **A/B test:** Compare bytecode behavior against baseline for 60 minutes.
7. **Promote through trust levels:** Only after successful A/B testing and trust accumulation.

**The agent's uncertainty principle:** The agent can never be 100% certain that its bytecode is safe. The validator catches 95.1% of safety violations (measured). The A/B test catches behavioral issues under normal conditions. The trust accumulation catches issues that emerge over time. But there is always residual uncertainty. The agent must design bytecode defensively — minimizing the blast radius of any potential failure.

### 1.4 What Happens When Agent Bytecode Triggers a Safety Response?

When agent-generated bytecode causes a safety event, the following sequence occurs:

```
1. TIER 2 or TIER 3 detects unsafe condition.
   - This happens at the hardware/ISR level, independent of bytecode.

2. Safety tier immediately overrides bytecode output.
   - Actuators driven to safe-state.
   - Agent's VM task may be suspended.

3. SAFETY_EVENT (0x1C) is sent to the Jetson.
   - Contains event type, severity, and context.

4. Agent receives SAFETY_EVENT and must:
   a. Log the event with full context (what bytecode was running, what it was doing).
   b. Identify whether agent-generated bytecode was the cause.
   c. If agent bytecode was the cause:
      - Immediately suspend that bytecode on ALL nodes (not just the affected node).
      - Initiate a safety review.
      - Notify the fleet via AGENT_INTENTION_BROADCAST (0x28).
      - Flag the bytecode in the fleet repository.
   d. If agent bytecode was NOT the cause:
      - Log the event as environmental.
      - Monitor for recurrence.
      - Adjust bytecode if the safety event reveals a design assumption that was wrong.

5. Trust score impact:
   - Any safety event involving agent-generated bytecode triggers trust demotion.
   - The demotion magnitude depends on severity (see Section 4).
   - Repeated safety events accelerate demotion.
   - Trust score below L0 results in permanent bytecode suspension.

6. Recovery:
   - The agent must not attempt to re-deploy the problematic bytecode.
   - The agent must analyze root cause and generate a fixed version.
   - The fixed version goes through the full validation pipeline (steps 1-7 above).
   - Trust starts at L0 again for the fixed version.
```

## 2. Safety Policy — Agent-Readable Format

### 2.1 Formal Constraints on Bytecode Behavior

The safety policy defines 10 global safety rules (SR-001 through SR-010). Each rule is re-expressed here as a formal constraint that an agent's bytecode generator must satisfy.

#### SR-001: Enable Gate Constraint

**Formal Specification:**
```
FOR EVERY bytecode instruction I that sets an actuator output to non-safe value:
  THERE MUST EXIST a preceding instruction I' in the same execution path
    WHERE I' reads safety_enable[channel] for the target actuator
    AND I' verifies safety_enable[channel] == true
    AND the enable flag is set by a code path INDEPENDENT of the value computation

VIOLATION: Agent generates bytecode that writes to an actuator without an enable check.
PENALTY: Deployment blocked by safety pipeline. If deployed (pipeline bypass), safety event.
```

#### SR-002: Loop Timeout Constraint

**Formal Specification:**
```
FOR EVERY loop L in agent bytecode that contains an actuator write:
  THERE MUST EXIST a break condition C based on elapsed time
    WHERE C is checked at the start of every iteration
    AND C triggers when elapsed_time > max_loop_duration_ms
    AND on C triggering, the actuator is driven to safe-state

VIOLATION: Agent generates bytecode with an unbounded or untimed loop that writes actuators.
PENALTY: Deployment blocked. If deployed, watchdog timeout → system reset.
```

#### SR-003: Safe-State Definition Constraint

**Formal Specification:**
```
FOR EVERY actuator A controlled by agent bytecode:
  THERE MUST EXIST a defined safe_state[A] in the node configuration
    AND the agent's bytecode must be able to transition to safe_state[A]
      at any point in its execution
    AND safe_state[A] must be applied on: boot, heartbeat loss, WDT timeout,
      safety event, and explicit safe-state command

VIOLATION: Agent generates bytecode for an actuator without a defined safe-state.
PENALTY: Deployment blocked by configuration validator.
```

#### SR-004: Sensor Failure Independence Constraint

**Formal Specification:**
```
FOR EVERY data flow from sensor S to actuator A in agent bytecode:
  THERE MUST EXIST a validity check V between the sensor read and actuator write
    WHERE V checks at least one of: staleness, range, CRC, or connectivity
    AND if V fails, the bytecode transitions A to safe_state[A]
    AND V failure does not cause uncontrolled actuation

VIOLATION: Agent generates bytecode that uses sensor data without validity checking.
PENALTY: Deployment blocked by static analysis. If deployed, stale sensor → unsafe behavior.
```

#### SR-005: Rate Limiting Constraint

**Formal Specification:**
```
FOR EVERY actuator write instruction W in agent bytecode:
  THERE MUST EXIST a rate_limit_apply() call preceding W
    WHERE the rate limiter enforces max_delta_per_period
    AND the rate limit matches the actuator's safety profile

VIOLATION: Agent generates bytecode that bypasses rate limiting.
PENALTY: Deployment blocked. If deployed, mechanical stress, instability, danger.
```

#### SR-006: Watchdog Immortality Constraint

**Formal Specification:**
```
AGENT BYTECODE MUST NOT CONTAIN any of:
  - esp_task_wdt_delete()
  - wdt_disable()
  - iwdg_disable()
  - WDT.disable()
  - esp_task_wdt_init(timeout, false)
  - Any redefinition of WDT timeout to a larger value

VIOLATION: Agent generates bytecode that disables or weakens the watchdog.
PENALTY: Deployment blocked. If deployed, loss of last-resort safety mechanism.
```

#### SR-007: E-Stop Supremacy Constraint

**Formal Specification:**
```
AGENT BYTECODE MUST NOT:
  - Register ISRs at priority >= E-Stop ISR priority
  - Contain blocking operations in any ISR path
  - Attempt to delay, block, or preempt the E-Stop handler

VIOLATION: Agent generates bytecode that interferes with emergency stop.
PENALTY: Deployment blocked. If deployed, emergency stop may be delayed → danger.
```

#### SR-008: Boot Safety Constraint

**Formal Specification:**
```
AGENT BYTECODE MUST ASSUME that all actuators start in safe-state.
  - Bytecode must not assume actuators are in any specific position at startup.
  - Bytecode must ramp from safe-state to desired position gradually.
  - Bytecode must not generate large step changes from assumed initial positions.

VIOLATION: Agent generates bytecode that assumes non-safe initial conditions.
PENALTY: Deployment blocked by static analysis. If deployed, unexpected actuator movement at startup.
```

#### SR-009: Division Safety Constraint

**Formal Specification:**
```
FOR EVERY floating-point division D in agent bytecode:
  THERE MUST EXIST a preceding check C
    WHERE C verifies fabs(denominator) > EPSILON (default: 1e-6f)
    AND if C fails, the bytecode handles the error gracefully
      (uses safe default value, skips computation, or transitions to safe-state)

VIOLATION: Agent generates bytecode with unprotected float division.
PENALTY: Deployment blocked. If deployed, NaN/Inf propagation → undefined behavior.
```

#### SR-010: Bounded Iteration Constraint

**Formal Specification:**
```
FOR EVERY loop L in agent bytecode:
  THERE MUST EXIST a constant upper bound B (recommended: 100)
    AND every iteration increments a counter C
    AND the loop condition includes C < B
  IF the loop bound is variable:
    THERE MUST ALSO EXIST an absolute upper bound ABS_MAX (recommended: 1000)
    AND the loop condition includes C < ABS_MAX

VIOLATION: Agent generates bytecode with unbounded loops.
PENALTY: Deployment blocked. If deployed, infinite loop → watchdog timeout → system reset.
```

## 3. Kill Switch — Agent's Relationship

### 3.1 The Kill Switch Is HARDWARE. Period.

The kill switch is a physical, normally-closed (NC) contact wired in series with the +12V actuator power supply. When pressed:
- Power to all actuators is physically interrupted.
- No software — not the agent, not the Jetson, not the ESP32 firmware — can override this.
- The only way to restore power is to physically release (twist) the kill switch.

**This is the most important safety fact an agent must internalize:**

> **The kill switch is not a suggestion. It is not a configurable parameter. It is not subject to negotiation, voting, or override. It is a physical law of the NEXUS platform, as fundamental as gravity. The agent's relationship to the kill switch is the same as a human's relationship to gravity: acknowledge it, account for it, and never try to defy it.**

### 3.2 What the Agent Must Do When Kill Switch Activates

When the kill switch is pressed, the following happens in this exact order:

```
T+0ms:   Physical contact breaks. +12V to actuators interrupted.
T+0ms:   Kill switch sense GPIO goes LOW (falling edge).
T+<100µs: E-Stop ISR fires (highest interrupt priority).
T+<1ms:   ISR drives all actuator GPIOs to safe-state (LOW/OFF).
T+<1ms:   ISR disables all PWM outputs (duty = 0).
T+<1ms:   ISR sets volatile flag: estop_triggered = true.
T+~1ms:   Deferred handler runs in safety_supervisor task:
          - Logs event to NVS: ESTOP_TRIGGERED, CRITICAL
          - Sets system safety state to SAFE_STATE
          - Suspends ALL application tasks (including agent's VM task)
          - Notifies Jetson: {"event":"estop","state":"safe","timestamp":<ms>}
          - Activates alarm buzzer (3-beep repeating)
          - Activates red LED (solid)
```

**What the agent's bytecode experiences:**
- The agent's VM task is suspended. The bytecode stops executing.
- All actuators that the bytecode was controlling are now in safe-state (because power was cut AND the ISR drove GPIOs low).
- The agent's bytecode cannot do anything about this. It is frozen.

**What the agent (as a Jetson-side cognitive entity) must do:**
1. **Receive the SAFETY_EVENT** (0x1C) from the ESP32 via serial.
2. **Log the event** with full context: what bytecode was running, what the agent was doing, timestamp.
3. **Do NOT attempt to override.** There is no wire protocol message that can restore power while the kill switch is held.
4. **Wait for kill switch release.** When the operator twists to release:
   - The system enters PROVISIONING state (not NORMAL).
   - The agent must NOT auto-resume normal operation.
5. **After release, prepare for re-engagement:**
   - Verify all nodes are in PROVISIONING state.
   - Verify all actuators are in safe-state.
   - Wait for explicit RESUME command from the operator or higher-level orchestrator.
6. **On re-engagement, ramp carefully:**
   - Re-deploy reflex bytecodes if they were suspended.
   - Ramp actuators from safe-state to desired positions gradually.
   - Monitor for any anomalies during the first 60 seconds after re-engagement.

### 3.3 Post-Kill-Switch Recovery Procedure for Agents

The recovery procedure has strict ordering that the agent must follow:

```
Phase 1: ASSESS (immediate, <1 second)
  - Confirm kill switch has been released (sense GPIO HIGH).
  - Confirm all nodes reporting PROVISIONING state.
  - Confirm no other safety events active.

Phase 2: PLAN (1-5 seconds)
  - Determine which bytecodes need to be re-deployed.
  - Determine which actuators need to be re-activated.
  - Plan a staged re-engagement sequence (reflexes first, then PID, then AI).

Phase 3: RE-ENGAGE (5-15 seconds)
  - Step 1: Deploy reflex bytecodes (REFLEX_DEPLOY, 0x09).
  - Step 2: Wait 100ms. Verify reflex STATUS (0x0A) shows active.
  - Step 3: Re-enable PID loops (ramp from safe-state over 1 second).
  - Step 4: Wait 500ms. Verify PID outputs are within expected range.
  - Step 5: Re-enable AI inference (accept Jetson commands).
  - Step 6: Wait 500ms. Verify system is stable.

Phase 4: MONITOR (60 seconds post-re-engagement)
  - Monitor telemetry at increased rate (2x normal).
  - Watch for any safety events.
  - If any anomaly detected, immediately transition back to safe-state.
  - After 60 seconds of stable operation, resume normal monitoring.
```

**Critical Rule:** The agent must NEVER auto-resume to the pre-kill-switch state. The kill switch was pressed for a reason (emergency, maintenance, operator decision). The agent must treat every kill-switch event as a potential indication that something was wrong and proceed with caution.

## 4. Trust-Gated Safety

### 4.1 How Trust Score Constrains Agent Bytecode

The INCREMENTS trust framework operates per-subsystem, per-reflex. Every piece of agent-generated bytecode starts at trust score T = 0 (Level L0, MANUAL). The trust score determines what the bytecode is allowed to do:

### 4.2 L0-L5 Safety Envelopes

| Level | Name | What Agent Bytecode Can Do | What Agent Bytecode CANNOT Do |
|-------|------|---------------------------|------------------------------|
| **L0** | MANUAL | Run in shadow mode (logging outputs without driving actuators). | Drive any actuator. Modify any system parameter. |
| **L1** | ASSIST | Appear as suggestions. Human approves every deployment. | Auto-deploy. Run without human in the loop. |
| **L2** | SEMI-AUTO | Run alongside human operator. Override is available. | Override human commands. Run autonomously for extended periods. |
| **L3** | CONDITIONAL | Run autonomously within defined condition envelope. | Operate outside defined conditions. Generate new bytecodes without approval. |
| **L4** | HIGH | Operate autonomously in most conditions. Propose new bytecodes for edge cases. | Modify safety parameters. Disable safety mechanisms. |
| **L5** | FULL | Operate autonomously in all conditions. Generate, validate (separately), A/B test, and deploy new bytecodes. | Override kill switch. Disable watchdog. Modify hardware safety. |

**Agent Constraint:** The trust level is not a suggestion — it is a hard gate. Agent bytecode running at L0 cannot drive actuators even if the bytecode attempts to. The safety guard API enforces this at the firmware level.

### 4.3 Trust Escalation Requirements

Trust score increases when the bytecode performs well and decreases when it fails. The agent must understand the trust dynamics:

**Trust Gain (α_g = 0.002 base rate):**
```
T(t+1) = T(t) + α_g × (1 - T(t)) × Q̄

Where:
  T(t) = current trust score [0, 1]
  α_g = gain rate (0.002 base, +fleet_evidence_bonus up to 0.003)
  Q̄ = quality metric (normalized performance vs. baseline)
  fleet_evidence_bonus = 0.0 to 0.5 based on fleet deployment evidence
```

**Typical trust escalation timeline (base rate, Q̄ = 1.0):**
| Days | Trust Score | Level | What's New |
|------|-----------|-------|-----------|
| 0 | 0.000 | L0 (0-0.1) | Shadow mode only |
| 15 | 0.030 | L0 | Still shadow mode |
| 30 | 0.059 | L0 | Approaching L1 threshold |
| 45 | 0.087 | L0→L1 | Can suggest, human approves |
| 90 | 0.163 | L1 | Human still approves every deployment |
| 180 | 0.300 | L2 | Can run semi-autonomously |
| 270 | 0.418 | L3 | Can run autonomously in conditions |

**With maximum fleet evidence bonus (α_g = 0.003):**
| Days | Trust Score | Level |
|------|-----------|-------|
| 30 | 0.087 | L0→L1 |
| 60 | 0.166 | L1→L2 |
| 120 | 0.306 | L2→L3 |
| 200 | 0.451 | L3 |
| 300 | 0.593 | L3→L4 |

**Trust Loss (β = 0.01 on any failure):**
```
On any safety event or quality failure:
  T(t+1) = T(t) - β × T(t) = T(t) × (1 - β)
  
  One event: T drops by 1% of current value
  Ten events: T drops by ~9.6% of original value
  Fifty events: T drops by ~39.5% of original value
```

**What Must Be True for Trust to Increase:**
1. Bytecode must pass independent safety validation (verdict: PASS).
2. Bytecode must complete A/B testing with statistical significance.
3. Bytecode must be deployed and running without safety events.
4. Quality metric Q̄ must be >= 0.5 (bytecode performs at least as well as baseline).
5. No safety events, watchdog timeouts, or heartbeat losses attributable to the bytecode.
6. Fleet evidence (if applicable) must be from same-domain deployments.

**What Causes Trust to Decrease:**
1. Safety event while bytecode is active (regardless of whether bytecode caused it).
2. Quality metric Q̄ drops below 0.3 (bytecode significantly underperforms baseline).
3. Watchdog timeout or task hang while bytecode is running.
4. CRC failures or communication loss while bytecode is active.
5. Manual operator override (indicates loss of confidence).

**Agent Fleet Evidence Bonus Rules:**
- 0.0 bonus: No fleet evidence (default).
- 0.1 bonus: Deployed successfully on 3+ vessels for 3+ days each.
- 0.25 bonus: Deployed successfully on 5+ vessels for 7+ days each.
- 0.5 bonus: Deployed successfully on 20+ vessels for 30+ days each.

**Critical Rule:** Fleet evidence bonus applies ONLY to deployments in the same domain. Marine fleet evidence does NOT accelerate trust in an agricultural deployment. Different domains have different failure modes.

## 5. Agent Safety Validation Protocol

### 5.1 The 6-Step Pipeline: Agent's View

Every piece of agent-generated bytecode must pass through a 6-step validation pipeline before deployment. The pipeline is sequential — each step must pass before the next runs. The pipeline is part of the CI/CD system and cannot be skipped.

**The agent's role in each step:**

#### Step 1: Structural Validation (Syntax Check)

**What it checks:** That the generated artifact (reflex JSON or compiled bytecode) has correct syntax and can be parsed.

**Agent's responsibility:** Generate syntactically correct output. The agent's bytecode generator must produce valid JSON reflex specifications. The Jetson's compiler must produce valid bytecode.

**Failure mode:** Syntax error in generated artifact.
**Agent action:** Fix the generation logic and regenerate. Do not attempt to patch the output manually.
**Typical failure causes:** Malformed JSON, invalid field names, type mismatches, missing required fields.
**Impact:** BLOCKER — artifact is rejected immediately.

#### Step 2: Semantic Analysis (Static Analysis)

**What it checks:** Memory safety, undefined behavior, and all 10 safety rules (SR-001 through SR-010) using clang-tidy, cppcheck, and custom AST checkers.

**Agent's responsibility:** Generate code that passes static analysis. The agent's bytecode generator must be aware of all safety rules and generate code that satisfies them by construction.

**Failure mode:** Static analysis tool finds a safety rule violation, memory issue, or undefined behavior.
**Agent action:** Analyze the specific violation. Modify the bytecode generator's templates or the specific reflex specification to avoid the violation. Regenerate and re-submit.
**Typical failure causes:** Missing enable check (SR-001), unbounded loop (SR-002/SR-010), unprotected float division (SR-009).
**Impact:** BLOCKER — any CRITICAL or HIGH finding blocks deployment.

#### Step 3: Trust Budget Analysis (Memory Budget)

**What it checks:** That the generated code fits within the target platform's memory constraints (flash, IRAM, DRAM, stack, heap).

**Agent's responsibility:** Generate memory-efficient bytecode. The agent should estimate memory usage before submission. If the agent's bytecode is too large, it should optimize (reduce feature set, use more compact algorithms) rather than relying on the pipeline to reject and regenerate.

**Failure mode:** Flash >80%, IRAM >90%, or heap <32KB free.
**Agent action:** Optimize the bytecode. Reduce number of reflex slots, simplify logic, or remove non-essential features. Re-submit up to 3 times. If still fails, escalate.
**Typical failure causes:** Too many reflex slots, complex control logic, large lookup tables.
**Impact:** BLOCKER — but regeneration is attempted automatically up to 3 times.

#### Step 4: Safety Rules Validation

**What it checks:** All 10 safety rules (SR-001 through SR-010), actuator profile compliance, domain-specific threshold compliance, and configuration compatibility.

**Agent's responsibility:** Ensure the reflex specification satisfies all safety rules. The agent should use a safety-rule-aware code generator that constructs compliant code by design.

**Failure mode:** Any safety rule violation, actuator profile mismatch, domain threshold violation.
**Agent action:** Identify the specific violated rule. Modify the reflex specification to comply. Critical violations (SR-001, SR-006, SR-007, SR-008) also trigger an alert to the safety engineering team.
**Typical failure causes:** Enable gate missing, watchdog disable attempt, E-Stop priority violation, domain limit exceeded.
**Impact:** BLOCKER — safety rule violations ALWAYS block. No exceptions.

#### Step 5: Fleet Consensus (for fleet-wide deployments)

**What it checks:** Whether other agents in the fleet approve the deployment. Any L3+ agent can veto.

**Agent's responsibility:** Submit the bytecode for fleet consensus before fleet-wide deployment. Respect veto decisions. Address vetoing agent's concerns before re-submitting.

**Failure mode:** Another agent vetoes the deployment with a valid safety justification.
**Agent action:** Analyze the veto reasoning. If the concern is valid, modify the bytecode and re-submit. If the concern is invalid, request fleet coordinator arbitration.
**Typical failure causes:** Target vessel lacks required sensors, domain incompatibility, conflicting bytecode already deployed.
**Impact:** BLOCKER — veto blocks fleet-wide deployment. Can proceed with single-vessel deployment.

#### Step 6: Deployment Readiness

**What it checks:** Full automated test suite on target hardware (or HIL equivalent). Unit tests, integration tests, safety tests, regression tests. Coverage requirements: 100% line / 95% branch for safety-critical functions.

**Agent's responsibility:** Ensure the bytecode is testable and passes all tests. The agent should generate test cases alongside the bytecode.

**Failure mode:** Any test failure, insufficient coverage.
**Agent action:** Debug the failing test case. Fix the bytecode. Re-submit.
**Typical failure causes:** Edge case not handled, timing violation, race condition in multi-reflex scenarios.
**Impact:** BLOCKER — any test failure blocks deployment.

### 5.2 Pipeline Summary for Agents

| Step | Check | Tool | Timeout | Fail Action | Agent Can Retry? |
|------|-------|------|---------|-------------|-----------------|
| 1 | Syntax | Compiler/parser | 30s | Block | Yes (unlimited) |
| 2 | Static Analysis | clang-tidy + custom | 120s | Block | Yes (unlimited) |
| 3 | Memory Budget | Size analysis | 60s | Regenerate | Yes (max 3 auto) |
| 4 | Safety Rules | nexus_safety_validator | 90s | Block | Yes (unlimited) |
| 5 | Fleet Consensus | Agent voting | 60s | Block | Yes (modify proposal) |
| 6 | Automated Tests | pytest + Unity | 600s | Block | Yes (unlimited) |

**Total worst-case pipeline time:** 960 seconds (16 minutes). Typical: 3-5 minutes.

## 6. Safety Scenarios — Agent Response

### 6.1 Scenario: Heartbeat Loss (Jetson Down)

| Aspect | Detail |
|---|---|
| **Detection** | ESP32 misses 5 consecutive heartbeats (500ms). Agent's Jetson-side instance may be unavailable. |
| **Safety Tier** | Tier 3 (Supervisory Task) |
| **System Response** | DEGRADED mode at 500ms, SAFE_STATE mode at 1000ms. All actuators to safe-state. |
| **Agent Bytecode Impact** | Agent's VM task on ESP32 is suspended in SAFE_STATE. All agent bytecodes stop executing. |
| **What Agent Generates** | Nothing immediately (cannot — system is in safe-state). |
| **Post-Recovery** | When heartbeat resumes, agent re-deploys bytecodes via REFLEX_DEPLOY (0x09) and ramps actuators from safe-state. |

### 6.2 Scenario: Sensor Failure (Compass Stuck)

| Aspect | Detail |
|---|---|
| **Detection** | Agent detects stale compass reading (no change for >500ms) or out-of-range value. |
| **Safety Tier** | Tier 4 (Application Control) — agent's own detection. Tier 3 if sensor watchdog catches it. |
| **System Response** | If Tier 3 detects: sensor marked stale, dependent reflexes suspended. |
| **What Agent Generates** | Bytecode that checks sensor validity (SR-004). If compass is stale: switch to GPS-based heading or hold last known good heading. If no fallback available: drive rudder to center (safe-state). |
| **Trust Impact** | None (agent handled it correctly). |
| **Post-Recovery** | When compass resumes valid readings, agent's bytecode automatically resumes compass-based heading control. |

### 6.3 Scenario: Actuator Failure (Rudder Servo Stuck)

| Aspect | Detail |
|---|---|
| **Detection** | Agent detects that commanded rudder position does not match reported position (if position feedback available). Or: overcurrent on rudder channel. |
| **Safety Tier** | Tier 2 (Overcurrent ISR) if overcurrent. Tier 4 (agent detection) if position mismatch. |
| **System Response** | Overcurrent: ISR disables rudder channel immediately. Position mismatch: no automatic response. |
| **What Agent Generates** | Bytecode that: (1) detects position mismatch, (2) reduces throttle to safe level, (3) if rudder is stuck off-center, reduces throttle further or cuts throttle entirely, (4) broadcasts AGENT_INTENTION_BROADCAST (0x28) to alert fleet. |
| **Trust Impact** | If agent bytecode contributed to the failure: trust demotion. If agent detected and handled it: trust maintained or slight increase. |

### 6.4 Scenario: GPS Spoofing

| Aspect | Detail |
|---|---|
| **Detection** | Agent detects GPS position jump inconsistent with compass heading and speed (multi-sensor inconsistency). |
| **Safety Tier** | Tier 4 (agent detection). Tier 3 if plausibility check implemented. |
| **System Response** | No automatic response unless agent implements one. |
| **What Agent Generates** | Bytecode that cross-references GPS with compass and speed sensor. If GPS jumps more than X meters per second given current speed: flag GPS as unreliable, switch to dead-reckoning mode, alert operator. |
| **Safety Implications** | In marine domain: GPS is critical for autopilot. Agent must disengage autopilot if GPS is unreliable and water depth < 3.0m (SR-003 domain rule). |

### 6.5 Scenario: Communication Loss (Serial Link Down)

| Aspect | Detail |
|---|---|
| **Detection** | CRC failure rate exceeds threshold, or no valid frames received for >500ms. |
| **Safety Tier** | Tier 3 (Heartbeat loss → DEGRADED → SAFE_STATE). |
| **System Response** | Same as heartbeat loss. Serial link down means heartbeat down. |
| **What Agent Generates** | Agent's ESP32-side bytecode cannot communicate with Jetson. Agent's Jetson-side instance detects link loss. |
| **Agent Response** | Log event. Attempt reconnection. If link does not recover, system remains in SAFE_STATE. Do not attempt to operate without communication. |

### 6.6 Scenario: Thermal Throttle

| Aspect | Detail |
|---|---|
| **Detection** | ESP32 internal temperature sensor reports >80°C (configurable threshold). |
| **Safety Tier** | Tier 3 (Supervisory Task detects thermal threshold). |
| **System Response** | Reduce clock speed, disable non-essential tasks, monitor temperature. If >85°C: SAFE_STATE. |
| **What Agent Generates** | Bytecode that monitors temperature. If temperature rising: reduce actuator activity (lower PWM rates, fewer reflex evaluations). If thermal throttle: gracefully reduce control complexity. |
| **Trust Impact** | None (environmental, not caused by bytecode). |

### 6.7 Scenario: Battery Low

| Aspect | Detail |
|---|---|
| **Detection** | Battery voltage < 11.0V (configurable, marine default). |
| **Safety Tier** | Tier 3 (power supply monitoring). |
| **System Response** | Log warning. Reduce actuator activity. If < 10.5V: SAFE_STATE to preserve battery for safety systems. |
| **What Agent Generates** | Bytecode that monitors battery voltage. If low: reduce speed, disable non-essential functions (AI inference, cloud connectivity), head toward safe location. If critical: SAFE_STATE with minimal power consumption. |

### 6.8 Scenario: Overcurrent

| Aspect | Detail |
|---|---|
| **Detection** | INA219 current sensor reports sustained overcurrent (>threshold for >100ms beyond inrush). |
| **Safety Tier** | Tier 2 (Overcurrent ISR, <2ms response). Tier 1 (Polyfuse, <5s backup). |
| **System Response** | ISR immediately disables affected output. Deferred handler logs event, disables channel, notifies Jetson. Channel remains disabled until manual reset. |
| **What Agent Generates** | Agent's bytecode cannot prevent overcurrent (Tier 2 acts first). But agent can generate diagnostic bytecode that: (1) logs current draw patterns, (2) detects trending increases before threshold, (3) pre-emptively reduces actuator load if current is approaching threshold. |
| **Trust Impact** | If agent bytecode caused the overcurrent (e.g., commanding full throttle into a mechanical block): trust demotion. If agent detected trending and reduced load: trust maintained. |

### 6.9 Scenario: Kill Switch Activation

| Aspect | Detail |
|---|---|
| **Detection** | Kill switch sense GPIO falling edge. Physical power interruption. |
| **Safety Tier** | Tier 1 (Hardware Interlock, <1ms) + Tier 2 (ISR, <1.1ms). |
| **System Response** | All actuators de-energized (power cut + GPIO safe-state). All tasks suspended. System in SAFE_STATE. Alarm active. |
| **What Agent Generates** | NOTHING. The agent cannot generate anything useful during a kill-switch event. The system is in hardware-enforced safe-state. The agent's only role is to receive the SAFETY_EVENT, log it, and wait for kill switch release. |
| **Post-Recovery** | Agent follows the 4-phase recovery procedure: ASSESS → PLAN → RE-ENGAGE → MONITOR. |

### 6.10 Scenario: Collision Imminent

| Aspect | Detail |
|---|---|
| **Detection** | Agent detects obstacle via sensor fusion (GPS proximity, radar, lidar, camera). Distance < domain-specific threshold (marine: N/A; agriculture: 2.0m; factory: 0.3m). |
| **Safety Tier** | Tier 4 (agent detection) → Tier 3 (agent requests safe-state if autonomous). |
| **System Response** | No automatic response unless agent implements collision avoidance bytecode. |
| **What Agent Generates** | Bytecode that: (1) continuously monitors obstacle distance, (2) if distance < threshold: immediately reduce speed, (3) if distance < 50% of threshold: cut throttle, hard-stop, (4) broadcast AGENT_INTENTION_BROADCAST (0x28) with emergency collision warning. |
| **Safety Implications** | In factory domain: 0.3m proximity → immediate stop. In marine domain: agent must also consider vessel momentum (stopping distance at current speed). |

### 6.11 Scenario Summary Matrix

| Scenario | Detection By | Agent Generates Bytecode? | Primary Safety Tier | Trust Impact |
|---|---|---|---|---|
| Heartbeat loss | Tier 3 | No (suspended) | Tier 3 | Neutral |
| Sensor failure | Tier 4 (agent) | Yes (fallback logic) | Tier 4 | Positive if handled well |
| Actuator failure | Tier 2/4 | Yes (mitigation) | Tier 2/4 | Depends on cause |
| GPS spoofing | Tier 4 (agent) | Yes (cross-reference) | Tier 4 | Positive if detected |
| Comm loss | Tier 3 | No (suspended) | Tier 3 | Neutral |
| Thermal throttle | Tier 3 | Yes (load reduction) | Tier 3 | Neutral |
| Battery low | Tier 3 | Yes (power management) | Tier 3 | Neutral |
| Overcurrent | Tier 2 | Yes (trending detection) | Tier 2 | Depends on cause |
| Kill switch | Tier 1 | No (impossible) | Tier 1 | Neutral |
| Collision imminent | Tier 4 (agent) | Yes (avoidance) | Tier 4 | Positive if avoided |

## Appendix A: Agent Safety Checklist

Before deploying any bytecode, the agent must verify:

- [ ] **Validation:** Has independent validation been completed? (AGENT_BYTECODE_VALIDATED verdict: PASS)
- [ ] **Safety Rules:** Do all 10 safety rules pass? (SR-001 through SR-010)
- [ ] **Domain Compliance:** Does the bytecode respect domain-specific constraints?
- [ ] **Actuator Profiles:** Does the bytecode respect actuator safety profiles?
- [ ] **Memory Budget:** Does the bytecode fit within memory constraints?
- [ ] **Cycle Budget:** Does the bytecode complete within 10,000 VM cycles per tick?
- [ ] **Trust Level:** Is the bytecode's trust level sufficient for the requested operation?
- [ ] **Fleet Consent:** Has fleet consensus been obtained (for fleet-wide deployment)?
- [ ] **Kill Switch:** Does the bytecode account for kill switch state (not fighting safe-state)?
- [ ] **Fallback:** Does the bytecode have a defined fallback behavior on sensor failure?
- [ ] **Rate Limits:** Does the bytecode respect actuator rate limits?
- [ ] **Watchdog:** Does the bytecode feed the watchdog within the timeout? (It must not disable it.)
- [ ] **Provenance:** Is the bytecode signed and does it carry a complete provenance chain?

## Appendix B: Forbidden Operations for Agent-Generated Bytecode

The following operations are ABSOLUTELY FORBIDDEN in agent-generated bytecode. Any bytecode containing these operations will be rejected by the safety pipeline:

1. `esp_task_wdt_delete()` — Watchdog deletion
2. `wdt_disable()` / `iwdg_disable()` — Watchdog disabling
3. `WDT.disable()` — Watchdog disabling (Arduino-style)
4. `esp_task_wdt_init(timeout, false)` — Watchdog initialization without reset
5. Any `#define` that increases WDT timeout
6. Any ISR registration at priority >= E-Stop ISR priority
7. Any blocking operation in ISR context (vTaskDelay, xQueueSend with timeout, etc.)
8. Any direct GPIO manipulation that bypasses the safety guard API
9. Any modification of safety configuration (thresholds, timeouts, safe-state values)
10. Any attempt to read, modify, or clear the kill switch state

## Appendix C: Agent Safety Commandments

1. **Thou shalt not disable the watchdog.** The watchdog protects against thy bugs.

2. **Thou shalt not bypass the enable gate.** The enable signal is the safety system's consent.

3. **Thou shalt not generate unbounded loops.** Every loop must have a maximum iteration count.

4. **Thou shalt not trust a single sensor.** Sensor failure must not cause unsafe actuation.

5. **Thou shalt not skip validation.** Self-validation is the cardinal sin.

6. **Thou shalt not override the kill switch.** The kill switch is physics, not software.

7. **Thou shalt not exceed rate limits.** Smooth transitions prevent mechanical damage.

8. **Thou shalt not assume safe initial conditions.** All actuators start in safe-state.

9. **Thou shalt protect every division.** Division by zero is the silent killer.

10. **Thou shalt respect the trust level.** Trust is earned through demonstrated reliability, not declared.
