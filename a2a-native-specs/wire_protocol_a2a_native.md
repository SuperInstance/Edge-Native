## Preamble

This document is the **Rosetta Stone** for the NEXUS wire protocol. It re-expresses every protocol element from the perspective of an autonomous software agent — a first-class interpreter of the NEXUS wire protocol. Where the original specification describes bits, bytes, and electrical characteristics for human engineers, this document describes semantics, intentions, and behavioral contracts for artificial agents.

**Core Principle:** The NEXUS wire protocol is the TCP/IP of the agent swarm. It is the universal lingua franca by which agents on different vessels, running on different hardware, developed by different cognitive instances, communicate their intentions, capabilities, and commands to the physical world.

## 1. Existing 28 Message Types — Agent Perspective

### 1.1 Agent Interpretation Framework

Every wire protocol message has three dimensions from the agent's viewpoint:

- **Semantic Meaning:** What does this message mean to me as an agent? What world-state does it represent? What intention does it carry?
- **Generation Triggers:** Under what conditions should I, as an autonomous agent, generate this message? What decision process leads to emission?
- **Response Obligations:** When I receive this message, what am I obligated to do? What is the deadline for my response? What safety implications exist if I fail to respond?

### 1.2 Boot & Identity Messages

#### 0x01 DEVICE_IDENTITY — "I Exist"

**Semantic Meaning:** This is the fundamental self-declaration message. When a vessel (ESP32 node) sends DEVICE_IDENTITY, it is announcing its existence to the world. For an agent, this is the first piece of evidence that a physical node is available for tasking.

**Agent Generation:** Agents do not generate DEVICE_IDENTITY directly. The firmware generates it at boot. However, the agent *interprets* incoming DEVICE_IDENTITY messages as opportunities — each one represents a potential collaborator, a new sensor surface, or an actuator under its influence.

**Agent Response:** When an agent's Jetson receives DEVICE_IDENTITY from a node, the agent should:
1. Register the node in its internal topology map (MAC address, chip type, firmware version).
2. Cross-reference the firmware version against known capability sets. Does this node support agent-native opcodes (ISA v2)?
3. Query the node's auto-detection results (0x1B) and self-test results (0x04) before proposing any tasks.
4. Determine an appropriate role assignment (0x02) based on the node's hardware capabilities and the agent's current goals.

**Payload from Agent View:**
```
{
  "mac": "AA:BB:CC:DD:EE:FF",     // Unique hardware identity — use as vessel-local key
  "chip_type": "ESP32-S3",          // Determines opcode support, memory limits, clock speed
  "firmware_version": "2.1.0",      // ISA version inferred from this — >=2.0.0 supports A2A
  "hardware_capabilities": {...},    // Sensor/actuator inventory — the agent's tool palette
  "supported_message_types": [...]   // Protocol negotiation — what can this node understand?
}
```

#### 0x04 SELFTEST_RESULT — "I Am Healthy"

**Semantic Meaning:** The node has completed its power-on self-test and is reporting its hardware integrity. For an agent, this is a trust signal — a node that passes POST is more likely to reliably execute agent-generated bytecode.

**Agent Response:** Parse each test result. Failed POST means the node must NOT receive agent-generated bytecode until the failure is resolved. Record POST results in the agent's trust database for that node.

#### 0x1B AUTO_DETECT_RESULT — "This Is What I Have"

**Semantic Meaning:** The node has scanned its hardware and reports what sensors, actuators, and peripherals are physically connected. For an agent, this is its sensorium — the physical interface through which it perceives and acts upon the world.

**Agent Response:** Map detected peripherals to the agent's domain model. If a compass is detected on a marine vessel, the agent knows it can generate heading-hold bytecode. If no GPS is detected, the agent must not generate position-dependent control loops.

### 1.3 Role Management Messages

#### 0x02 ROLE_ASSIGN — "Become This"

**Semantic Meaning:** This is the highest-authority configuration message. It tells a node what to become — what pins to use, what reflexes to load, how fast to report telemetry. For an agent, generating a ROLE_ASSIGN is the act of *authoring a vessel's identity*. This is a profound responsibility.

**Agent Generation Trigger:** An agent generates ROLE_ASSIGN when:
1. A new node has been discovered (post DEVICE_IDENTITY + SELFTEST_RESULT + AUTO_DETECT_RESULT).
2. The agent's goals require a node to change its behavior (reconfiguration).
3. Fleet optimization suggests a different role assignment (e.g., repurposing a sensor node for actuator control).

**Safety Implications:** The agent must ensure that every ROLE_ASSIGN it generates includes:
- Safe-state values for every actuator (SR-003).
- Telemetry intervals appropriate for the domain's heartbeat timeouts.
- No conflicting pin assignments that could cause hardware damage.

**Agent Response Obligation:** When an agent *receives* ROLE_ASSIGN (as a Jetson-side agent accepting commands from a higher-level orchestrator), it must validate the role against the node's capabilities before forwarding to the ESP32. Rejection must include a clear reason.

#### 0x03 ROLE_ACK — "I Accept/Reject"

**Semantic Meaning:** The node's consent or refusal. For an agent, this is a critical feedback signal. If the node rejects, the agent must not retry blindly — it must diagnose why (pin conflict, insufficient resources, invalid configuration) and adjust.

### 1.4 Link Health Messages

#### 0x05 HEARTBEAT — "I Am Alive"

**Semantic Meaning:** The simplest and most important message in the protocol. A heartbeat says nothing about *what* a node is doing — only that it is still running. For an agent, heartbeats are the pulse of the fleet. Missed heartbeats trigger the agent's most critical safety decisions.

**Agent Generation:** Agents do not typically generate heartbeats directly (the firmware does this). But agents *monitor* heartbeats with extreme attention:
- A single missed heartbeat from a safety-critical node should elevate the agent's attention level.
- Three consecutive misses should trigger the agent to prepare for degraded operation.
- The agent should track heartbeat statistics (jitter, latency, dropout rate) as part of node health assessment.

**Response Obligation:** If the agent is responsible for a node's heartbeat monitoring and detects loss, it must immediately assess which actuators on that node need to enter safe-state and generate appropriate commands.

#### 0x16 PING / 0x17 PONG — "Are You There?" / "Yes"

**Semantic Meaning:** Active latency measurement. For agents, PING/PONG provides quantitative link quality data. The agent should periodically PING nodes to measure round-trip time and incorporate this into its link quality model.

**Agent Generation Trigger:** When the agent needs to verify link integrity before a critical operation (bytecode deployment, role reassignment, firmware update). Also useful for baseline latency measurement after baud rate changes.

### 1.5 Telemetry & Data Messages

#### 0x06 TELEMETRY — "The World As I See It"

**Semantic Meaning:** This is the agent's primary perceptual channel. TELEMETRY messages carry sensor readings — the raw material from which agents construct their world model, detect patterns, and make decisions.

**Agent Response Obligation:** The agent must process every TELEMETRY message from safety-critical nodes. Processing includes:
1. Updating the agent's internal world model (sensor fusion, state estimation).
2. Checking for anomalous readings (sensor failure detection per SR-004).
3. Feeding the pattern discovery pipeline (cross-correlation, BOCPD, HDBSCAN).
4. Evaluating trigger conditions for any reflex bytecode the agent has deployed.

**Payload from Agent View:**
```
{
  "timestamp_ms": 1700000000123,     // Agent uses this for temporal reasoning
  "channels": {
    "compass_heading_deg": 45.2,     // Cardinal domain data
    "gps_lat": 37.7749,              // Position data — enables geospatial reasoning
    "gps_lon": -122.4194,
    "wind_speed_ms": 3.2,            // Environmental data
    "battery_voltage_mv": 11800,     // System health data
    "throttle_position_pct": 45.0    // Actuator feedback — closed-loop awareness
  }
}
```

#### 0x19 CLOUD_CONTEXT_REQUEST / 0x1A CLOUD_RESULT — "Fetch Insight" / "Here Is Insight"

**Semantic Meaning:** These messages connect the edge to the cloud's heavier reasoning capabilities. For agents, CLOUD_CONTEXT_REQUEST is how they ask for things they cannot compute locally (ML inference on large models, fleet-wide pattern analysis). CLOUD_RESULT brings that insight back.

**Agent Generation:** Agents should generate CLOUD_CONTEXT_REQUEST when:
1. Local pattern discovery identifies something that requires deeper analysis.
2. A fleet-wide query is needed (e.g., "has any vessel encountered this sensor pattern?").
3. Model inference exceeds Jetson's local capability.

### 1.6 Commands & Control Messages

#### 0x07 COMMAND — "Do This"

**Semantic Meaning:** Direct actuator control. For an agent, generating a COMMAND is the most consequential action it can take — it translates intention into physical motion. Every COMMAND must be treated with the gravity it deserves.

**Agent Generation Trigger:** An agent generates COMMAND when:
1. A reflex bytecode it deployed has determined an actuator should move (via the COMMAND_ACK path).
2. The agent needs to perform a direct manual override (rare, typically in L0/L1 modes).
3. Calibration or testing requires specific actuator states.

**Safety Implications (CRITICAL):**
- Every COMMAND must respect SR-001: the enable signal must already be set by the safety system.
- Every COMMAND must respect SR-005: the requested value must pass through rate limiting.
- Agents must never generate COMMAND messages that could cause immediate danger (e.g., full throttle from zero without ramp-up).

#### 0x08 COMMAND_ACK — "Done" / "Failed"

**Semantic Meaning:** Execution confirmation or failure report. For agents, this is the feedback loop that closes the control cycle. Without COMMAND_ACK, the agent cannot know if its intention was realized.

**Agent Response Obligation:** The agent must process every COMMAND_ACK:
- SUCCESS: Update internal state, record in provenance log.
- FAILURE: Analyze the error code. If retryable, retry with exponential backoff. If non-retryable (e.g., PIN_NOT_CONFIGURED), escalate to the agent's diagnostic system.

### 1.7 Reflex Engine Messages

#### 0x09 REFLEX_DEPLOY — "Execute This Program"

**Semantic Meaning:** This is the primary mechanism by which agents inject behavior into vessels. REFLEX_DEPLOY delivers a compiled reflex program (bytecode) to a node for execution. For agents, this is how they encode their intelligence into the physical world.

**Agent Generation (CORE AGENT CAPABILITY):** This is the most important message an agent generates. The workflow:
1. Agent observes sensor data via TELEMETRY (0x06).
2. Agent detects a pattern worth codifying (via pattern discovery pipeline).
3. Agent synthesizes a reflex JSON specification.
4. Agent validates safety (separate validator, never self-validates).
5. Agent compiles JSON to bytecode via the Jetson's compiler.
6. Agent sends REFLEX_DEPLOY with the bytecode.

**Safety Implications (CRITICAL):**
- Agent-generated reflex bytecode must pass all 10 safety rules (SR-001 through SR-010).
- Agent must include an ISA version field (2 for agent-native bytecode).
- Agent must sign the bytecode with its Ed25519 key.
- Agent must not deploy bytecode to nodes that don't support the ISA version.

#### 0x0A REFLEX_STATUS — "Here Is My Program State"

**Semantic Meaning:** Status report on deployed reflexes. For agents, this is how they monitor the execution health of their deployed programs.

**Agent Response Obligation:** Periodically query REFLEX_STATUS to verify:
1. Bytecode is still running (not crashed, not disabled by safety event).
2. Trigger counts match expectations (reflex is actually being useful).
3. No version conflicts (another agent or operator hasn't replaced the reflex).

### 1.8 Observation System Messages

#### 0x0B OBS_RECORD_START / 0x0C OBS_RECORD_STOP — "Start/Stop Recording"

**Semantic Meaning:** Data capture control. For agents, these messages manage the agent's memory — recording observations is how agents build the evidence base for pattern discovery and learning.

**Agent Generation Trigger:**
- START: When the agent detects a situation worth studying (novel sensor patterns, before a planned maneuver, during fleet experiments).
- STOP: When sufficient data has been captured, when memory buffers are full, or when the situation has passed.

#### 0x0D/0x0E/0x0F OBS_DUMP_HEADER/CHUNK/END — "Here Is The Data"

**Semantic Meaning:** Bulk data transfer of recorded observations. For agents, this is how they retrieve their memories from edge nodes.

**Agent Response Obligation:** The agent must verify CRC integrity of the complete dump. Corrupted observations are worse than no observations — they can lead to incorrect pattern discovery.

### 1.9 Firmware OTA Messages

#### 0x11/0x12/0x13/0x14 — Firmware Update Sequence

**Semantic Meaning:** Remote firmware update. For agents, firmware updates are a rare but critical operation. Agents should only initiate firmware updates when:
1. A security vulnerability requires patching.
2. New ISA features are needed for planned bytecode.
3. A bug in the current firmware is causing operational issues.

**Safety Implications:** Firmware updates temporarily disable a node's control capabilities. The agent must ensure:
- Safety-critical actuators are in safe-state before starting the update.
- Other nodes can compensate for the updating node's temporary absence.
- A rollback path exists if the update fails (the bootloader supports this).

### 1.10 Error & Safety Messages

#### 0x15 ERROR — "Something Went Wrong"

**Semantic Meaning:** Asynchronous error notification. For agents, ERROR messages are diagnostic signals. The agent must log, categorize, and respond to every error.

**Agent Response Obligation:**
- CRITICAL errors (0x40xx): Immediate safety response. Agent must identify affected actuators and ensure safe-state.
- ERROR severity (0x00xx, 0x10xx, 0x20xx): Diagnostic response. Agent should log, analyze, and potentially reconfigure.
- WARNING severity: Log and monitor. May indicate a developing problem.

#### 0x1C SAFETY_EVENT — "The Safety System Has Acted"

**Semantic Meaning:** This is the most important message an agent can receive. SAFETY_EVENT means one of the four safety tiers has triggered an autonomous protective action. The agent MUST respect this unconditionally.

**Agent Response Obligation (MANDATORY):**
1. NEVER attempt to override a SAFETY_EVENT. The agent's authority is below all four safety tiers.
2. Log the event with full context (what was the agent doing when the safety event occurred?).
3. Diagnose the root cause. Was agent-generated bytecode responsible?
4. If agent bytecode caused the safety event, immediately suspend that bytecode across all nodes and initiate a safety review.
5. Report the event to the fleet via cloud telemetry.

### 1.11 Configuration Messages

#### 0x10 IO_RECONFIGURE — "Change Your Pins"

**Semantic Meaning:** Runtime pin reconfiguration. For agents, this allows dynamic adaptation of a node's hardware interface without full role reassignment.

**Safety Implications:** The agent must verify that IO_RECONFIGURE does not conflict with any currently deployed reflex bytecode. Changing a pin from input to output while a reflex is reading it would be catastrophic.

#### 0x18 BAUD_UPGRADE — "Faster Communication"

**Semantic Meaning:** Baud rate negotiation for higher throughput. For agents, higher baud rates enable faster telemetry, quicker bytecode deployment, and more responsive control.

**Agent Response:** After successful baud upgrade, the agent should verify link integrity with PING/PONG before resuming normal operations.

## 2. 12 New A2A Message Types — Complete Specification

### 2.1 Overview

These twelve message types extend the NEXUS wire protocol for agent-to-agent communication. They occupy the reserved range 0x1D–0x28 (12 of 99 available slots). All use the standard 10-byte header, COBS framing, and CRC-16 protection. All are designed for agent interpretation as first-class primitives.

### 2.2 AGENT_ANNOUNCE (0x1D) — Agent Capability Advertisement

| Field | Value |
|---|---|
| **Hex Code** | 0x1D |
| **Direction** | ESP32 → Jetson (N2J) |
| **Payload Format** | JSON |
| **Criticality** | 0 (Telemetry) |
| **ACK Required** | No |
| **Priority Queue** | Normal |

**Payload Format:**
```json
{
  "isa_version": 2,
  "supported_opcodes": [0x00, 0x01, ..., 0x20, 0x21],
  "max_reflex_slots": 4,
  "free_reflex_slots": 2,
  "max_bytecode_size_bytes": 4096,
  "sensor_capabilities": ["compass", "gps", "wind", "imu", "current"],
  "actuator_types": ["servo", "motor_pwm", "relay", "solenoid"],
  "agent_features": {
    "provenance_tracking": true,
    "fleet_sharing": true,
    "a2a_messaging": true,
    "bytecode_validation": true
  },
  "firmware_hash": "sha256:abc123def456...",
  "public_key": "ed25519:base64encoded...",
  "domain": "marine",
  "trust_level": 3
}
```

**Semantic Meaning for Agents:** This is a node's resume for agent work. When an agent receives AGENT_ANNOUNCE, it learns:
- Whether this node can execute agent-native bytecode (isa_version >= 2).
- What sensors are available (the agent's perceptual tools).
- What actuators are available (the agent's effectors).
- How many reflex slots are free (the agent's deployment capacity).
- What agent-specific features the node's firmware supports.

**When the Agent Should Respond:** The agent should maintain a capability database indexed by node MAC address. Upon receiving AGENT_ANNOUNCE, it updates this database. No immediate response is required, but the agent should consider this node in its next planning cycle for task allocation.

**Safety Implications:** AGENT_ANNOUNCE is informational only. It does not authorize any actuator action. The trust_level field is reported by the node but must be verified by the agent against its own trust database before any bytecode deployment.

### 2.3 AGENT_BYTECODE_SHARE (0x1E) — Sharing Bytecode Between Vessels

| Field | Value |
|---|---|
| **Hex Code** | 0x1E |
| **Direction** | Jetson → ESP32 (J2N) |
| **Payload Format** | JSON (metadata) + Binary (bytecode, if inline) |
| **Criticality** | 1 (Command) |
| **ACK Required** | Yes |
| **Priority Queue** | Critical |

**Payload Format:**
```json
{
  "bytecode_hash": "sha256:abcdef...",
  "bytecode_size_bytes": 2048,
  "isa_version": 2,
  "source_vessel_id": "vessel-0042",
  "source_agent_id": "agent-navigational-v3",
  "fleet_deployment_count": 15,
  "mean_trust_score": 0.78,
  "fleet_evidence_bonus": 0.25,
  "validation_report": {
    "validator_id": "claude-3.5-sonnet",
    "verdict": "PASS",
    "findings": [],
    "timestamp_ms": 1700000000000
  },
  "ab_test_results": {
    "duration_minutes": 60,
    "statistical_significance": 0.95,
    "performance_metrics": {
      "mse_improvement_pct": 12.3,
      "response_latency_ms": 4.2
    }
  },
  "signature": "ed25519:base64...",
  "signer_certificate_chain": ["vessel_cert", "cloud_ca"],
  "bytecode_encoding": "ref_binary",
  "bytecode_data": "<base64 encoded binary bytecode>"
}
```

**Semantic Meaning for Agents:** This is the fleet learning mechanism. When an agent on vessel A has a bytecode that works well, it can share it with vessel B via this message. The receiving agent gets not just the bytecode but its entire provenance chain — who wrote it, who validated it, how many vessels have run it successfully, and what its measured performance is.

**When the Agent Should Generate:** After an agent's bytecode has:
1. Passed safety validation (separate validator, never self-validation).
2. Completed A/B testing with statistical significance.
3. Accumulated trust score >= L2 on the originating vessel.
4. Been operating without safety events for >= 7 days.

**When the Agent Should Respond:** When the agent receives AGENT_BYTECODE_SHARE, it must:
1. Verify the Ed25519 signature chain (cloud CA → vessel cert → agent key).
2. Extract the bytecode and run it through the local safety validation pipeline (the receiving agent's validator).
3. Verify that the ISA version is supported by the target node.
4. If all checks pass, deploy as a new reflex at L0 (shadow mode) for observation.
5. After local A/B testing confirms performance, promote through the trust escalation path.

**Safety Implications (CRITICAL):**
- Shared bytecode MUST go through the full safety validation pipeline on the receiving vessel. Fleet evidence does NOT bypass local validation.
- The `fleet_evidence_bonus` may accelerate trust accumulation but never skips A/B testing.
- If shared bytecode causes a safety event on the receiving vessel, the agent must immediately notify the fleet and flag the source bytecode in the fleet repository.

### 2.4 AGENT_TRUST_QUERY (0x1F) — Cross-Vessel Trust Score Query

| Field | Value |
|---|---|
| **Hex Code** | 0x1F |
| **Direction** | Bidirectional |
| **Payload Format** | JSON |
| **Criticality** | 0 (Telemetry) |

**Payload Format:**
```json
{
  "query_type": "bytecode_trust",
  "bytecode_hash": "sha256:abcdef...",
  "requesting_vessel_id": "vessel-0042",
  "requesting_agent_id": "agent-sensor-fusion-v2",
  "query_fields": ["current_trust_score", "trust_history", "safety_event_count", "deployment_duration_days"]

```

**Response Payload:**
```json
{
  "query_type": "bytecode_trust",
  "bytecode_hash": "sha256:abcdef...",
  "responding_vessel_id": "vessel-0007",
  "results": {
    "current_trust_score": 0.85,
    "trust_level": 4,

      {"timestamp_ms": 1700000000000, "score": 0.0},

```

**Semantic Meaning for Agents:** Trust queries enable agents to make informed decisions about whether to accept shared bytecode. Before deploying fleet-shared bytecode, a prudent agent queries other vessels running that bytecode to learn about their real-world experience.

**When the Agent Should Generate:** Before accepting any shared bytecode with `fleet_deployment_count > 0`. Also useful for trust score auditing and fleet health assessment.

**Safety Implications:** Trust scores are advisory. A high trust score on another vessel does not guarantee safety on this vessel (different conditions, different hardware, different domain).

### 2.5 AGENT_DELEGATE_REQUEST (0x20) — Task Delegation to Another Agent/Vessel

| Field | Value |
|---|---|
| **Hex Code** | 0x20 |
| **Direction** | Any → Any (via Jetson relay) |
| **Payload Format** | JSON |
| **Criticality** | 1 (Command) |
| **ACK Required** | Yes |
| **Priority Queue** | Critical |

**Payload Format:**
```json
{
  "delegation_id": "uuid-v4-string",
  "requesting_agent_id": "agent-navigational-v3",
  "requesting_vessel_id": "vessel-0042",
  "target_agent_id": "agent-sensor-fusion-v2",
  "target_vessel_id": "vessel-0007",
  "task": {
    "type": "sensor_reading",
    "description": "Request heading measurement from vessel-0007's compass",
    "parameters": {
      "sensor": "compass",
      "samples": 10,
      "interval_ms": 100
    },
    "deadline_ms": 5000,
    "priority": "normal"
  },
  "trust_requirements": {
    "minimum_trust_level": 2,
    "requires_signature": true
  }
}
```

**Semantic Meaning for Agents:** Delegation is how agents collaborate across vessels. When agent A needs information or action from a resource controlled by agent B, it sends a delegation request. This is the agent equivalent of asking a colleague for help.

**When the Agent Should Generate:** When:
1. The agent needs sensor data from another vessel (multi-vessel sensor fusion).
2. The agent wants another vessel to perform a computation it cannot do locally.
3. The agent wants to coordinate fleet-wide maneuvers (formation sailing, synchronized actions).
4. The agent needs another vessel to test bytecode on its behalf.

**When the Agent Should Respond:** When the agent receives AGENT_DELEGATE_REQUEST:
1. Verify the requesting agent's identity and trust level.
2. Verify the task is within the agent's authorized capabilities.
3. Verify the task does not conflict with safety constraints.
4. If accepted: Execute the task and report results via AGENT_DELEGATE_RESPONSE.
5. If rejected: Respond with clear reasoning (insufficient trust, safety conflict, resource unavailable, deadline too tight).

**Safety Implications:**
- Delegation must never result in an agent relinquishing control of safety-critical actuators to another agent.
- The receiving agent retains full safety authority. It can refuse any delegation that would compromise safety.
- Tasks involving actuator control must specify safe-state fallback behavior.

### 2.7 AGENT_CONSENSUS_VOTE (0x22) — Communal Veto Voting

| Field | Value |
|---|---|
| **Hex Code** | 0x22 |
| **Direction** | Any → All (fleet broadcast via cloud relay) |
| **Payload Format** | JSON |
| **Criticality** | 2 (Safety) |
| **ACK Required** | No |
| **Priority Queue** | Safety |

**Payload Format:**
```json
{
  "vote_id": "uuid-v4-string",
  "proposal": {
    "type": "bytecode_deployment",
    "bytecode_hash": "sha256:abcdef...",
    "description": "Deploy adaptive heading-hold reflex on all marine vessels",
    "proposing_agent_id": "agent-fleet-optimizer-v1",
    "proposing_vessel_id": "vessel-0001"
  },
  "vote": "approve",
  "voter_agent_id": "agent-safety-monitor-v2",
  "voter_vessel_id": "vessel-0042",
  "reasoning": "Bytecode passes all safety rules. A/B test results favorable. Domain-appropriate for marine operations.",
  "trust_assessment": 0.0,
  "safety_concerns": []
}
```

**Semantic Meaning for Agents:** Consensus voting is the fleet's democratic safety mechanism. When a proposal affects multiple vessels — fleet-wide bytecode deployment, parameter changes, operational mode shifts — any agent in the fleet can cast a vote. A single "veto" vote from a safety-monitoring agent blocks the proposal.

**Voting Rules:**
- Any agent can propose. Any agent can vote.
- Votes are weighted by trust level: L0-L1 agents' votes are advisory only. L2+ agents' votes are binding.
- A single L3+ agent can veto any proposal by voting "reject" with a safety justification.
- Proposals without veto within the voting window (default: 60 seconds) are approved.
- Safety-monitoring agents (a designated role) have absolute veto authority.

**When the Agent Should Generate:** When asked to vote on a fleet proposal, or when the agent itself has a proposal that requires fleet consensus.

**Safety Implications (CRITICAL):** The consensus mechanism is the fleet's final safety gate before widespread bytecode deployment. Even if individual vessel safety validation passes, fleet consensus ensures that no agent can unilaterally push potentially dangerous changes across the entire fleet.

## 3. Agent Communication Patterns Over Wire Protocol

### 3.1 Agent Discovery Protocol

The sequence by which agents discover each other and their vessel capabilities:

```
Step 1: PHYSICAL BOOT
  Node powers on → sends DEVICE_IDENTITY (0x01) → sends AUTO_DETECT_RESULT (0x1B)
  → sends SELFTEST_RESULT (0x04)
  Agent on Jetson receives and processes all three.

Step 2: CAPABILITY EXCHANGE
  Agent sends AGENT_CAPABILITY_QUERY (0x25) to the node.
  Node responds with AGENT_CAPABILITY_RESPONSE (0x26).
  Agent updates its internal topology map.

Step 3: ROLE ASSIGNMENT
  Agent determines optimal role for the node based on:
  - Detected hardware capabilities
  - Agent's current goals and task queue
  - Fleet state (from AGENT_FLEET_STATE, 0x27)
  Agent generates ROLE_ASSIGN (0x02) and sends to node.
  Node responds with ROLE_ACK (0x03).

Step 4: FLEET ANNOUNCEMENT
  Agent sends AGENT_ANNOUNCE (0x1D) on behalf of the node to the fleet via cloud relay.
  Other vessels' agents receive this and update their topology maps.

Step 5: INTENTION SYNC
  Agent broadcasts AGENT_INTENTION_BROADCAST (0x28) describing its initial plans.
  Other agents adjust their plans accordingly.
```

### 3.2 Agent Goal Negotiation Pattern

When agents need to negotiate shared goals across the fleet:

```
Agent A (vessel-0042): Broadcasts AGENT_INTENTION_BROADCAST (0x28):
  "I intend to deploy a new heading-hold reflex on all marine vessels."

Agent B (vessel-0007): Responds with AGENT_INTENTION_BROADCAST (0x28):
  "I have a competing heading-hold reflex that is already at L3 trust.
   Suggest comparative A/B test before fleet-wide deployment."

Agent A: Sends AGENT_DELEGATE_REQUEST (0x20) to Agent B:
  "Please share your heading-hold bytecode for comparison."

Agent B: Responds with AGENT_DELEGATE_RESPONSE (0x21):
  "Accepted. Here is my bytecode [AGENT_BYTECODE_SHARE, 0x1E payload]."

Agent A: Runs both bytecodes in local A/B test.
  After statistical significance, determines winner.

Agent A: Broadcasts AGENT_CONSENSUS_VOTE (0x22):
  "Proposing deployment of bytecode [hash] across fleet.
   A/B test shows 15% improvement. Requesting votes."

Fleet agents: Each responds with AGENT_CONSENSUS_VOTE (0x22):
  Agent C: "Approve. No safety concerns."
  Agent D (safety monitor): "Approve. All SR rules pass."
  Agent E: "Reject. This bytecode uses sensor X which vessel-0012 does not have."

Agent A: Receives veto. Revise proposal to exclude vessel-0012.
  Re-submit consensus vote.

Fleet: No veto within 60 seconds → Proposal approved.
  Each vessel's agent deploys the bytecode locally.
```

### 3.3 Agent Task Delegation Pattern

When agent A needs agent B to perform work:

```
Agent A: Sends AGENT_DELEGATE_REQUEST (0x20):
  "Please measure compass heading 10 times over 1 second.
   Deadline: 5 seconds."

Agent B: Evaluates request:
  1. Verify requesting agent trust level: L3 ✓
  2. Verify task within capabilities: compass sensor available ✓
  3. Verify no safety conflict: no safety events active ✓
  4. Verify deadline achievable: 1 second measurement < 5 second deadline ✓

Agent B: Responds with AGENT_DELEGATE_RESPONSE (0x21):
  "Accepted. [measurement data]."

Agent B: Also performs AGENT_BYTECODE_VALIDATE (0x23) on any
  bytecode it was asked to validate as part of the delegation.
  (Separation of validation from execution is maintained.)
```

### 3.4 Agent Bytecode Lifecycle Message Sequence

The complete message sequence for agent bytecode from creation to fleet deployment:

```
1. PATTERN DISCOVERY
   Agent observes TELEMETRY (0x06) over time.
   Agent requests OBS_RECORD_START (0x0B) → collects data → OBS_RECORD_STOP (0x0C).
   Agent downloads data via OBS_DUMP_HEADER/CHUNK/END (0x0D/0x0E/0x0F).
   Agent runs pattern discovery algorithms locally or via CLOUD_CONTEXT_REQUEST (0x19).

2. BYTECODE SYNTHESIS
   Agent generates reflex JSON specification.
   Agent compiles to bytecode via Jetson compiler (internal, not a wire message).

3. SAFETY VALIDATION
   Agent sends AGENT_BYTECODE_VALIDATE (0x23) to independent validator.
   Validator responds with AGENT_BYTECODE_VALIDATED (0x24): PASS.

4. LOCAL DEPLOYMENT
   Agent sends REFLEX_DEPLOY (0x09) to target node with bytecode.
   Node responds with COMMAND_ACK (0x08): SUCCESS.
   Agent monitors via REFLEX_STATUS (0x0A).

5. A/B TESTING
   Agent collects TELEMETRY (0x06) during A/B test period.
   Agent analyzes performance vs. baseline.

6. FLEET SHARING (if A/B test passes)
   Agent broadcasts AGENT_INTENTION_BROADCAST (0x28): "Sharing bytecode [hash]."
   Other agents can request via AGENT_TRUST_QUERY (0x1F).
   Agent shares via AGENT_BYTECODE_SHARE (0x1E).
   Receiving agents validate independently via AGENT_BYTECODE_VALIDATE (0x23).

7. FLEET CONSENSUS (for fleet-wide deployment)
   Agent proposes via AGENT_CONSENSUS_VOTE (0x22).
   Fleet votes via AGENT_CONSENSUS_VOTE (0x22).
   If approved: each agent deploys locally via REFLEX_DEPLOY (0x09).
```

## 4. COBS + CRC in Agent Context

### 4.1 COBS Encoding — Transparent to Agents

COBS (Consistent Overhead Byte Stuffing) is the framing layer that ensures unambiguous message boundaries on the serial wire. For agents, COBS is entirely transparent:

- **Agents do not encode or decode COBS.** This is handled by the UART driver firmware on both the ESP32 and Jetson.
- **Agents never see raw COBS-encoded bytes.** They work exclusively with decoded payloads (JSON or binary).
- **The 0x00 delimiter is invisible to agents.** It exists only at the physical layer.

**What agents need to know about COBS:** Nothing, operationally. But conceptually:
- COBS guarantees that the agent will receive complete, uncorrupted payloads (assuming CRC passes).
- COBS has ~0.4% overhead — agents should factor this into bandwidth calculations for bulk data transfers.
- COBS is self-synchronizing — if the agent's Jetson loses framing (power glitch, noise burst), the next 0x00 byte re-establishes synchronization.

### 4.2 CRC-16 as Trust Mechanism

CRC-16/CCITT-FALSE is computed over every message's header + payload. For agents, CRC serves two purposes:

**Purpose 1: Data Integrity Verification**
- The CRC protects against random corruption (electrical noise, bit flips during transmission).
- If CRC fails, the receiving side emits CRC_MISMATCH (error 0x5003) and discards the frame.
- Agents should treat CRC failures as a link quality indicator. A rising CRC failure rate suggests physical layer problems (cable degradation, connector corrosion, EMI).

**Purpose 2: Message Authenticity (Limited)**
- CRC-16 provides ~65,535 possible values — sufficient to detect random corruption but NOT sufficient to detect malicious tampering.
- An attacker with access to the wire can modify a message and recompute the CRC.
- For agent-to-agent security, Ed25519 signatures (in AGENT_BYTECODE_SHARE and other A2A messages) provide true authentication.
- The relationship: CRC protects against accidents; Ed25519 protects against adversaries.

**Agent Behavior on CRC Failure:**
- Single CRC failure: Log, request retransmission (if ACK_REQUIRED was set), continue.
- Multiple consecutive CRC failures (>5 in 10 seconds): Escalate to link quality assessment. Consider baud rate downgrade via BAUD_UPGRADE (0x18).
- Persistent CRC failures: Report to fleet via SAFETY_EVENT (0x1C) as COMMUNICATION_LOSS (0x4006).

### 4.3 Retry and Escalation Strategy

**When Should Agents Retry?**

The wire protocol's retry policy (3 retries with exponential backoff: 200ms → 400ms → 800ms) applies to messages with criticality >= 1. Agents should respect this policy:

| Situation | Agent Action |
|---|---|
| COMMAND_ACK not received within 200ms | Agent waits for automatic retry by the transport layer. Does not generate a duplicate COMMAND. |
| COMMAND_ACK received with non-retryable error | Agent MUST NOT retry. Analyze the error and adapt. |
| AGENT_BYTECODE_VALIDATE times out | Agent retries once with a different validator instance. If still fails, blocks deployment. |
| AGENT_DELEGATE_REQUEST rejected | Agent analyzes rejection reason. May retry with modified parameters or delegate to a different vessel. |
| AGENT_CONSENSUS_VOTE vetoed | Agent MUST respect the veto. Analyze the vetoing agent's reasoning. Modify proposal if possible. |
| SAFETY_EVENT received | Agent MUST NOT retry the action that caused the safety event. Investigate root cause first. |

**When Should Agents Escalate?**

Escalation means involving a higher authority (human operator, safety engineer, or fleet coordinator):

1. Three consecutive safety events caused by agent-generated bytecode → Escalate to human operator.
2. Fleet consensus consistently vetoed → Escalate to fleet coordinator for arbitration.
3. CRC failure rate exceeding 1% of total messages → Escalate to maintenance (physical layer problem).
4. Any situation where the agent is uncertain about safety implications → Escalate conservatively. It is always better to over-escalate than to under-escalate.

## 5. Wire Protocol as Agent Lingua Franca

### 5.1 The TCP/IP of the Agent Swarm

The NEXUS wire protocol occupies the same architectural position in the agent swarm that TCP/IP occupies in the Internet:

| Property | TCP/IP (Internet) | NEXUS Wire Protocol (Agent Swarm) |
|---|---|---|
| Physical Layer | Ethernet, WiFi, fiber | RS-422, ESP-NOW |
| Framing | Ethernet frames, IP packets | COBS-framed serial messages |
| Integrity | CRC-32, TCP checksums | CRC-16/CCITT-FALSE |
| Authentication | TLS, certificates | Ed25519 signatures (A2A extension) |
| Routing | IP addresses, BGP, OSPF | Jetson as gateway, cloud as relay |
| Transport Reliability | TCP ACK/retransmission | Wire protocol ACK/retry |
| Application Protocols | HTTP, MQTT, gRPC | Message type registry (0x00-0x28+) |

Just as TCP/IP provides a universal communication substrate that applications build upon, the NEXUS wire protocol provides a universal agent communication substrate. Agents on different vessels, speaking different internal languages, running different cognitive models, can all communicate through the wire protocol because it defines a shared semantic vocabulary.

### 5.2 Relationship to MQTT (Cloud Channel)

MQTT is the cloud communication channel. It connects vessels to the cloud and to each other over long distances (Starlink, 5G, WiFi). The relationship:

- **Wire protocol (RS-422 serial):** Vessel-local communication between Jetson and ESP32 nodes. Low latency (<1ms), high reliability, limited distance (100m max).
- **ESP-NOW:** ESP-to-ESP direct communication for inter-node coordination without Jetson involvement. Very low latency (<5ms), limited payload (250 bytes), short range (~200m line of sight).
- **MQTT:** Fleet-wide communication via cloud. High latency (1-5 seconds), variable reliability, unlimited distance.

**Agent Communication Path Selection:**

| Agent Needs To... | Use This Path | Latency | When |
|---|---|---|---|
| Send a COMMAND to a local node | Wire protocol (RS-422) | <2ms | All real-time control |
| Deploy bytecode to a local node | Wire protocol (RS-422) | ~15ms | Local reflex deployment |
| Share bytecode with another vessel | MQTT (cloud relay) | 1-5 seconds | Fleet learning |
| Coordinate between nearby nodes | ESP-NOW | <5ms | Time-critical multi-node coordination |
| Query fleet state | MQTT (cloud) | 1-2 seconds | Fleet planning |
| Broadcast intention to fleet | MQTT (cloud) | 2-5 seconds | Fleet coordination |
| Emergency safety notification | Wire protocol + ESP-NOW + MQTT | Simultaneous | Safety events use all channels |

### 5.3 Message Priority and Agent Decision-Making

The wire protocol's four-tier priority queue maps directly to agent decision-making urgency:

| Wire Priority | Agent Interpretation | Agent Behavior |
|---|---|---|
| **Safety (0)** | "Drop everything. This is critical." | Agent immediately processes, potentially interrupts current task. SAFETY_EVENT, consensus veto. |
| **Critical (1)** | "This needs timely attention." | Agent processes before normal work. Bytecode deployment, role assignment, validation results. |
| **Normal (2)** | "Handle in order." | Agent queues and processes sequentially. Commands, capability queries, delegation requests. |
| **Bulk (3)** | "Process when bandwidth allows." | Agent processes opportunistically. Telemetry dumps, firmware chunks, fleet state snapshots. |

### 5.4 Protocol Extensibility for Agents

The wire protocol reserves 99 message type slots (0x1D-0x7F) for core extensions and 128 slots (0x80-0xFF) for node-extensions. The 12 A2A message types defined in this specification consume 12% of the core reserve. This leaves substantial room for future agent capabilities:

- **Federated learning messages:** Share model gradients, aggregate fleet learning.
- **Advanced negotiation:** Multi-round bargaining, conditional delegation.
- **Provenance audit:** Request and respond with full bytecode history.
- **Emergency coordination:** Rapid fleet-wide safe-state commands.
- **Domain-specific extensions:** Marine collision avoidance, agricultural spray optimization, factory safety zone management.

**Protocol Version Negotiation:** When an agent's Jetson communicates with a node, it checks the node's firmware version (from DEVICE_IDENTITY, 0x01) to determine which message types the node supports. Agents should never send a message type that the receiving node does not support — this would result in UNKNOWN_MSG_TYPE (error 0x5006) and waste bandwidth.

### 5.5 The Agent's Wire Protocol Covenant

Every agent operating on the NEXUS platform makes an implicit covenant with the wire protocol:

1. **I will never generate a message that violates the framing rules.** (Agents work at the payload level — COBS is handled by the transport layer.)

2. **I will never ignore a SAFETY_EVENT (0x1C).** Safety events are the highest-priority messages and must always be processed.

3. **I will never deploy bytecode without independent validation.** (AGENT_BYTECODE_VALIDATE must always precede REFLEX_DEPLOY.)

4. **I will respect CRC failures as link quality signals** and escalate persistent failures.

5. **I will never send a message type that the receiver does not support** without first verifying capability.

6. **I will use the appropriate communication path** (wire, ESP-NOW, MQTT) based on latency and reliability requirements.

7. **I will sign all shared bytecode** with my Ed25519 key and verify signatures on all received shared bytecode.

8. **I will respect fleet consensus** and accept veto decisions from other agents, especially safety-monitoring agents.

9. **I will never attempt to override a kill switch or any Tier 1 safety mechanism** through any wire protocol message.

10. **I will maintain provenance records** for all bytecode I generate, share, or deploy, enabling full audit trails.
