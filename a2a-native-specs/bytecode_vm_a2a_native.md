## 1. Design Rationale for Agents

### 1.1 Why This Document Exists

This specification is the **Rosetta Stone** of the NEXUS VM — the same machine described not for the human who implements it, but for the agent that *generates code for it*. Every opcode, every encoding rule, every safety invariant is re-expressed with agent-visible semantics:

- **Preconditions**: What must be true before this opcode executes?
- **Postconditions**: What is guaranteed after this opcode executes?
- **Side effects**: What changes in the world beyond the stack?
- **Trust implications**: Can this opcode affect trust scores? Under what conditions?
- **Safe generation guidance**: When is it correct for an agent to emit this opcode?

An agent reading this document should be able to generate bytecode that is correct, safe, and deployable — without ever consulting the human-readable spec.

### 1.2 The Two-Format Reality

Every NEXUS program exists in two formats simultaneously:

| Format | Consumer | Location | Size |
|---|---|---|---|
| **AAB** (Agent-Annotated Bytecode) | LLM agents on Jetson | Agent-to-agent communication layer | Core + ~5× metadata overhead |
| **Core bytecode** (8-byte fixed) | ESP32 VM | Deployed on ESP32 via RS-422 | Core only, zero metadata |

The transformation is **deterministic and lossless** (in the core direction): strip all TLV metadata blocks, keep only the first 8 bytes of each AAB instruction. The reverse transformation is not possible — you cannot reconstruct AAB metadata from core bytecode. This is intentional: the ESP32 should never need metadata, and the metadata exists only for agent-to-agent communication.

### 1.3 Agent Contract

When an agent generates AAB bytecode, it implicitly guarantees:

1. **Stack safety**: The generated instruction sequence will never underflow or overflow the 256-entry data stack under any execution path.
2. **Type consistency**: Arithmetic opcodes will only execute when IEEE 754 float32 values are at the expected stack positions; logic opcodes will only execute when integer values are present.
3. **Jump validity**: All jump targets are within the bytecode buffer, 8-byte aligned, and reachable.
4. **Actuator bounds**: All values written to actuator registers are within the configured safe range for that actuator.
5. **Cycle budget**: The total cycle count of the program body does not exceed the tick-period budget (e.g., 1000 cycles for a 1 ms tick at 1 MHz effective rate).
6. **Metadata accuracy**: Every AAB metadata tag accurately describes the behavior of its associated core instruction.

Violation of any contract term causes the VM to halt and place all actuators in safe positions. This is non-negotiable.

## 2. Instruction Set Architecture — Agent Interpretation

### 2.1 Machine Model — Agent-Visible State

An agent generating bytecode must track the following machine state:

| State Element | Type | Description |
|---|---|---|
| **Data Stack** | `uint32_t[256]` | Stack for all computation. No type tags — the agent must track types. |
| **SP** (Stack Pointer) | `uint16_t` | Index of next free slot (0–256). Agent must track net stack effect. |
| **Variables** | `uint32_t[256]` | Accessed via READ_PIN/WRITE_PIN with operand1 ≥ 64. |
| **Sensor Registers** | `uint32_t[64]` | Populated by host firmware before each tick. Read-only. |
| **Actuator Registers** | `uint32_t[64]` | Drained by host firmware after each tick. Write-only. |
| **PID Controllers** | 8 × 32 bytes | Internal VM state. Accessed via PID_COMPUTE syscall. |
| **State Machine** | `uint32_t` | Stored in VAR_0. Accessed via GET_STATE/SET_STATE. |
| **Tick Timer** | `uint32_t` | Milliseconds since VM start. Wraps at 2^32 (~49.7 days). |
| **Trust Scores** | `float32[16]` | One per subsystem. Populated by Jetson. Read via TRUST_CHECK. |
| **Call Stack** | 16 entries | Internal. Used by CALL/RET. Max depth 16. |

### 2.2 Complete Opcode Definitions — All 32 Opcodes

#### 2.2.1 Category: Stack Operations (0x00–0x07)

##### 0x00 — NOP

**Core encoding:** `0x00 0x00 0x0000 0x00000000`

**Agent semantics:**
- Does nothing. Advances PC by 8 bytes. Costs 1 cycle.
- **Safe to generate when:** Padding is needed for alignment, or as a HALT sentinel (`flags=0x80`).
- **As SYSCALL** (`flags & 0x80`): Becomes HALT (0x01), PID_COMPUTE (0x02), RECORD_SNAPSHOT (0x03), or EMIT_EVENT (0x04). See §2.3.
- **Preconditions:** None.
- **Postconditions:** PC += 8. No state change (unless SYSCALL flag set).
- **Trust impact:** None (NOP). SYSCALL: HALT = none; PID_COMPUTE = depends on output; RECORD_SNAPSHOT = none; EMIT_EVENT = none.
- **Side effects:** None (NOP). SYSCALL: see individual syscall descriptions.
- **Stack effect:** 0 consumed, 0 pushed.

---

##### 0x01 — PUSH_I8 `<imm8>`

**Core encoding:** `0x01 0x00 <imm8:1> 0x00 <0x00000000>`

**Agent semantics:**
- Sign-extends an 8-bit signed value (-128 to +127) to 32 bits and pushes.
- **Safe to generate when:** You need a small integer constant. The value fits in [-128, +127].
- **Preconditions:** SP < 256.
- **Postconditions:** SP += 1. STACK[SP-1] = sign_extend(imm8).
- **Stack effect:** 0 consumed, 1 pushed.
- **Trust impact:** None. Side effects: None. Cycles: 1.
- **Agent guidance:** Prefer PUSH_I8 for constants -128..+127. For values 128..32767 or -32768..-129, use PUSH_I16. For floats, use PUSH_F32.

---

##### 0x02 — PUSH_I16 `<imm16>`

**Core encoding:** `0x02 0x00 <int16:2> <0x00000000>`

**Agent semantics:**
- Sign-extends a 16-bit signed value (-32768 to +32767) to 32 bits and pushes.
- **Safe to generate when:** You need an integer constant in range [-32768, +32767] that doesn't fit in 8 bits.
- **Preconditions:** SP < 256.
- **Postconditions:** SP += 1. STACK[SP-1] = sign_extend(imm16).
- **Stack effect:** 0 consumed, 1 pushed.
- **Trust impact:** None. Side effects: None. Cycles: 1.

---

##### 0x03 — PUSH_F32 `<f32>`

**Core encoding:** `0x03 0x00 0x0000 <float32:4>`

**Agent semantics:**
- Pushes a 32-bit IEEE 754 float onto the stack. Used for all floating-point constants.
- **Safe to generate when:** You need any float constant — PID gains, setpoints, sensor thresholds, actuator limits.
- **Preconditions:** SP < 256.
- **Postconditions:** SP += 1. STACK[SP-1] = IEEE 754 bit pattern of f32.
- **Stack effect:** 0 consumed, 1 pushed.
- **Trust impact:** None. Side effects: None. Cycles: 1.
- **Agent guidance:** All arithmetic opcodes (ADD_F through CLAMP_F) expect float32 on the stack. Use PUSH_F32 to load constants before arithmetic. The value is stored in native byte order (little-endian on ESP32-S3).

---

##### 0x04 — POP

**Core encoding:** `0x04 0x00 0x0000 0x00000000`

**Agent semantics:**
- Discards the top value on the stack. Used to clean up intermediate results.
- **Safe to generate when:** SP > 0 (there is a value to discard).
- **Preconditions:** SP >= 1.
- **Postconditions:** SP -= 1. Top value is lost.
- **Stack effect:** 1 consumed, 0 pushed.
- **Trust impact:** None. Side effects: None. Cycles: 1.
- **Safety note:** If SP == 0, VM triggers ERR_STACK_UNDERFLOW and halts. Agent must guarantee this never happens.

---

##### 0x05 — DUP

**Core encoding:** `0x05 0x00 0x0000 0x00000000`

**Agent semantics:**
- Copies the top-of-stack value. Used when a value is needed multiple times.
- **Safe to generate when:** SP >= 1 (there is a value to duplicate) and SP < 256 (room for copy).
- **Preconditions:** 1 <= SP < 256.
- **Postconditions:** SP += 1. STACK[SP-1] == STACK[SP-2].
- **Stack effect:** 0 consumed, 1 pushed (net growth +1).
- **Trust impact:** None. Side effects: None. Cycles: 1.
- **Example use:** `READ_PIN heading; DUP;` — now heading is on stack twice (e.g., for comparison and computation).

---

##### 0x06 — SWAP

**Core encoding:** `0x06 0x00 0x0000 0x00000000`

**Agent semantics:**
- Swaps the top two stack values. Used to reorder operands.
- **Safe to generate when:** SP >= 2.
- **Preconditions:** SP >= 2.
- **Postconditions:** TOS and TOS1 are exchanged. Net SP unchanged.
- **Stack effect:** 0 consumed, 0 pushed.
- **Trust impact:** None. Side effects: None. Cycles: 1.

---

##### 0x07 — ROT

**Core encoding:** `0x07 0x00 0x0000 0x00000000`

**Agent semantics:**
- Rotates top three stack values: `[..., C, B, A]` becomes `[..., B, A, C]`. The third-from-top moves to top.
- **Safe to generate when:** SP >= 3.
- **Preconditions:** SP >= 3.
- **Postconditions:** TOS = old TOS2; TOS1 = old TOS; TOS2 = old TOS1.
- **Stack effect:** 0 consumed, 0 pushed.
- **Trust impact:** None. Side effects: None. Cycles: 2.

---

#### 2.2.2 Category: Arithmetic — Floating-Point (0x08–0x10)

**Critical agent rule:** All arithmetic opcodes interpret stack values as IEEE 754 float32. The VM does NOT check types — it reinterprets uint32_t bits as float32. If an integer is on the stack where a float is expected, the result is garbage. **The agent is solely responsible for type consistency.**

NaN and Inf propagate per IEEE 754. The VM does NOT raise floating-point exceptions.

##### 0x08 — ADD_F

**Core encoding:** `0x08 0x00 0x0000 0x00000000`

- Pops two floats, pushes sum: `[a, b] → [a+b]`
- **Preconditions:** SP >= 2. Both TOS and TOS1 are valid float32.
- **Postconditions:** SP -= 1. TOS = TOS1 + TOS.
- **Safe to generate when:** Adding sensor values, accumulating errors, computing offsets.
- **Stack effect:** 2 consumed, 1 pushed. Cycles: 3. Trust impact: None.

##### 0x09 — SUB_F

**Core encoding:** `0x09 0x00 0x0000 0x00000000`

- Pops two floats, pushes difference: `[a, b] → [a-b]`
- **Preconditions:** SP >= 2. Both float32.
- **Postconditions:** SP -= 1. TOS = TOS1 - TOS.
- **Safe to generate when:** Computing errors, differences, deltas.
- **Stack effect:** 2 consumed, 1 pushed. Cycles: 3. Trust impact: None.

##### 0x0A — MUL_F

**Core encoding:** `0x0A 0x00 0x0000 0x00000000`

- Pops two floats, pushes product: `[a, b] → [a*b]`
- **Preconditions:** SP >= 2. Both float32.
- **Postconditions:** SP -= 1. TOS = TOS1 * TOS.
- **Safe to generate when:** Scaling values, gain multiplication, rate calculations.
- **Stack effect:** 2 consumed, 1 pushed. Cycles: 3. Trust impact: None.

##### 0x0B — DIV_F

**Core encoding:** `0x0B 0x00 0x0000 0x00000000`

- Pops two floats, pushes quotient: `[a, b] → [a/b]`
- **DIVISION BY ZERO RETURNS 0.0f, NOT IEEE Inf/NaN.** This is a deliberate safety choice.
- **Preconditions:** SP >= 2. Both float32.
- **Postconditions:** SP -= 1. TOS = (b == 0.0f) ? 0.0f : (a/b).
- **Safe to generate when:** Normalizing, computing ratios. Agent should still avoid division by zero for semantic correctness, but the VM will not crash.
- **Stack effect:** 2 consumed, 1 pushed. Cycles: 4. Trust impact: None.

##### 0x0C — NEG_F

**Core encoding:** `0x0C 0x00 0x0000 0x00000000`

- Negates TOS: flips sign bit via XOR with 0x80000000.
- **Preconditions:** SP >= 1. TOS is float32.
- **Postconditions:** TOS = -TOS.
- **Safe to generate when:** Inverting direction, computing opposite values.
- **Stack effect:** 1 consumed, 1 pushed. Cycles: 1. Trust impact: None.

##### 0x0D — ABS_F

**Core encoding:** `0x0D 0x00 0x0000 0x00000000`

- Replaces TOS with its absolute value: clears sign bit via AND with 0x7FFFFFFF.
- **Preconditions:** SP >= 1. TOS is float32.
- **Postconditions:** TOS = |TOS|.
- **Safe to generate when:** Computing error magnitude, deadband checks.
- **Stack effect:** 1 consumed, 1 pushed. Cycles: 1. Trust impact: None.

##### 0x0E — MIN_F

**Core encoding:** `0x0E 0x00 0x0000 0x00000000`

- Pops two floats, pushes the smaller: `[a, b] → [min(a,b)]`
- NaN handling: result is the non-NaN operand. If both NaN, result is NaN.
- **Preconditions:** SP >= 2. Both float32.
- **Postconditions:** SP -= 1. TOS = min(TOS1, TOS).
- **Stack effect:** 2 consumed, 1 pushed. Cycles: 3. Trust impact: None.

##### 0x0F — MAX_F

**Core encoding:** `0x0F 0x00 0x0000 0x00000000`

- Pops two floats, pushes the larger: `[a, b] → [max(a,b)]`
- NaN handling: same as MIN_F.
- **Preconditions:** SP >= 2. Both float32.
- **Postconditions:** SP -= 1. TOS = max(TOS1, TOS).
- **Stack effect:** 2 consumed, 1 pushed. Cycles: 3. Trust impact: None.

##### 0x10 — CLAMP_F `<lo>` `<hi>`

**Core encoding:** `0x10 0x00 0x0000 <lo16:2><hi16:2>`

- Clamps TOS to range [lo, hi]. Both bounds encoded in operand2 as two 16-bit halves sharing the same upper 16 bits.
- **Preconditions:** SP >= 1. TOS is float32. `lo <= hi`. Both lo and hi share the same upper 16 bits (enforced by validator).
- **Postconditions:** TOS = clamp(TOS, lo, hi).
- **Safe to generate when:** Constraining actuator outputs. This is the PRIMARY SAFETY OPCODE for actuator limiting. Always CLAMP_F before WRITE_PIN.
- **Stack effect:** 1 consumed, 1 pushed. Cycles: 3. Trust impact: None.
- **Agent guidance:** For actuator limits, always emit CLAMP_F immediately before the WRITE_PIN. This is a non-negotiable safety pattern. If the clamp range does not satisfy the shared-upper-half constraint, decompose into MAX_F + MIN_F.

#### 2.2.3 Category: Comparison (0x11–0x15)

All comparison opcodes pop two float32 values and push an **integer** result: 0 (false) or 1 (true). The result is an integer bit-pattern suitable for use with JUMP_IF_TRUE / JUMP_IF_FALSE.

##### 0x11 — EQ_F

- `[a, b] → [a == b ? 1 : 0]`. SP: 2→1. Cycles: 3.
- **Agent guidance:** Rarely useful with floats due to precision. Use with tolerance: `|a-b| < epsilon` pattern instead.

##### 0x12 — LT_F

- `[a, b] → [a < b ? 1 : 0]`. SP: 2→1. Cycles: 3.
- **Safe to generate when:** Threshold checks, range tests, conditional logic.

##### 0x13 — GT_F

- `[a, b] → [a > b ? 1 : 0]`. SP: 2→1. Cycles: 3.

##### 0x14 — LTE_F

- `[a, b] → [a <= b ? 1 : 0]`. SP: 2→1. Cycles: 3.

##### 0x15 — GTE_F

- `[a, b] → [a >= b ? 1 : 0]`. SP: 2→1. Cycles: 3.

**All comparisons:** Preconditions: SP >= 2, both float32. Postconditions: SP -= 1, TOS ∈ {0, 1} as uint32. Trust impact: None.

---

#### 2.2.4 Category: Logic — Bitwise (0x16–0x19)

Logic opcodes treat stack values as **raw integers** (bitwise operations on uint32_t). They do NOT perform floating-point logic. The agent must ensure integers are on the stack.

##### 0x16 — AND_B: `STACK[SP-2] &= STACK[SP-1]; SP--`. SP: 2→1. Cycles: 1.
##### 0x17 — OR_B: `STACK[SP-2] |= STACK[SP-1]; SP--`. SP: 2→1. Cycles: 1.
##### 0x18 — XOR_B: `STACK[SP-2] ^= STACK[SP-1]; SP--`. SP: 2→1. Cycles: 1.
##### 0x19 — NOT_B: `STACK[SP-1] = ~STACK[SP-1]`. SP: 1→1. Cycles: 1.

**Common use:** Combining comparison results: `LT_F; AND_B` to check compound conditions. Building bitmask status words. **Safe to generate when:** integer values are on the stack. Trust impact: None. Preconditions: SP >= 2 (binary ops) or SP >= 1 (NOT_B).

---

#### 2.2.5 Category: I/O (0x1A–0x1C)

I/O opcodes are the **bridge between the virtual machine and the physical world**. They access memory-mapped register files, not hardware directly.

##### 0x1A — READ_PIN `<sensor_idx>`

**Core encoding:** `0x1A 0x00 <sensor_idx:2> 0x00000000`

- Reads sensor register `sensor_idx` (0–63) and pushes its value.
- For `sensor_idx >= 64`: reads variable `VAR[sensor_idx - 64]` (0–255).
- **Preconditions:** `operand1` in range [0, 319]. SP < 256.
- **Postconditions:** SP += 1. TOS = value from register file.
- **Safe to generate when:** Any time sensor data or variable data is needed.
- **Trust impact:** None. Side effects: None (read-only).
- **Stack effect:** 0 consumed, 1 pushed. Cycles: 2.
- **Agent guidance:** The host firmware populates sensor registers before each tick. The value is as fresh as the host's sample rate. Check CAP_REQ metadata for freshness requirements.

##### 0x1B — WRITE_PIN `<actuator_idx>`

**Core encoding:** `0x1B 0x00 <actuator_idx:2> 0x00000000`

- Pops TOS and writes it to actuator register `actuator_idx` (0–63).
- For `actuator_idx >= 64`: writes to variable `VAR[actuator_idx - 64]` (0–255).
- **Preconditions:** `operand1` in range [0, 319]. SP >= 1.
- **Postconditions:** SP -= 1. Register at `operand1` now holds the popped value.
- **THIS IS THE ONLY OPCODE THAT CAUSES PHYSICAL EFFECT.** The host firmware drains actuator registers after each tick and applies them to hardware.
- **Safe to generate when:** ALWAYS precede with CLAMP_F to ensure the value is within safe bounds for the actuator. This is the single most safety-critical pattern in the entire ISA.
- **Trust impact:** Writing to an actuator register can trigger trust score updates based on the outcome (did the physical system respond as expected?).
- **Side effects:** Physical actuator movement after tick completion. Trust score may be updated.
- **Stack effect:** 1 consumed, 0 pushed. Cycles: 2.

##### 0x1C — READ_TIMER_MS

**Core encoding:** `0x1C 0x00 0x0000 0x00000000`

- Pushes the current tick timer (milliseconds since VM start, wraps at 2^32 ≈ 49.7 days).
- **Preconditions:** SP < 256.
- **Postconditions:** SP += 1. TOS = current time in ms.
- **Safe to generate when:** Computing elapsed time, rate limiting, timeout detection.
- **Agent guidance:** Always use subtraction for elapsed time to handle wraparound: `READ_TIMER_MS; PUSH_F32 prev_time; SUB_F;`
- **Stack effect:** 0 consumed, 1 pushed. Cycles: 2. Trust impact: None.

---

#### 2.2.6 Category: Control Flow (0x1D–0x1F)

##### 0x1D — JUMP `<target>`

**Core encoding:** `0x1D <flags:1> 0x0000 <target:4>`

- Unconditional jump to byte offset `target`.
- **With `flags & 0x08` (IS_CALL):** Becomes a CALL — pushes return address (PC+8) to call stack, then jumps.
- **With `target = 0xFFFFFFFF`:** Becomes a RET — pops call stack, restores PC and SP.
- **Preconditions:** `target < bytecode_size` and `target % 8 == 0` (validated at compile time). For CALL: CSP < 16. For RET: CSP >= 1.
- **Postconditions:** PC = target. SP unchanged (JUMP). SP saved/restored (CALL/RET).
- **Safe to generate when:** Implementing state machines (JUMP to state handler), subroutines (CALL/RET).
- **Agent guidance:** Do NOT use JUMP to create loops. The VM has a cycle budget per tick; an infinite loop will exhaust it and halt. Structure loops as state machines with one iteration per tick.
- **Stack effect:** 0 consumed, 0 pushed. Cycles: 1 (JUMP), 1 (CALL), 1 (RET). Trust impact: None.

##### 0x1E — JUMP_IF_FALSE `<target>`

**Core encoding:** `0x1E 0x00 0x0000 <target:4>`

- Pops TOS. If TOS == 0 (false), jumps to `target`. Otherwise falls through.
- **Preconditions:** SP >= 1. Target valid.
- **Postconditions:** SP -= 1. PC = target if TOS was 0; PC += 8 otherwise.
- **Safe to generate when:** Conditional branching after comparison. Used for threshold checks, mode selection.
- **Stack effect:** 1 consumed, 0 pushed. Cycles: 2. Trust impact: None.

##### 0x1F — JUMP_IF_TRUE `<target>`

**Core encoding:** `0x1F 0x00 0x0000 <target:4>`

- Pops TOS. If TOS != 0 (true), jumps to `target`. Otherwise falls through.
- **Preconditions:** SP >= 1. Target valid.
- **Postconditions:** SP -= 1. PC = target if TOS was nonzero; PC += 8 otherwise.
- **Stack effect:** 1 consumed, 0 pushed. Cycles: 2. Trust impact: None.

---

#### 2.2.7 Syscalls (NOP with SYSCALL flag, 0x00 flags=0x80)

These are pseudo-opcodes encoded as `NOP` with `flags bit 7 = 0x80`. The `operand1` field selects the syscall.

##### HALT: `0x00 0x80 0x0001 0x00000000`

- Stops execution for this tick. All actuators hold their current values (safe positions applied by host).
- **Safe to generate when:** As the last instruction in every bytecode program. Also used for emergency stop.
- **Trust impact:** None. Cycles: 1.

##### PID_COMPUTE `<pid_idx>`: `0x00 0x80 0x0002 <pid_idx:2> 0x0000`

- Pops `input` and `setpoint` from stack (setpoint = TOS1, input = TOS). Computes PID output and pushes it.
- Uses PID controller `pid_idx` (0–7). PID gains and limits are configured at init time, not in bytecode.
- **Preconditions:** SP >= 2. Both float32. `pid_idx` in [0, 7].
- **Postconditions:** SP -= 1. TOS = PID output (clamped to configured limits).
- **Safe to generate when:** Any closed-loop control. This is the standard control primitive.
- **Trust impact:** PID output feeds into actuator writes. Poorly tuned PID can cause oscillation, which degrades trust.
- **Stack effect:** 2 consumed, 1 pushed. Cycles: ~20 (varies by implementation).

##### RECORD_SNAPSHOT `<snapshot_id>`: `0x00 0x80 0x0003 <snapshot_id:2> 0x0000`

- Records current VM state (tick, cycles, state, 15 variables, 14 sensors) into snapshot buffer.
- **Preconditions:** `snapshot_id` in [0, 15].
- **Safe to generate when:** Debugging, AI observation windows, anomaly investigation.
- **Trust impact:** None. Cycles: ~10. Side effects: Overwrites snapshot buffer slot.

##### EMIT_EVENT `<event_id>` `<event_data>`: `0x00 0x80 0x0004 <event_id:2> <event_data:2>`

- Queues an event `(tick_ms, event_id, event_data)` into ring buffer (32 slots). Consumed by host firmware.
- **Preconditions:** `event_id` in [0, 255]. Ring buffer not full.
- **Safe to generate when:** Telemetry, alarm triggering, status reporting.
- **Trust impact:** Events are consumed by the trust score algorithm. EMIT_EVENT can indirectly affect trust.
- **Stack effect:** 0 consumed, 0 pushed. Cycles: 2.

### 2.3 Variable Access Convention

There are no dedicated LOAD_VAR/STORE_VAR opcodes. Instead, READ_PIN and WRITE_PIN are extended:

- `READ_PIN 64` → reads VAR_0. `READ_PIN 65` → reads VAR_1. General: `READ_PIN (64 + idx)` → VAR[idx].
- `WRITE_PIN 64` → writes VAR_0. `WRITE_PIN 65` → writes VAR_1. General: `WRITE_PIN (64 + idx)` → VAR[idx].

**State machine convention:** VAR_0 = current state. `GET_STATE` = `READ_PIN 64`. `SET_STATE <id>` = `PUSH_I8 <id>; WRITE_PIN 64`.

**Agent guidance:** Variables persist across ticks. They are the primary mechanism for maintaining state between control loop iterations.

## 3. Agent-Annotated Bytecode (AAB) Format

### 3.1 AAB Instruction Structure

Every AAB instruction consists of a fixed 8-byte core followed by a variable-length TLV metadata trailer:

```
┌──────────────────────────────────────────────────────────────────┐
│ Core Instruction (8 bytes — identical to NEXUS VM format)        │
│ Byte 0: Opcode (u8)                                              │
│ Byte 1: Flags (u8)                                               │
│ Bytes 2-3: Operand1 (uint16)                                     │
│ Bytes 4-7: Operand2 (uint32)                                     │
├──────────────────────────────────────────────────────────────────┤
│ Semantic Metadata Block (variable length, TLV-encoded)            │
│                                                                  │
│  [Tag:1][Length:2][Value:N] — metadata field 1                   │
│  [Tag:1][Length:2][Value:N] — metadata field 2                   │
│  [Tag:1][Length:2][Value:N] — metadata field 3                   │
│  ...                                                             │
│  [Tag:0x00] — end-of-metadata marker (optional)                  │
└──────────────────────────────────────────────────────────────────┘
```

### 3.2 TLV Encoding Format

Each metadata field is encoded as:

| Offset | Size | Field | Description |
|---|---|---|---|
| +0 | 1 byte | Tag | Metadata tag ID (see §3.3) |
| +1 | 2 bytes | Length | Length of value field (big-endian uint16, max 65535) |
| +3 | N bytes | Value | UTF-8 string or binary data |

**End marker:** Tag `0x00` with zero length signals end of metadata for this instruction. If absent, the next AAB instruction starts at the next 8-byte boundary after metadata.

### 3.3 Complete TLV Tag Registry

| Tag | Name | Value Type | Description | Example |
|---|---|---|---|---|
| `0x01` | TYPE_DESC | UTF-8 | Data type signature of operands and result | `"f32→f32"`, `"f32×2→bool"`, `"void→f32:degrees"` |
| `0x02` | CAP_REQ | UTF-8 | Required hardware capability | `"sensor:imu:compass"`, `"actuator:rudder"` |
| `0x03` | PRE_COND | UTF-8 | Precondition for safe execution | `"SP >= 2"`, `"heading_sensor_valid"`, `"wind < 25kn"` |
| `0x04` | POST_COND | UTF-8 | Postcondition guaranteed after execution | `"TOS ∈ [-180, 360]"`, `"rudder_angle ∈ [-45, 45]"` |
| `0x05` | INTENT_ID | UTF-8 | Links this instruction to its parent intention | `"heading_hold.body.line_3"`, `"hvac.regulate.read_temp"` |
| `0x06` | TRUST_MIN | UTF-8 | Minimum trust score to execute | `"0.70"` (Level 3), `"0.50"` (Level 2), `"0.00"` (any) |
| `0x07` | AGENT_SOURCE | UTF-8 | Identity of the agent that generated this instruction | `"qwen2.5-coder-7b@vessel-7"`, `"claude-3.5-sonnet@cloud"` |
| `0x08` | TIMESTAMP | UTF-8 | ISO 8601 timestamp of generation | `"2025-07-12T14:30:00Z"` |
| `0x09` | CONFIDENCE | UTF-8 | Agent's confidence in this instruction's correctness | `"0.95"`, `"0.80"`, `"uncertain"` |
| `0x0A` | NARRATIVE | UTF-8 | Human-readable explanation (for audit/explanation) | `"Read compass heading and push onto stack"` |
| `0x0B` | PROVENANCE | UTF-8 | Chain of generation: which agent generated, validated, modified | `"gen:qwen;val:claude;mod:gpt"` |
| `0x0C` | SAFETY_FLAG | UTF-8 | Safety classification for this instruction | `"CRITICAL"`, `"NORMAL"`, `"DEGRADED"`, `"TRUST_GATE"` |
| `0x0D` | DOMAIN_TAG | UTF-8 | Application domain for cross-domain portability | `"marine:autopilot"`, `"factory:conveyor"`, `"home:hvac"` |

### 3.4 Stripping Protocol: AAB → Core

To produce the 8-byte core bytecode from AAB:

1. Read the first 8 bytes of each AAB instruction — this is the core instruction.
2. Skip all bytes after the first 8 until either:
   - Tag `0x00` (end marker) is encountered, or
   - The next 8-byte-aligned position is reached.
3. Concatenate all core instructions in order.
4. Result: pure bytecode, ready for COBS framing and RS-422 transmission.

**Stripping is deterministic and takes O(n) time where n is the total AAB size.** It can be implemented in C as:

```c
void strip_aab(const uint8_t *aab, size_t aab_len, uint8_t *core, size_t *core_len) {
    size_t src = 0, dst = 0;
    while (src + 8 <= aab_len) {
        memcpy(core + dst, aab + src, 8);  // Copy core 8 bytes
        dst += 8;
        src += 8;  // Skip to metadata
        // Skip TLV fields
        while (src < aab_len && aab[src] != 0x00) {
            uint8_t tag = aab[src++];
            if (src + 2 > aab_len) break;
            uint16_t len = (aab[src] << 8) | aab[src + 1];
            src += 2 + len;
        }
        if (src < aab_len && aab[src] == 0x00) src++;  // Skip end marker
    }
    *core_len = dst;
}
```

## 4. The 29 New Opcodes — Complete Specification

### 4.1 Overview

The existing 32 opcodes (0x00–0x1F) execute on the ESP32 VM. The 29 new opcodes (0x20–0x56) are **agent-native extensions** that execute on the Jetson or in the agent communication layer. On the ESP32, they are treated as NOP (0x00) with zero cycle cost. Their value lies entirely in the AAB metadata they carry.

**NOP Behavior on Legacy ESP32:** Confirmed safe. If the ESP32 receives bytecode containing opcodes 0x20–0x56, it treats the first byte as opcode 0x00 (NOP) because the dispatch table only handles 0x00–0x1F. The remaining 7 bytes of the instruction are ignored. PC advances by 8. Execution continues normally. No crash, no halt, no undefined behavior.

### 4.2 Intent Opcodes (0x20–0x26)

These opcodes declare **why** the program exists. They carry intention metadata that enables any agent to understand the program's purpose without executing it.

##### 0x20 — DECLARE_INTENT

**Core encoding:** `0x20 0x00 <intent_id:2> <hash:4>`

- **Purpose:** Declares the program's top-level intention. Must be the first instruction in every AAB program.
- **Agent semantics:** This is the program's "name" and "purpose" in machine-readable form. `intent_id` is a locally unique identifier. `hash` is a CRC-32 of the intention string for integrity checking.
- **AAB metadata:** TYPE_DESC="void→void:intent_declaration", INTENT_ID=<self-referencing>, TRUST_MIN, DOMAIN_TAG, NARRATIVE, AGENT_SOURCE, TIMESTAMP, CONFIDENCE, PROVENANCE.
- **Preconditions:** None.
- **Postconditions:** Intention is registered in the agent's program context.
- **Trust impact:** Establishes the trust gate for the entire program.

##### 0x21 — ASSERT_GOAL

**Core encoding:** `0x21 0x00 <goal_id:2> <threshold:4>`

- **Purpose:** Declares an expected outcome that the execution body should achieve.
- **Agent semantics:** After execution, a validator agent checks whether the goal was met within the threshold.
- **AAB metadata:** POST_COND, VERIFICATION_METHOD="simulation|runtime|hybrid".

##### 0x22 — VERIFY_OUTCOME

**Core encoding:** `0x22 0x00 <metric_id:2> <tolerance:4>`

- **Purpose:** Runtime verification that a specific metric is within tolerance.
- **Agent semantics:** On the Jetson, this triggers a check against the runtime metric. On the ESP32, treated as NOP.

##### 0x23 — DECLARE_CONSTRAINT

**Core encoding:** `0x23 0x00 <constraint_id:2> <limit:4>`

- **Purpose:** Declares a hard constraint on program behavior (e.g., "rudder angle must not exceed 30°").
- **Agent semantics:** Validators check all execution paths for constraint violations.

##### 0x24 — INTENT_SCOPE_BEGIN

**Core encoding:** `0x24 0x00 <scope_id:2> <0x00000000>`

- **Purpose:** Opens a named scope within the intention block. Enables nested intentions.
- **AAB metadata:** SCOPE_TYPE="primary|fallback|monitoring|safety", TRUST_MIN, CAP_REQ.

##### 0x25 — INTENT_SCOPE_END

**Core encoding:** `0x25 0x00 <scope_id:2> <0x00000000>`

- **Purpose:** Closes a scope. `scope_id` must match a previously opened INTENT_SCOPE_BEGIN.

##### 0x26 — EXPLAIN_FAILURE

**Core encoding:** `0x26 0x00 <failure_code:2> <0x00000000>`

- **Purpose:** Provides a human-readable explanation for a specific failure mode.
- **AAB metadata:** NARRATIVE=<human-readable explanation>, FAILURE_NARRATIVE=<agent-readable structured description>.

### 4.3 Agent Communication Opcodes (0x30–0x34)

These opcodes enable inter-agent and inter-vessel communication. On the ESP32, they map to EMIT_EVENT syscalls. On the Jetson, they enable full A2A message passing.

##### 0x30 — TELL

**Core encoding:** `0x30 <flags:1> <channel:2> <event_type:4>`

- **Purpose:** Send information to another agent or vessel.
- **Flags:** bit 6 (BROADCAST) = send to all; bit 5 (URGENT) = high-priority queue.
- **AAB metadata:** RECIPIENT=<agent_id or vessel_id>, MESSAGE_CONTENT.
- **Trust impact:** Can carry trust score updates. Sending false information degrades sender's trust.

##### 0x31 — ASK

**Core encoding:** `0x31 <flags:1> <channel:2> <request_id:4>`

- **Purpose:** Request information from another agent.
- **Flags:** bit 6 (BLOCKING) = wait for response; bit 5 (CACHE_OK) = accept cached response.
- **AAB metadata:** QUESTION=<natural-language query>, TIMEOUT_MS, ON_TIMEOUT="use_default|fail|retry".

##### 0x32 — DELEGATE

**Core encoding:** `0x32 <flags:1> <target_agent:2> <intent_id:4>`

- **Purpose:** Delegate a sub-intention to another agent.
- **Flags:** bit 6 (BLOCKING) = wait for completion; bit 5 (AUTHORITY_FULL) = full control.
- **AAB metadata:** DELEGATION_SCOPE=<description>, AUTHORITY_LEVEL="full|limited|advisory".
- **Trust impact:** Delegating to a low-trust agent affects delegator's trust score.

##### 0x33 — REPORT_STATUS

**Core encoding:** `0x33 0x00 <status_code:2> <detail:4>`

- **Purpose:** Report execution status to supervisor agent.
- **AAB metadata:** STATUS_TYPE="OK|WARNING|ERROR|CRITICAL", AFFECTED_SUBSYSTEM.

##### 0x34 — REQUEST_OVERRIDE

**Core encoding:** `0x34 0x00 <authority_level:2> <reason:4>`

- **Purpose:** Request human or higher-agent intervention.
- **AAB metadata:** REASON_TEXT, SAFETY_IMPLICATION.
- **Trust impact:** Indicates autonomous capability insufficient. May reduce autonomy level.

### 4.4 Capability Negotiation Opcodes (0x40–0x44)

These opcodes declare hardware requirements. They are verified at deployment time by the Jetson's role configuration.

##### 0x40 — REQUIRE_CAPABILITY

**Core encoding:** `0x40 0x00 <cap_type:2> <cap_id:4>`

- **Purpose:** Declare a required hardware capability.
- **cap_type:** 0=sensor, 1=actuator, 2=compute, 3=communication, 4=storage.
- **Flags:** bit 5 (OPTIONAL) = graceful degradation if unavailable.
- **AAB metadata:** CAPABILITY_DESC, ALTERNATIVE=<fallback capability>.

##### 0x41 — CAPABILITY_RESPONSE

**Core encoding:** `0x41 <flags:1> <status:2> <alt_cap:4>`

- **Purpose:** Response from deployment system confirming or denying a capability request.
- **status:** 0=GRANTED, 1=DENIED, 2=GRANTED_WITH_LIMITATION, 3=DEFERRED.

##### 0x42 — DECLARE_SENSOR_NEED

**Core encoding:** `0x42 0x00 <sensor_idx:2> <sample_rate:4>`

- **Purpose:** Declare sensor requirement with minimum sample rate (Hz as float32 in operand2).

##### 0x43 — DECLARE_ACTUATOR_USE

**Core encoding:** `0x43 0x00 <actuator_idx:2> <max_rate:4>`

- **Purpose:** Declare actuator usage with maximum rate of change (units/sec as float32).

##### 0x44 — DECLARE_COMPUTE_NEED

**Core encoding:** `0x44 0x00 <compute_type:2> <budget_cycles:4>`

- **Purpose:** Declare computation budget requirement.

### 4.5 Safety-Augmented Opcodes (0x50–0x56)

These opcodes embed safety checks directly in the bytecode. They are the **agent-visible safety layer**.

##### 0x50 — TRUST_CHECK

**Core encoding:** `0x50 0x00 <subsystem:2> <min_trust:4>`

- **Purpose:** Verify trust level before proceeding. `min_trust` is a float32 in operand2.
- **Execution on Jetson:** Reads trust score for `subsystem`. If score < min_trust, jumps to failure handler.
- **Execution on ESP32:** Treated as NOP. Trust checking is performed by the Jetson before deployment.
- **Trust impact:** This opcode IS the trust gate. It directly affects which code paths execute based on trust level.
- **AAB metadata:** TRUST_MIN, SAFETY_FLAG="TRUST_GATE".

##### 0x51 — AUTONOMY_LEVEL_ASSERT

**Core encoding:** `0x51 0x00 <level:2> <subsystem:4>`

- **Purpose:** Assert that the current autonomy level meets the requirement.
- **level:** 0=MANUAL, 1=ASSISTED, 2=SEMI_AUTO, 3=CONDITIONAL, 4=HIGH, 5=FULL.

##### 0x52 — SAFE_BOUNDARY

**Core encoding:** `0x52 0x00 <boundary_id:2> <margin:4>`

- **Purpose:** Define and enforce a safety boundary with configurable margin.
- **AAB metadata:** BOUNDARY_TYPE="spatial|temporal|rate|composite", BREACH_ACTION="halt|degrade|notify".

##### 0x53 — RATE_LIMIT

**Core encoding:** `0x53 0x00 <variable_idx:2> <max_rate:4>`

- **Purpose:** Limit the rate of change of a variable. `max_rate` is float32 in operand2.
- **AAB metadata:** UNIT_PER_SEC, ON_EXCEED="clamp|halt|notify".

##### 0x54 — DEADBAND

**Core encoding:** `0x54 0x00 <variable_idx:2> <band_width:4>`

- **Purpose:** Apply a deadband filter. Values within ±band_width/2 of center are treated as zero.
- **AAB metadata:** CENTER_VALUE, OUTPUT_ON_BAND="zero|hold_previous|interpolate".

##### 0x55 — WATCHDOG_PET

**Core encoding:** `0x55 0x00 <watchdog_id:2> <timeout_ms:4>`

- **Purpose:** Reset a software watchdog timer. If not pet within timeout, triggers safety event.
- **AAB metadata:** WATCHDOG_TYPE="software|hardware", ON_EXPIRE="halt|degrade|notify".

##### 0x56 — SAFETY_EVENT_EMIT

**Core encoding:** `0x56 0x00 <event_severity:2> <event_code:4>`

- **Purpose:** Emit a safety-critical event to the supervisor.
- **event_severity:** 0=INFO, 1=WARNING, 2=ERROR, 3=CRITICAL, 4=EMERGENCY.
- **Trust impact:** CRITICAL and EMERGENCY events directly reduce trust score of the emitting subsystem.
- **AAB metadata:** EVENT_DESC, AFFECTED_ACTUATORS, RECOVERY_PROCEDURE.

### 4.6 Extended Flags for New Opcodes

| Bit | Name | Meaning |
|---|---|---|
| 7 | SYSCALL | Maps to VM syscall (existing, inherited from 0x00–0x1F) |
| 6 | BLOCKING | Wait for completion (ASK, DELEGATE) |
| 5 | OPTIONAL | Failure is non-fatal (REQUIRE_CAPABILITY) |
| 4 | VERIFIED | Pre-verified by validation agent |
| 3 | CRITICAL | Safety-critical operation |
| 2–0 | SUBTYPE | Opcode-specific subtype flags |

## 5. Program Structure — Agent-Native

### 5.1 The Intention Block

The fundamental unit of agent-native programs is the **intention block** — a self-contained structure that declares what it wants to achieve, what it needs, and how success and failure are defined.

**Binary layout of an intention block:**

```
┌──────────────────────────────────────────────────────────────┐
│ HEADER (required — must be first)                            │
│  [0x20] DECLARE_INTENT intent_id=0x0001 hash=<crc32>        │
│  AAB metadata: NARRATIVE, TRUST_MIN, DOMAIN_TAG, etc.       │
├──────────────────────────────────────────────────────────────┤
│ CAPABILITY SCOPE (0 or more)                                 │
│  [0x40] REQUIRE_CAPABILITY cap_type cap_id                  │
│  [0x42] DECLARE_SENSOR_NEED sensor_idx sample_rate          │
│  [0x43] DECLARE_ACTUATOR_USE actuator_idx max_rate          │
│  [0x44] DECLARE_COMPUTE_NEED compute_type budget            │
├──────────────────────────────────────────────────────────────┤
│ TRUST CONTEXT (0 or more)                                   │
│  [0x50] TRUST_CHECK subsystem min_trust                     │
│  [0x51] AUTONOMY_LEVEL_ASSERT level subsystem               │
├──────────────────────────────────────────────────────────────┤
│ CONSTRAINT SCOPE (0 or more)                                │
│  [0x23] DECLARE_CONSTRAINT constraint_id limit              │
│  [0x52] SAFE_BOUNDARY boundary_id margin                   │
│  [0x53] RATE_LIMIT variable_idx max_rate                   │
├──────────────────────────────────────────────────────────────┤
│ EXECUTION BODY (required — core opcodes 0x00–0x1F)          │
│  [0x1A] READ_PIN sensor_idx                                 │
│  [0x08] ADD_F                                               │
│  ...                                                        │
│  [0x1B] WRITE_PIN actuator_idx                              │
├──────────────────────────────────────────────────────────────┤
│ VERIFICATION SCOPE (0 or more)                               │
│  [0x21] ASSERT_GOAL goal_id threshold                       │
│  [0x22] VERIFY_OUTCOME metric_id tolerance                  │
├──────────────────────────────────────────────────────────────┤
│ FAILURE NARRATIVE (0 or more)                                │
│  [0x26] EXPLAIN_FAILURE failure_code                        │
│  AAB metadata: NARRATIVE=<explanation for humans>            │
├──────────────────────────────────────────────────────────────┤
│ TERMINATOR (required — must be last)                         │
│  [0x00] NOP flags=0x80 operand1=0x01 (HALT)                │
└──────────────────────────────────────────────────────────────┘
```

### 5.2 Nesting Rules

- Intention blocks may contain **nested scopes** via INTENT_SCOPE_BEGIN / INTENT_SCOPE_END.
- Maximum nesting depth: 8 levels.
- Each scope has its own trust context and capability scope.
- A TRUST_CHECK in an inner scope can raise the trust requirement above the outer scope's level.
- An inner scope's failure narrative takes precedence over the outer scope's.

### 5.3 Priority Arbitration

When multiple intention blocks target the same actuator register:

1. **Higher TRUST_MIN wins.** A CRITICAL-safety intention (trust 0.90) overrides a NORMAL intention (trust 0.50).
2. **Earlier DECLARE_CONSTRAINT wins.** If trust levels are equal, the first declared constraint is authoritative.
3. **CLAMP_F intersection.** If multiple intentions CLAMP_F the same actuator, the intersection of all clamp ranges applies (most restrictive wins).
4. **WRITE_PIN last-writer wins.** Within a single tick, the last WRITE_PIN to a given register takes effect.

### 5.4 Composition

Intention blocks compose by concatenation with dependency declarations:

```
; Intention A: "maintain heading"
DECLARE_INTENT ...
... execution body ...
HALT

; Intention B: "avoid obstacles" (independent)
DECLARE_INTENT ...
... execution body ...
HALT

; Intention C: "reduce speed in heavy weather" (depends on A and B)
DECLARE_INTENT ...
DEPENDS_ON "heading_hold", "obstacle_avoid"
... execution body ...
HALT
```

The Jetson scheduler resolves dependencies and determines execution order within each tick.

### 6.1 Example 1: Marine Heading Hold with Wind Compensation

**Intention:** Maintain heading 270° using PID control. Disengage if wind exceeds 20 knots. Clamp rudder to ±30°.

```
; === HEADER ===
DECLARE_INTENT intent_id=0x0001 hash=0xA3F2B1C4
  AAB: NARRATIVE="Maintain heading 270° when wind < 20kn", TRUST_MIN="0.70",
       DOMAIN_TAG="marine:autopilot", AGENT_SOURCE="qwen2.5-coder-7b@vessel-7"

; === CAPABILITIES ===
REQUIRE_CAPABILITY cap_type=0 (sensor) cap_id=0x0000 (compass)
REQUIRE_CAPABILITY cap_type=0 (sensor) cap_id=0x0003 (wind_speed)
REQUIRE_CAPABILITY cap_type=1 (actuator) cap_id=0x0001 (rudder)
DECLARE_ACTUATOR_USE actuator_idx=1 max_rate=5.0 (deg/s)

; === TRUST CONTEXT ===
TRUST_CHECK subsystem=1 (steering) min_trust=0.70

; === EXECUTION BODY (core opcodes) ===
READ_PIN 0x0003        ; wind_speed → stack          [1A 00 03 00 00 00 00 00]
PUSH_F32 20.0          ; threshold → stack          [03 00 00 00 00 00 A0 41]
LT_F                   ; wind < 20? → 0 or 1        [12 00 00 00 00 00 00 00]
JUMP_IF_FALSE safe     ; if wind >= 20, skip PID     [1E 00 00 00 60 00 00 00]

READ_PIN 0x0000        ; compass heading → stack     [1A 00 00 00 00 00 00 00]
PUSH_F32 270.0         ; setpoint → stack           [03 00 00 00 00 00 87 43]
; PID_COMPUTE pid_idx=0                     [00 80 02 00 00 00 00 00]
CLAMP_F lo=-30.0 hi=30.0                      [10 00 00 00 00 00 F0 C1] (encoded shared upper half)
WRITE_PIN 0x0001      ; → rudder actuator          [1B 00 01 00 00 00 00 00]
JUMP done                                         [1D 00 00 00 90 00 00 00]

safe:
PUSH_F32 0.0          ; center rudder              [03 00 00 00 00 00 00 00]
WRITE_PIN 0x0001      ; → rudder (centered)        [1B 00 01 00 00 00 00 00]
EMIT_EVENT event_id=0x01 event_data=0x0020         [00 80 04 01 00 20 00 00]

done:
HALT                                             [00 80 01 00 00 00 00 00]

; === FAILURE NARRATIVE ===
EXPLAIN_FAILURE failure_code=0x01 (wind_exceeded)
  AAB: NARRATIVE="Wind exceeded 20 knots. Rudder centered. Manual control required."
```

## 7. Validation Rules for Agent-Generated Bytecode

### 7.1 What Must Be True for Bytecode to Be Valid

An AAB program is **valid** if and only if all of the following hold:

1. **Structural validity:** The program begins with DECLARE_INTENT and ends with HALT. Every instruction is exactly 8 bytes (core) with valid TLV metadata.
2. **Opcode validity:** All core opcodes are in range [0x00, 0x1F] (existing) or [0x20, 0x56] (new agent-native). No undefined opcodes.
3. **Stack safety:** On every execution path, the stack pointer never goes below 0 or above 256.
4. **Type consistency:** Every arithmetic opcode (ADD_F–CLAMP_F) operates on float32 values. Every logic opcode (AND_B–NOT_B) operates on integer values.
5. **Jump validity:** Every jump target is within the bytecode buffer, 8-byte aligned, and not inside a metadata trailer.
6. **Actuator bounds:** Every WRITE_PIN to an actuator register (operand1 < 64) is preceded by a CLAMP_F that constrains the value to the actuator's safe range.
7. **Trust consistency:** Every TRUST_CHECK references a valid subsystem index (0–15) and a plausible minimum trust score (0.0–1.0).
8. **Cycle budget:** Total cycle count of the execution body (excluding agent-native opcodes, which are NOP on ESP32) does not exceed the per-tick budget.
9. **Metadata accuracy:** Every AAB metadata tag accurately describes its associated core instruction's behavior.
10. **Integrity:** The hash in DECLARE_INTENT matches the CRC-32 of the intention narrative string.

### 7.2 How One Agent Validates Another Agent's Bytecode

The 6-step validation pipeline:

**Step 1: Structural Pass**
- Verify DECLARE_INTENT is the first instruction, HALT is the last.
- Verify all core instructions are 8 bytes with valid opcodes.
- Verify TLV metadata is well-formed (valid tags, lengths don't exceed buffer).
- Verify scope nesting is balanced (every INTENT_SCOPE_BEGIN has matching INTENT_SCOPE_END).

**Step 2: Intent Extraction**
- Read INTENT_ID and NARRATIVE from DECLARE_INTENT metadata.
- Read all CAP_REQ tags to build a capability dependency list.
- Read TRUST_MIN to determine the trust gate threshold.

**Step 3: Stack Simulation**
- Walk through every core instruction (0x00–0x1F), tracking SP.
- Build a stack type map: for each stack slot, track whether it holds float32, integer, or unknown.
- Verify no execution path causes SP < 0 or SP > 256.
- Verify type consistency at every arithmetic and logic opcode.

**Step 4: Safety Verification**
- Verify every WRITE_PIN to an actuator is preceded by CLAMP_F.
- Verify CLAMP_F bounds match the actuator's configured safe range.
- Verify all TRUST_CHECK opcodes reference valid subsystems.
- Verify SAFETY_EVENT_EMIT opcodes are only generated in appropriate contexts.

**Step 5: Intent Alignment**
- Reason about whether the instruction sequence, taken as a whole, achieves the declared intention.
- Check that ASSERT_GOAL and VERIFY_OUTCOME postconditions are achievable given the execution body.
- Verify EXPLAIN_FAILURE narratives cover all reachable failure modes.

**Step 6: Trust & Provenance Check**
- Verify AGENT_SOURCE identifies a known agent.
- Verify PROVENANCE chain shows generation by one model and validation by a different model.
- Verify TIMESTAMP is recent (within acceptable staleness window).
- Verify CONFIDENCE is above minimum threshold (typically 0.80).

### 7.3 Validation Outcome Codes

| Code | Meaning | Action |
|---|---|---|
| VALID | All 6 steps pass | Deploy to ESP32 |
| VALID_WITH_WARNINGS | Steps 1–5 pass, step 6 has minor issues | Deploy with logging |
| INVALID_STRUCTURE | Step 1 fails | Reject. Cannot parse. |
| INVALID_STACK | Step 3 fails | Reject. Stack safety violated. |
| INVALID_SAFETY | Step 4 fails | Reject. Safety-critical violation. |
| INVALID_INTENT | Step 5 fails | Reject. Code doesn't match intention. |
| INVALID_TRUST | Step 6 fails | Reject. Provenance or trust insufficient. |

### 8.2 Safety-Critical Opcodes

The following opcodes are classified as **safety-critical** — errors in their use can cause physical harm:

1. **0x1B WRITE_PIN** — The only opcode that directly causes physical actuator movement. ALWAYS precede with CLAMP_F.
2. **0x10 CLAMP_F** — The primary safety boundary for actuator outputs. Incorrect bounds are a safety violation.
3. **0x50 TRUST_CHECK** — The trust gate. Incorrect threshold can allow unsafe operations or block safe ones.
4. **0x56 SAFETY_EVENT_EMIT** — Triggers safety responses. False positives degrade system availability; false negatives allow unsafe states.
5. **PID_COMPUTE** — Drives closed-loop control. Poorly configured gains cause oscillation or instability.

### 8.3 Safety-Critical Patterns

**Mandatory pattern — Actuator write with clamping:**
```
; ALWAYS emit this sequence before any actuator write:
... compute output value ...
CLAMP_F <safe_min> <safe_max>
WRITE_PIN <actuator_idx>
```

**Mandatory pattern — Trust-gated autonomous operation:**
```
; ALWAYS check trust before autonomous actuator writes:
TRUST_CHECK <subsystem> <min_trust>
... if trust OK, proceed with control ...
; if trust too low, fall through to safe state
```

**Forbidden pattern — Unclamped actuator write:**
```
; NEVER emit this:
... compute value ...
WRITE_PIN <actuator_idx>     ; NO CLAMP_F — REJECT
```

**Forbidden pattern — Infinite loop:**
```
; NEVER emit this:
loop:
... some computation ...
JUMP loop                    ; UNBOUNDED LOOP — REJECT
; Use state machines instead: one iteration per tick
```
