## 1. Introduction and Philosophy

### 1.1 "Code Is a Hidden Artifact"

The NEXUS platform operates under a founding principle that most engineers find unsettling at first: **no human ever writes, reads, or debugs code in production**. Code is a hidden artifact — generated by AI, validated by a separate AI instance, tested in simulation, and deployed to hardware automatically. The human operator's role is limited to wiring hardware, describing intent in natural language, demonstrating behaviors, and approving or rejecting AI-generated proposals.

As a developer on the NEXUS team, your relationship to code is fundamentally different from traditional embedded systems work. You do not write the control logic that runs on ESP32 nodes. You build the *platform* that makes it possible for AI to generate, validate, and deploy control logic safely. You maintain the bytecode VM, the safety system, the serial protocol, the Jetson cognitive layer, and the build infrastructure. The control logic itself — the reflexes — is generated by large language models and deployed through a rigorous pipeline.

This means your work has an outsized impact. A bug in the VM's cycle counter can affect every reflex on every node in the fleet. A mistake in the safety guard API can compromise the four-tier safety architecture. Your code is the foundation upon which the entire system operates.

### 1.2 The v3.1 Codebase: Two Years of Evolution

The NEXUS codebase has grown significantly since v1.0 shipped in early 2026. What started as a 320KB universal firmware binary and ~15,000 lines of Python for the Jetson layer has become a mature platform:

- **Firmware:** 450KB universal binary (up from 320KB), 9 I2C drivers, 32-opcode VM with flight recorder, OTA canary deployment, reflex versioning
- **Jetson:** 6 microservices with hot-reload, EMQX MQTT broker, DeepSeek-Coder-7B LLM, edge training capability, reflex marketplace
- **Specifications:** 21 core spec files totaling ~19,200 lines, plus 7 engineering addenda covering pitfalls, performance budgets, hardware bring-up, safety validation, integration testing, and code review
- **Testing:** Unit tests (Unity for C, pytest for Python), integration tests (22 tests), HIL test bench that runs on every PR, formal model verification for all deployed reflexes

### 1.4 Build System

**Firmware:** ESP-IDF v5.2+ with CMake. The partition table is custom (`partitions.csv`). Build with `idf.py -p /dev/ttyUSB0 -b 921600 flash monitor`. The universal binary compiles all drivers, all opcodes, and all protocol handlers into one `.bin`. Conditional compilation uses `#ifdef SAFETY_MODE` and `#ifdef DEVELOPMENT_MODE` to select safety heartbeat intervals (100ms production vs 1000ms development) and disable boot counter lockout during development.

**Jetson:** Python 3.11 with Poetry for dependency management. The `pyproject.toml` declares dependencies on `grpcio`, `paho-mqtt`, `llama-cpp-python`, `numpy`, `scikit-learn`, and `pydantic`. Run with `poetry install && poetry run python -m services.main`. LLM inference uses `llama.cpp` via the Python bindings for local model serving.

**Key configuration flags:**

| Flag | Purpose | Value |
|------|---------|-------|
| `SAFETY_MODE` | Production safety intervals | Defined in release builds |
| `DEVELOPMENT_MODE` | Relaxed boot counter, debug logging | Defined during dev |
| `ENABLE_FLIGHT_RECORDER` | Circular buffer of VM ticks | Enabled since v2.5 |
| `ENABLE_DMA_UART` | DMA-based UART transfers | Enabled since v3.0 |
| `ENABLE_WIFI_BACKUP` | WiFi 6 backup link | Enabled since v3.1 |

### 1.5 The Lesson from v1.0

> "We shipped without the CLAMP_F shared-upper-half encoding fix and 3 customers hit it in the first week."

The CLAMP_F opcode encodes two float bounds (lower, upper) into a single 4-byte `operand2` field by requiring both floats to share the same upper 16 bits (same sign and exponent range). In v1.0, the validator allowed CLAMP_F with mismatched upper halves, producing silently wrong float values — not NaN, but plausible-looking garbage that passed all range checks. Three customers reported erratic actuator behavior that we traced to CLAMP_F producing values outside the intended range.

The v1.1 fix was simpler: we deprecated the shared-upper-half encoding entirely and now the compiler emits `MAX_F lower_bound` + `MIN_F upper_bound` as two separate instructions. The CLAMP_F opcode still exists in the ISA for backward compatibility, but the validator rejects any standalone CLAMP_F — it must be expanded to MAX_F+MIN_F at compile time. **If you are adding a new opcode, ask yourself: could the encoding have a subtle edge case that produces wrong-but-plausible output? If yes, simplify the design.**

**3. `hil_test.yml`** — Hardware-in-the-Loop test bench (the v3.0 addition):
This is the most important CI pipeline. It runs on a dedicated test machine with physical ESP32-S3 boards connected. The test bench:
- Flashes the compiled firmware to 3 test nodes
- Connects them to the Jetson serial bridge
- Runs a 45-minute automated test sequence covering: boot timing, heartbeat monitoring, COBS framing, VM execution, observation buffer, OTA update, kill switch response
- Collects timing data with the logic analyzer
- Fails the PR if any safety-critical timing is violated (>1ms kill switch response, >100ms heartbeat loss detection)

### 2.6 The v3.0 Addition: Hardware-in-the-Loop

The HIL test bench was the single most impactful addition to our development workflow. Before v3.0, we shipped firmware that passed all unit tests but failed in the field due to timing interactions that only appear with real hardware. The HIL bench has caught 12 bugs that pure software testing missed, including:

- A UART DMA alignment issue that corrupted 1 in 10,000 frames
- A flight recorder buffer race condition during observation dump
- An OTA rollback timing window where the system could boot a partially-written image

The HIL bench runs on every PR. It takes 45 minutes to complete. This is acceptable because the alternative — shipping a firmware bug to production — is far more expensive.

## 3. Firmware Architecture (ESP32-S3)

### 3.1 The Universal Binary

NEXUS ships **one `.bin` file** for all ESP32-S3 nodes. The role (autopilot controller, engine monitor, bilge pump controller, lighting controller, etc.) is assigned at boot via a JSON configuration message from the Jetson. This means:

- **Hot-swap field replacement:** If an ESP32 fails, any fresh unit can be plugged in and will automatically receive its role. An operator does not need a laptop — the system provisions itself.
- **Simplified testing and signing:** Only one binary needs to be tested, validated, and cryptographically signed. Per-role testing is unnecessary.
- **Inventory simplicity:** Stock one firmware image. No risk of flashing the wrong binary to the wrong node.

The trade-off is increased flash usage: 450KB (v3.1) vs ~150KB for minimal per-role builds. This is 2.8% of the 16MB flash — trivially acceptable.

### 3.2 Directory Structure

The firmware is organized into five component directories under `components/`:

| Component | Responsibility | Key Files | Lines of C |
|-----------|---------------|-----------|-----------|
| `nexus_core/` | Configuration parsing, NVS storage, logging | `nexus_config.c`, `nvs_store.c` | ~2,800 |
| `safety/` | Four-tier safety system | `estop_isr.c`, `safety_guard.c`, `safety_supervisor.c` | ~3,200 |
| `vm/` | Bytecode VM, flight recorder, reflex versioning | `vm_core.c`, `vm_dispatch.c`, `vm_validator.c` | ~4,500 |
| `protocol/` | COBS, CRC-16, message dispatch, OTA | `cobs.c`, `message_dispatch.c`, `ota_manager.c` | ~3,800 |
| `io_drivers/` | I2C sensor/actuator drivers | `bme280_driver.c`, `ina219_driver.c`, ... | ~5,200 |

### 3.3 Task Architecture (FreeRTOS)

The firmware runs seven FreeRTOS tasks with carefully chosen priorities and stack sizes:

| Task | Priority | Stack | Core | Period | Purpose |
|------|----------|-------|------|--------|---------|
| `safety_supervisor` | 23 (highest app) | 2048B | 0 | 10ms | Heartbeat, watchdog, state machine |
| `watchdog_feed` | 22 | 1024B | 0 | 200ms | MAX6818 kick pattern |
| `protocol_rx` | 20 | 4096B | 0 | Event-driven | Serial receive + COBS decode |
| `protocol_tx` | 19 | 4096B | 0 | Event-driven | Serial transmit + COBS encode |
| `vm_tick` | 18 | 8192B | 0 | 1ms (configurable) | VM execution loop |
| `io_poll` | 17 | 4096B | 1 | Variable | I2C sensor reads, ADC sampling |
| `telemetry` | 16 | 4096B | 1 | 100ms | Telemetry message construction + send |

**Core affinity:** Safety-critical tasks (supervisor, watchdog, protocol RX, VM tick) are pinned to Core 0. I/O polling and telemetry run on Core 1. This prevents I/O bus contention from delaying the VM tick.

**Stack sizing:** The VM tick task has the largest stack (8192B) because the VM's fetch-decode-execute loop is not reentrant and uses local variables for instruction decoding. The protocol tasks have 4096B stacks to accommodate COBS encode/decode buffers and JSON message construction.

### 3.4 The Boot Sequence in Code

The boot sequence is safety-critical and must complete within 500ms. Here is the exact code flow:

```c
// firmware/main/app_main.c

void app_main(void) {
    // T+0: All GPIO outputs forced LOW (safe state)
    gpio_safe_state_all();

    // T+5: Hardware watchdog feeding begins
    watchdog_feed_init();
    xTaskCreatePinnedToCore(watchdog_feed_task, "watchdog", 1024, NULL, 22, NULL, 0);

    // T+10: UART initialization at 115200 baud
    uart_init(UART_NUM_1, 115200);
    cobs_init();

    // T+20: Send DEVICE_IDENTITY (0x01)
    device_identity_t identity = {
        .mac_addr = esp_read_mac(),
        .chip_type = CHIP_ESP32_S3,
        .fw_version = NEXUS_FW_VERSION,
        .capabilities = CAP_I2C | CAP_PWM | CAP_ADC | CAP_UART
    };
    protocol_send(MSG_DEVICE_IDENTITY, &identity, sizeof(identity));

    // T+30: I2C bus scan → AUTO_DETECT_RESULT (0x1B)
    io_driver_scan_and_report();

    // T+50: Self-test → SELFTEST_RESULT (0x04)
    selftest_result_t result = run_selftest();
    protocol_send(MSG_SELFTEST_RESULT, &result, sizeof(result));

    // T+50-300: Wait for ROLE_ASSIGN from Jetson
    role_config_t role;
    esp_err_t err = role_wait_for_assign(&role, pdMS_TO_TICKS(300));

    if (err == ESP_OK) {
        // T+300: Apply role configuration
        io_driver_apply_config(&role);
        role_cache_to_nvs(&role);  // Cache for offline boot

        // Load cached reflexes from LittleFS
        reflex_load_cached();

        // T+500: Enter OPERATIONAL state
        operational_mode = MODE_OPERATIONAL;
        xTaskCreatePinnedToCore(vm_tick_task, "vm_tick", 8192, NULL, 18, NULL, 0);
        xTaskCreatePinnedToCore(telemetry_task, "telemetry", 4096, NULL, 16, NULL, 1);
    } else {
        // No Jetson → IDLE mode, retry every 10s
        operational_mode = MODE_IDLE;
        xTaskCreate(idle_retry_task, "idle", 2048, NULL, 15, NULL);
    }
}
```

**Timing milestones** are verified by the HIL test bench. If any milestone exceeds its deadline, the test fails. In production, the boot counter in NVS tracks how many times the system has booted; if it exceeds 5 boots in 10 minutes, the system enters FAULT mode (requires manual reset).

### 3.5 How to Add a New I2C Driver

Adding a new I2C sensor driver requires implementing the `io_driver_interface.h` vtable. Here is a step-by-step example for a hypothetical BMP388 barometric pressure sensor:

**Step 1: Create the driver file** at `components/io_drivers/bmp388_driver.c`:

```c
#include "io_driver_interface.h"
#include "driver/i2c.h"

#define BMP388_ADDR         0x76
#define BMP388_CHIP_ID      0x50
#define BMP388_REG_CHIPID   0x00
#define BMP388_REG_DATA     0x04
#define BMP388_REG_CTRL     0x1B

typedef struct {
    uint8_t i2c_bus;
    uint8_t addr;
    float pressure_hpa;
    float temperature_c;
    bool data_valid;
} bmp388_ctx_t;

// 1. init() — Initialize the sensor
static int bmp388_init(void *ctx, const io_pin_config_t *config) {
    bmp388_ctx_t *c = (bmp388_ctx_t *)ctx;
    c->i2c_bus = config->bus;
    c->addr = config->addr;

    uint8_t chip_id;
    i2c_master_read_from_device(c->i2c_bus, c->addr, BMP388_REG_CHIPID, &chip_id, 1, 100);
    if (chip_id != BMP388_CHIP_ID) return IO_ERR_DEVICE_NOT_FOUND;

    // Configure for normal mode, 50Hz
    uint8_t ctrl = 0x33;
    i2c_master_write_to_device(c->i2c_bus, c->addr, BMP388_REG_CTRL, &ctrl, 1, 100);
    c->data_valid = false;
    return IO_OK;
}

// 2. read() — Read sensor data
static int bmp388_read(void *ctx, io_reading_t *reading) {
    bmp388_ctx_t *c = (bmp388_ctx_t *)ctx;
    uint8_t buf[6];
    i2c_master_read_from_device(c->i2c_bus, c->addr, BMP388_REG_DATA, buf, 6, 50);
    // Parse 24-bit pressure and temperature from buf
    // ... (standard BMP388 conversion formulas)
    reading->values[0] = c->pressure_hpa;
    reading->values[1] = c->temperature_c;
    reading->num_values = 2;
    c->data_valid = true;
    return IO_OK;
}

// 3-7: write(), configure(), selftest(), deinit(), get_info()
// ... (implement per io_driver_interface.h spec)

// Vtable registration
const io_driver_vtable_t bmp388_vtable = {
    .init = bmp388_init,
    .read = bmp388_read,
    .write = bmp388_write,
    .configure = bmp388_configure,
    .selftest = bmp388_selftest,
    .deinit = bmp388_deinit,
    .get_info = bmp388_get_info,
    .name = "bmp388",
    .type = IO_DRIVER_TYPE_I2C_SENSOR,
    .num_readings = 2,
    .reading_names = {"pressure_hpa", "temperature_c"},
};
```

**Step 2: Register the driver** in `io_driver_registry.json`:
```json
{
  "bmp388": {
    "vtable": "bmp388_vtable",
    "i2c_addr": "0x76",
    "auto_detect_addr": "0x76",
    "auto_detect_chip_id_reg": "0x00",
    "auto_detect_chip_id_val": "0x50",
    "readings": ["pressure_hpa", "temperature_c"],
    "selftest": {
      "procedure": "read_chip_id",
      "expected": "0x50"
    },
    "known_limitations": [
      "Maximum I2C bus speed: 400kHz (not 1MHz)",
      "Data ready flag must be polled; no interrupt pin supported"
    ]
  }
}
```

**Step 3: Add self-test procedure** that the boot self-test checks. The self-test reads the chip ID register and verifies it returns `0x50`.

**Step 4: Add to AUTO_DETECT_RESULT** scanning. The I2C bus scan at T+30 probes address `0x76` and checks the chip ID register. If it matches, the driver is auto-registered.

**Step 5: Test on real hardware** before committing. The HIL test bench does not test every I2C driver — you must verify with a physical sensor on a dev board. Check: init succeeds, readings update, self-test passes, reading values are physically plausible (e.g., pressure near 1013 hPa).

### 3.6 How to Add a New VM Opcode

Adding a new opcode to the VM requires changes in **six files**:

1. **`vm_opcodes.h`** — Add the opcode enum value:
```c
#define OP_CUSTOM      0x10  // Replaces CLAMP_F or uses reserved slot
```

2. **`vm_dispatch.c`** — Add the switch case in the dispatch table:
```c
case OP_CUSTOM:
    if (vm->sp < 1) { vm_halt_error(vm, ERR_STACK_UNDERFLOW); break; }
    // Your opcode logic here
    vm->sp--;  // Pop operand if needed
    cycles_used += 3;  // Document cycle count
    break;
```

3. **`vm_validator.c`** — Add compile-time validation rules:
```c
case OP_CUSTOM:
    // Verify stack depth sufficient, operands valid
    if (simulated_sp < 1) return ERR_VALIDATE_STACK_UNDERFLOW;
    break;
```

4. **`compiler.py`** (Jetson) — Add the compilation rule:
```python
def compile_custom(self, node):
    """Emit bytecode for CUSTOM operation."""
    self.emit_opcode(OP_CUSTOM)
```

5. **`test_vm_opcodes.c`** — Add unit tests:
```c
TEST_CASE("OP_CUSTOM: basic operation", "[vm]") {
    vm_t vm;
    vm_init(&vm);
    // Set up stack with operands
    // Execute single instruction
    // Verify result
}
```

6. **`synthesis.md`** (or reflex schema) — Update the specification to document the new opcode's semantics, stack effect, cycle count, and safety constraints.

**Why we added CLAMP_F but would never do it again:** CLAMP_F required encoding two float bounds in a single 4-byte field. The "shared upper half" trick was clever but produced silent wrong values when the constraint was violated. The v1.1 fix decomposed it to MAX_F + MIN_F. **If you are considering adding an opcode with a complex encoding, decompose it into simpler existing opcodes instead. The ISA has 32 slots and we are not adding more.**

### 3.7 Memory Management Rules

NEXUS firmware follows strict memory placement rules:

| Memory Type | Contents | Size | Constraints |
|-------------|----------|------|-------------|
| **SRAM (internal)** | Safety ISR code, safety data, VM stack, protocol buffers, FreeRTOS heaps | ~512KB total | Safety code MUST be in IRAM (marked `IRAM_ATTR`). Never allocate safety-critical data in PSRAM. |
| **PSRAM (external)** | Observation buffer (8MB ring), flight recorder (circular buffer of last 1000 VM ticks), reflex bytecode cache | ~8MB total | No power-loss protection. Volatile. Never store safety-critical state here. |
| **Flash (internal)** | Firmware code, LittleFS partition for reflex JSON, NVS for persistent config | 16MB total | Flash is inaccessible during SPI operations (~100ms). ISR code must be in IRAM, not flash. |

**Key rules:**
- All ISR functions and any function called from an ISR must be marked `IRAM_ATTR` and must only call other `IRAM_ATTR` functions.
- The safety supervisor task's stack and data structures are allocated from internal SRAM.
- The observation buffer is the ONLY thing allowed in PSRAM. If you need to add another PSRAM consumer, file an architecture decision record (ADR).
- The VM's 3KB footprint (stack + variables + PID state + snapshot buffer) lives entirely in SRAM.

### 3.8 Debugging: ESP-IDF Trace, JTAG, and the Flight Recorder

Three debugging tools are available:

**1. ESP-IDF trace:** `esp_log_level_set("*", ESP_LOG_DEBUG)` in development mode. **Never enable debug logging in production** — it touches flash for format strings and can cause ISR failures.

**2. JTAG debugging:** Attach via ESP-Prog or FT2232H. Set breakpoints in `vm_dispatch.c` to step through VM execution. The JTAG connection pauses the ESP32 — this will cause heartbeat loss and trigger safe-state on the Jetson. Use `idf.py openocd --no-reset` to attach without resetting.

**3. The v2.5 Flight Recorder:** A circular buffer that records the last 1000 VM ticks (each tick stores: PC, opcode, stack depth, cycle count, actuator outputs). When a reflex faults (cycle budget exceeded, stack overflow, invalid opcode), the flight recorder dumps the last 1000 ticks to the serial link. This has been invaluable for debugging — in v1.0, we lost 3 critical debugging sessions because the VM fault corrupted the stack before we could read it.

Access the flight recorder via `flight_recorder_dump()` — it sends a formatted dump over the debug UART (not the Jetson serial link, to avoid interfering with the protocol).

### 4.3 MQTT Topic Hierarchy

13 MQTT topics are defined in `jetson/mqtt_topics.json`. Key design decisions:

### 4.4 Module Lifecycle: NexusModule ABC

All Jetson modules subclass `NexusModule` (defined in `jetson/modules/module_interface.py`). The lifecycle is:

1. **`init(config)`** — Allocate resources, load models, open connections. Called once before dependencies start.
2. **`start()`** — Begin async operation. Schedule background tasks. Return quickly.
3. **`stop()`** — Cancel all tasks, release GPU memory, close connections, flush buffers.
4. **`health_check()`** — Return a `HealthReport`. Called every 30 seconds by the `ModuleHost`.
5. **`on_telemetry(event)`** — Handle incoming sensor data. Must return within 5 seconds.
6. **`on_safety_event(event)`** — Handle safety events. Must return within 1 second.

**Hot-reload:** The `ModuleHost` supports hot-reloading modules at runtime without restarting the process. The `GRACEFUL` strategy calls `stop()` on the old instance, then `init()` + `start()` on the new instance, preserving state via `capture_state()` / `on_after_reload()`. The `IMMEDIATE` strategy swaps in-place with no stop/start cycle.

### 4.7 Resource Budget Enforcement: The ThrottleMonitor

The `ThrottleMonitor` is a reference implementation of a Jetson module that enforces resource budgets. It monitors the CPU temperature of each ESP32 node via telemetry events. When a node exceeds 85°C (configurable), it emits a `CRITICAL` safety event. The `ModuleHost` can be configured to shed non-essential services in response:

- At 70°C: Log warning
- At 80°C: Disable LLM inference (saves ~4GB VRAM), keep reflexes running
- At 85°C: Enter degraded mode — disable learning pipeline, disable chat interface
- At 90°C: Trigger safe-state — all actuators to safe position

The shedding order is defined in `safety_policy.json` under the `degradation_profile` key. Each module declares its `ResourceBudget` upfront, and the `ModuleHost` rejects module startup if the total budget exceeds available resources.

## 5. Serial Protocol Implementation

### 5.1 COBS Encode/Decode: Reference Implementation

COBS (Consistent Overhead Byte Stuffing) is the framing layer of the NEXUS wire protocol. The reference C implementation:

```c
// firmware/components/protocol/cobs.c

size_t cobs_encode(const uint8_t *src, size_t src_len, uint8_t *dst) {
    size_t read_idx = 0;
    size_t write_idx = 1;       // Leave room for first count byte
    size_t code_idx = 0;         // Where current count byte goes
    uint8_t code = 1;            // Count of non-zero bytes

    while (read_idx < src_len) {
        if (src[read_idx] == 0x00) {
            dst[code_idx] = code;
            code_idx = write_idx++;
            code = 1;
            read_idx++;
        } else {
            dst[write_idx++] = src[read_idx++];
            code++;
            if (code == 0xFF) {   // Max 254 non-zero bytes per block
                dst[code_idx] = code;
                code_idx = write_idx++;
                code = 1;
            }
        }
    }
    dst[code_idx] = code;
    dst[write_idx++] = 0x00;    // Terminator
    return write_idx;
}

size_t cobs_decode(const uint8_t *src, size_t src_len, uint8_t *dst) {
    size_t read_idx = 0;
    size_t write_idx = 0;
    uint8_t code;
    uint8_t i;

    while (read_idx < src_len) {
        code = src[read_idx++];
        for (i = 1; i < code; i++) {
            if (read_idx >= src_len) return 0;  // Error
            dst[write_idx++] = src[read_idx++];
        }
        if (code < 0xFF && read_idx < src_len) {
            dst[write_idx++] = 0x00;  // Decode implicit zero
        }
    }
    // Strip the trailing sentinel: output length = write_idx - 1
    return write_idx > 0 ? write_idx - 1 : 0;
}
```

**Common bugs:**
1. Forgetting to strip the trailing sentinel after decode (output is 1 byte too long)
2. Off-by-one in the 254-byte block boundary (0xFF means 254 bytes follow, not 255)
3. Not handling the edge case where the source is exactly 254 non-zero bytes followed by a zero

### 5.2 Message Dispatch

The message dispatch table maps message types to handler functions:

```c
typedef struct {
    uint8_t msg_type;
    handler_fn handler;
    bool requires_ack;
    uint8_t priority;
} dispatch_entry_t;

static const dispatch_entry_t dispatch_table[] = {
    { MSG_HEARTBEAT,       handle_heartbeat,       false, PRI_LOW },
    { MSG_PING,            handle_ping,            true,  PRI_LOW },
    { MSG_ROLE_ASSIGN,     handle_role_assign,     true,  PRI_CRITICAL },
    { MSG_REFLEX_DEPLOY,   handle_reflex_deploy,   true,  PRI_NORMAL },
    { MSG_COMMAND,         handle_command,         true,  PRI_NORMAL },
    { MSG_OBS_RECORD_START,handle_obs_start,       true,  PRI_NORMAL },
    { MSG_FIRMWARE_CHUNK,  handle_firmware_chunk,  true,  PRI_BULK },
    // ... 28 entries total
};
```

Handlers are registered at boot and looked up via binary search on message type. Each handler receives a parsed header and a pointer to the payload buffer.

### 5.3 The Baud Negotiation State Machine

Baud negotiation upgrades from 115200 to 921600 baud at boot. The v1.1 fix added a **50ms settling delay** after both sides switch baud rates:

```
State 0: SEND_REQUEST (Jetson sends BAUD_UPGRADE at 115200)
State 1: WAIT_ACK (Node ACKs at 115200)
State 2: SWITCH (Both sides switch to 921600)
State 3: SETTLE (50ms delay — v1.1 addition)
State 4: VERIFY (Jetson sends PING at 921600)
State 5: CONFIRMED (Node PONGs at 921600 → done)
State 6: TIMEOUT (No PONG → fall back to 115200)
```

**The 50ms settling delay** is implemented as `vTaskDelay(pdMS_TO_TICKS(50))` on the ESP32 and `time.sleep(0.05)` on the Jetson. Without this delay, the UART hardware doesn't stabilize and the PING is interpreted as garbage.

### 5.4 Observation Dump Protocol

The observation dump transfers binary sensor data from the ESP32's PSRAM ring buffer to the Jetson:

```
ESP32 → Jetson: OBS_DUMP_HEADER (total_size, num_chunks, crc32_expected)
Jetson → ESP32: OBS_DUMP_ACK (ready_for_chunk: 0)
ESP32 → Jetson: OBS_DUMP_CHUNK (chunk_index, data[1024 bytes], crc16)
Jetson → ESP32: OBS_DUMP_ACK (ready_for_chunk: 1)
... repeat until all chunks sent ...
ESP32 → Jetson: OBS_DUMP_END (crc32_actual)
Jetson: Verify CRC-32. If mismatch, request retransmit of failed chunks.
```

The dump uses a 1024-byte chunk size to stay within the COBS frame limit (1051 bytes max). CRC-32 verification covers the entire concatenated payload. Failed chunks can be individually retransmitted up to 3 times.

### 5.5 OTA Firmware Update

OTA firmware updates use chunked transfer with SHA-256 verification:

```
Jetson → ESP32: FIRMWARE_UPDATE_START (total_size, sha256_hash, num_chunks)
ESP32 → Jetson: FIRMWARE_UPDATE_ACK (ready)
Jetson → ESP32: FIRMWARE_UPDATE_CHUNK (chunk_index, data[512 bytes], crc16)
ESP32 → Jetson: FIRMWARE_UPDATE_CHUNK_ACK (chunk_index)
... repeat for all chunks ...
ESP32: Write to inactive OTA slot (ota_0 or ota_1)
ESP32: Verify SHA-256 of complete image
ESP32 → Jetson: FIRMWARE_UPDATE_RESULT (success/failure, sha256_actual)
Jetson: If success → mark new slot as bootable
ESP32: Reboot into new firmware
ESP32: If new firmware fails health check within 30s → rollback to previous slot
```

**Canary deployment (v2.5):** The OTA rollout strategy uses canary deployment: 10% of nodes get the new firmware first, then 50%, then 100%. If any canary node reports a safety event within the first hour, the rollout is automatically halted and remaining nodes stay on the previous version.

## 6. Safety System Implementation

### 6.1 The 4-Tier Architecture in Code

The four safety tiers map to specific code modules:

| Tier | Code Module | Mechanism | Response Time |
|------|------------|-----------|---------------|
| **Tier 1: Hardware** | External circuits | Kill switch NC contact, MAX6818 WDT, PTC polyfuses | <1ms |
| **Tier 2: Firmware Guard** | `estop_isr.c`, `overcurrent_isr.c`, `safety_guard.c` | GPIO ISR, output clamping, rate limiting | <10ms |
| **Tier 3: Supervisory** | `safety_supervisor.c`, `watchdog_feed.c` | FreeRTOS task, heartbeat monitor, task watchdog | <100ms |
| **Tier 4: Application** | `vm_tick_task`, reflex execution | PID loops, reflex execution, AI inference | <50ms |

### 6.2 ISR Implementation Rules

Every ISR in the safety system MUST follow these rules:

```c
void IRAM_ATTR estop_isr_handler(void* arg) {
    // RULE 1: Marked IRAM_ATTR — code lives in internal RAM, not flash
    // RULE 2: No float operations (FPU not available in ISR context)
    // RULE 3: No print/logging (touches flash)
    // RULE 4: No blocking operations (no delays, no mutex, no queue with timeout)
    // RULE 5: Total execution < 1ms

    BaseType_t xHigherPriorityTaskWoken = pdFALSE;

    // Immediately drive all outputs to safe state
    for (int i = 0; i < num_actuators; i++) {
        gpio_set_level(actuator_pins[i].gpio, actuator_pins[i].safe_value);
        if (actuator_pins[i].pwm_channel >= 0) {
            ledc_set_duty(actuator_pins[i].pwm_channel, 0);
            ledc_update_duty(actuator_pins[i].pwm_channel);
        }
    }

    // Notify deferred handler via ISR-safe semaphore
    xSemaphoreGiveFromISR(kill_switch_sem, &xHigherPriorityTaskWoken);
    portYIELD_FROM_ISR(xHigherPriorityTaskWoken);
}
```

**Allowed in ISR:** `gpio_set_level()`, `ledc_set_duty()`, `xSemaphoreGiveFromISR()`, `xQueueSendFromISR()`, `portYIELD_FROM_ISR()`, direct register writes (`GPIO.out_w1ts`).

**NOT allowed:** `printf()`, `ESP_LOGx()`, `esp_timer_get_time()`, `xQueueSend()`, `nvs_get_*()`, `cJSON_*()`, any function not marked `IRAM_ATTR`.

### 6.3 The Safety Guard API

Every actuator write goes through the safety guard API:

```c
int safety_write_actuator(uint8_t channel, float value) {
    // Check 1: Is safety system in normal mode?
    if (safety_state != MODE_NORMAL) return ERR_SAFETY_NOT_NORMAL;

    // Check 2: Is the channel enabled?
    if (!channel_enabled[channel]) return ERR_CHANNEL_DISABLED;

    // Check 3: Rate limiting (prevent control oscillation)
    uint32_t now = xTaskGetTickCount() * portTICK_PERIOD_MS;
    if (now - last_write_time[channel] < min_write_interval_ms[channel]) {
        return ERR_RATE_LIMITED;
    }

    // Check 4: Clamping to configured min/max
    float clamped = MAX(actuator_config[channel].min_val,
                   MIN(value, actuator_config[channel].max_val));

    // Apply to hardware
    gpio_set_level(actuator_pins[channel].gpio, (int)clamped);
    last_write_time[channel] = now;
    last_write_value[channel] = clamped;

    return IO_OK;
}
```

### 6.4 The Supervisory Task

The safety supervisor runs every 10ms at the highest FreeRTOS task priority (below ISRs):

```c
void safety_supervisor_task(void *arg) {
    TickType_t last_wake = xTaskGetTickCount();

    while (true) {
        vTaskDelayUntil(&last_wake, pdMS_TO_TICKS(10));

            log_safety_event(HEAP_LOW, free_heap);
        }

        // 4. Safety state machine management
        safety_state_machine_update();

        // 5. Feed hardware watchdog (every 200ms, via separate task)
        // (handled by watchdog_feed_task)
    }
}
```

### 6.6 Testing Safety: The HIL Test Bench

The HIL test bench provides fault injection capabilities:

- **Kill switch injection:** Physical relay simulates kill switch press. Oscilloscope measures response time (<1ms required).
- **Heartbeat loss simulation:** Jetson stops sending heartbeats. ESP32 should enter DEGRADED at 500ms, SAFE_STATE at 1000ms.
- **Overcurrent injection:** Programmable load ramps above threshold. ESP32 should disable the channel within 2ms.
- **Baud negotiation stress:** 1000 rapid baud switch cycles. No UART lockups allowed.
- **OTA failure injection:** Corrupt firmware chunks. ESP32 must detect SHA-256 mismatch and reject the update.

### 7.2 Formal Model Verification for Reflexes

Since v2.0, every deployed reflex bytecode program is verified against a formal model before deployment. The verification checks:

1. **Stack safety:** No possible execution path can overflow or underflow the 256-entry stack
2. **Cycle budget:** Worst-case execution time (WCET) is within the 10,000-cycle budget
3. **Actuator bounds:** All actuator writes are covered by clamping operations
4. **Termination:** No infinite loops (all JUMP targets verified to eventually reach HALT)
5. **No undefined behavior:** No division by zero (explicit checks before DIV_F), no NaN propagation

The formal verifier is implemented in `jetson/compiler/reflex_verifier.py` using a symbolic execution engine. It traces every possible execution path through the bytecode and checks the above invariants. Verification typically takes 200-500ms per reflex — fast enough to run at deploy time.

### 7.3 Fuzz Testing

The COBS decoder is fuzz-tested with 10M random inputs using AFL++ (American Fuzzy Lop). The fuzzer generates random byte sequences and feeds them to the COBS decoder, checking for:
- No buffer overflows (ASAN enabled)
- No assertion failures
- Correct output for all valid inputs
- Graceful handling of malformed inputs (return error, don't crash)

All edge cases are covered, including: all-zeros input, all-0xFF input, maximum-length payload (1051 bytes), minimum-length payload (5 bytes), and every permutation of the 254-byte block boundary.

### 7.4 Coverage Requirements

| Subsystem | Required Coverage | Current Coverage |
|-----------|------------------|-----------------|
| Safety code (Tier 1-3) | 90% | 94.2% |
| VM core + dispatch | 90% | 91.8% |
| Protocol (COBS, CRC, dispatch) | 85% | 87.5% |
| I/O drivers | 80% | 82.1% |
| Application code (Jetson) | 80% | 83.6% |

Coverage is measured by `gcov` for C and `coverage.py` for Python. The CI pipeline fails if coverage drops below the requirement.

## 8. Release Process

### 8.1 Semantic Versioning for NEXUS

NEXUS follows semantic versioning with domain-specific rules:

- **MAJOR (X.0.0):** Breaking wire protocol changes, VM ISA changes, safety architecture changes. Requires fleet-wide coordinated upgrade. Must include migration guide.
- **MINOR (X.Y.0):** New features, new opcodes (backward compatible), new Jetson modules. Can be deployed node-by-node. No wire protocol changes.
- **PATCH (X.Y.Z):** Bug fixes, performance improvements, documentation. Zero-downtime deployable via OTA.

### 8.2 Pre-Release Checklist

Before every release, the following must pass:

1. **Safety tests:** All 14 safety validation tests (see Safety Validation Playbook) pass on physical hardware
2. **HIL test bench:** 45-minute automated test sequence passes with zero failures
3. **Memory analysis:** `idf.py size-components` shows firmware < 500KB, heap usage within budget
4. **Timing verification:** VM tick worst-case execution time < 700µs (leaving 300µs margin)
5. **Formal verification:** All deployed reflexes pass the reflex verifier
6. **Coverage check:** All subsystems meet coverage requirements
7. **OTA rollback test:** Flash new firmware, verify it boots, trigger rollback, verify previous firmware restores
8. **Wire compatibility:** v1.0 firmware communicates with v3.1 Jetson software (the compatibility promise)

### 8.3 OTA Rollout Strategy

The OTA rollout uses canary deployment:

| Phase | Percentage | Duration | Abort Condition |
|-------|-----------|----------|----------------|
| Canary | 10% of nodes | 1 hour | Any safety event, any crash |
| Early majority | 50% of nodes | 6 hours | Safety event rate > 0.1% |
| Full rollout | 100% of nodes | 24 hours | Any critical regression |

If the rollout is aborted, remaining nodes stay on the previous version. Affected nodes automatically roll back if they fail health checks within 30 seconds of booting the new firmware.

### 8.4 Post-Release Monitoring

The first 72 hours after release are closely monitored:

- **Safety event rate:** Should be identical to pre-release baseline
- **Crash rate:** Zero crashes expected (any crash triggers automatic rollback)
- **Heartbeat loss rate:** Should be < 0.01% (brief Jetson reboots are acceptable)
- **Serial link error rate:** CRC mismatch rate should be < 0.001%
- **Memory usage trend:** No heap growth (indicates memory leak)

If any metric exceeds its threshold, the release is halted and the previous version is restored.

## 9. Contributing

### 9.1 Branch Naming Convention

```
feature/descriptive-name    # New features
fix/descriptive-name        # Bug fixes
safety/descriptive-name     # Safety system changes (triggers additional review)
refactor/descriptive-name   # Code refactoring
docs/descriptive-name       # Documentation
```

### 9.2 PR Template

Every PR must include:

1. **Description:** What changed and why
2. **Testing:** How the change was tested (unit tests, HIL bench, manual)
3. **Safety impact:** Does this PR touch safety code? (Yes/No)
4. **Wire compatibility:** Does this change break backward compatibility?
5. **Performance impact:** CPU, memory, latency measurements
6. **Checklist:** All items from `Code_Review_Checklist.md` addressed

### 9.3 The "Safety-First" Review Rule

Any PR that touches files in `components/safety/` or `safety_policy.json` requires **2 approvals** from team members who have completed the safety reviewer training. Additionally:

- The HIL test bench must pass with the changed code
- All 14 safety validation tests must pass
- The formal reflex verifier must still accept all currently deployed reflexes
- The CI pipeline runs an extended safety test suite (additional 30 minutes of fault injection)

This rule has been non-negotiable since v1.1 and has caught 4 safety regressions before they reached production.
