# NEXUS Platform — Terms and Definitions v3.1

**Version:** 3.1.0 | **Last Updated:** 2026-03-29  
This glossary defines every technical term used in NEXUS documentation. Terms are organized alphabetically. Each entry includes a category, a plain-English definition, optional technical details, and a real-world example. Cross-references point to related terms.

---

**A/B Test** — *Category: Learning*  
A comparison between how you handle a situation and how the system's proposed reflex would handle it. The system runs the reflex in "shadow mode" (recording its outputs without actually controlling anything) alongside your manual operation, then compares the results. Think of it as a test drive before buying.

*Example:* After you demonstrate how you steer in cross-winds, the system proposes a wind-compensation reflex and runs an A/B test. It shows you: "In the 30-minute test, your average heading error was 3.1 degrees. The reflex would have achieved 2.2 degrees. Fuel savings estimated at 6%."

*See also:* Observation, Proposal, Reflex

---

**Actuator Register** — *Category: Firmware*  
A numbered slot in the limb node's memory where a control output value is stored. When a reflex writes a value to an actuator register (for example, "rudder angle = 15 degrees"), the firmware reads that value and sends it to the physical actuator. There are 64 actuator registers per node.

*See also:* Sensor Register, Reflex, Bytecode VM

---

**Autonomy Level** — *Category: Safety & Autonomy*  
A number from 0 to 5 that describes how much control the system has over a specific subsystem. Level 0 means you do everything manually. Level 5 means the system handles everything including maintenance. Each subsystem (steering, throttle, bilge, etc.) has its own independent level. The level can only increase when the trust score is high enough and you approve.

*See also:* INCREMENTS, Trust Score, Gated Trust

---

**Behavioral Clustering** — *Category: Learning*  
A pattern discovery algorithm that groups similar segments of your behavior into categories. For example, it might identify that your steering behavior falls into distinct clusters: "cruising," "docking," "rough-weather maneuvering," and "fishing pattern." These clusters help the system understand that different situations require different responses.

*See also:* Cross-Correlation, Observation, Learning Loop

---

**Bootstrap Agent** — *Category: AI*  
The initial AI module that runs on the Jetson when the system first boots with no prior data. It handles basic tasks (sensor reading, heartbeat monitoring, chat) before any specialized reflexes are deployed. Think of it as the system's "innate" behavior before it has learned anything from you.

---

**Bytecode** — *Category: Firmware*  
A compact, efficient set of instructions that tells the limb node's VM exactly how to perform a reflex behavior. Bytecode is generated by the Jetson from a human-readable JSON description and sent to the limb node. Unlike raw code, bytecode is safe, bounded, and deterministic — the VM can execute it at 1,000 times per second without crashing.

*See also:* Bytecode VM, Reflex Deploy, Universal Binary

---

**Bytecode VM** — *Category: Firmware*  
The "virtual machine" running on each limb node that executes bytecode instructions. It is a tiny, self-contained computer within the ESP32 that performs math, reads sensors, and writes actuators according to the bytecode program. It uses a fixed 3 KB of memory, can never exceed 10,000 computation cycles per tick, and enforces strict safety rules. If it violates any rule (stack overflow, division by zero, too many cycles), it immediately halts and puts all actuators in their safe positions.

*See also:* Bytecode, Reflex, HALT

---

**CLAMP_F** — *Category: Firmware*  
A bytecode instruction that constrains a value within a minimum and maximum range. For example, `CLAMP_F -35 35` ensures a rudder angle value never goes below negative 35 or above positive 35 degrees. Every actuator output passes through a CLAMP_F before reaching the physical hardware, guaranteeing that no reflex can command an actuator beyond its safe range.

*See also:* Bytecode VM, Safe-State

---

**Cloud Offloading** — *Category: Architecture*  
Sending heavy computational tasks from the Jetson (the brain) to remote servers via the internet. The Jetson handles real-time tasks locally, but complex tasks like training new reflexes or running large simulations are offloaded to cloud computers with more processing power. Uses queue-and-forward semantics, so a temporary internet outage does not cause problems — tasks wait in a queue until connectivity is restored.

*See also:* Jetson, Learning Loop

---

**COBS** (Consistent Overhead Byte Stuffing) — *Category: Protocol*  
A method for packaging data so the receiving end can always tell where one message ends and the next begins. Think of it as putting each message in a clearly labeled envelope. COBS adds less than 1% overhead, making it far more efficient than older methods for sending large amounts of sensor data.

*See also:* UART, Serial Link

---

**Command** — *Category: System*  
An instruction you give to the system through voice, text, or the dashboard. Commands are distinct from intents: a command is an immediate action ("take over steering"), while an intent is a goal that the system works toward over time ("when the wind exceeds 25 knots, reduce throttle").

*See also:* Intent, Override, Proposal

---

**Configuration JSON** — *Category: System*  
A text file that describes how a limb node is wired and what it should do. It lists every pin assignment (which GPIO connects to which sensor or actuator), safety limits, telemetry settings, and reflex parameters. The configuration is sent from the Jetson to the limb node at boot. You never edit this file directly — the provisioning wizard generates it from your natural-language description.

*See also:* Role Assignment, Provisioning Wizard

---

**Cross-Correlation** — *Category: Learning*  
A mathematical technique that discovers time-lagged relationships between sensors and actuators. For example, cross-correlation might reveal that "when the wind speed increases by 5 knots, the captain increases rudder angle by 8 degrees, approximately 2 seconds later." These relationships form the basis for reflex proposals.

*See also:* Behavioral Clustering, Observation, Learning Loop

---

**Delta Encoding** — *Category: Data*  
A data compression technique introduced in v3.1 that stores only the *changes* in sensor values rather than the full values at every reading. If a temperature sensor reads 22.1°C, then 22.1°C, then 22.2°C, delta encoding stores [22.1, 0.0, 0.1] instead of [22.1, 22.1, 22.2]. This reduces data transmission bandwidth by 5-8x, making cloud sync practical over satellite links.

*See also:* Observation, Cloud Offloading

---

**Demonstration** — *Category: Learning*  
A session where you operate your equipment while the system records sensor data and your narration. Demonstrations are the primary way NEXUS learns. The system analyzes your behavior during demonstrations to discover patterns and propose reflexes. The quality of your demonstrations directly determines the quality of the resulting reflexes.

*See also:* Observation, Learning Loop, Proposal

---

**E-Stop** (Emergency Stop) — *Category: Safety*  
The immediate, unconditional shutdown of all actuators on a limb node. Triggered by the kill switch (hardware), a software command, or a safety event detection. Response time is under 1 millisecond from trigger to all actuators in their safe positions. The E-Stop interrupt handler runs at the highest priority on the ESP32 and cannot be blocked by any other software.

*See also:* Kill Switch, Safe-State, HALT

---

**Gated Trust** — *Category: Safety & Autonomy*  
The principle that the system's autonomy level is capped by two separate limits: the trust score (earned through performance) and the maximum allowed level (set by you). Even if the trust score qualifies for Level 4, you can set the maximum to Level 2 if you are not comfortable. The system will never exceed the lower of these two limits.

*See also:* Autonomy Level, Trust Score

---

**HALT** — *Category: Firmware*  
A bytecode instruction that immediately stops the VM execution and places all actuators in their safe positions. HALT is triggered when the VM detects a safety violation: stack overflow, cycle budget exceeded, or an invalid memory access. Unlike a crash, HALT is a controlled, predictable stop — the VM records what went wrong and reports it to the Jetson.

*See also:* Bytecode VM, E-Stop, Safe-State

---

**Heartbeat** — *Category: Protocol*  
A regular "I'm alive" message sent from the Jetson to each limb node every 100 milliseconds. If a limb node misses 5 consecutive heartbeats (500 milliseconds), it enters degraded mode. If it misses 10 (1 second), it enters safe-state. The heartbeat is the system's pulse — if it stops, something is wrong and the limbs protect themselves.

*See also:* Safe-State, Watchdog, RS-422

---

**Hot-Swap** — *Category: Operations*  
Replacing a failed limb node without shutting down the system or using a laptop. In v3.1, you simply unplug the failed node, plug in a fresh one, and wait 30-60 seconds for the Jetson to automatically detect it, assign it the same role, and send the cached configuration. No firmware flashing, no manual configuration.

*See also:* Limb Node, Role Assignment

---

**I2C** (Inter-Integrated Circuit) — *Category: Hardware*  
A two-wire communication bus used to connect sensors (compasses, temperature sensors, pressure sensors, etc.) to limb nodes. I2C is popular because it requires only two wires (data and clock) and supports multiple sensors on the same bus, each with a unique address. NEXUS supports I2C speeds up to 1 MHz.

*See also:* UART, Sensor Register

---

**INCREMENTS** — *Category: Safety & Autonomy*  
The NEXUS autonomy framework (Incremental Autonomy for Safety-Critical Systems). It defines six levels of autonomy (0-5), a trust score algorithm, advancement criteria, and fallback rules. The name reflects the core philosophy: autonomy is earned incrementally, one proven step at a time.

*See also:* Autonomy Level, Trust Score, Gated Trust

---

**Intent** — *Category: Learning*  
A goal you describe in natural language that the system works toward over time. Unlike a command (immediate action), an intent is a behavioral target: "When the bilge water level exceeds 65%, start the pump within 30 seconds." The system records observations, discovers patterns, and eventually proposes a reflex that implements your intent.

*See also:* Command, Demonstration, Proposal

---

**Jetson** — *Category: Hardware*  
The NVIDIA Jetson Orin Nano Super computer board that serves as the NEXUS "brain." It handles AI inference, natural language understanding, pattern discovery, reflex compilation, and the user interface. The Jetson communicates with limb nodes via RS-422 serial links and can connect to the cloud via WiFi, Ethernet, or satellite.

*See also:* Limb Node, Cloud Offloading, Cognitive Layer

---

**JSON State Machine** — *Category: Firmware*  
A human-readable text format that describes a reflex behavior. The JSON format specifies the reflex name, version, sensor inputs, actuator outputs, PID controller parameters, safety limits, and the bytecode-like instruction sequence. The Jetson compiles JSON state machines into bytecode before deploying them to limb nodes.

*See also:* Bytecode, Reflex Deploy

---

**Kill Switch** — *Category: Safety*  
A physical, mushroom-head, red button that cuts electrical power to all actuators. It operates independently of all software — even if the entire computer system crashes, the kill switch still works because it is wired directly in the power circuit. It is the ultimate safety device. Press it whenever you feel the need. There is no wrong time to press it.

*See also:* E-Stop, Safe-State

---

**Learning Loop** — *Category: Learning*  
The continuous cycle through which NEXUS acquires new behaviors: (1) Observe your actions during a demonstration, (2) Record sensor data and narration, (3) Analyze patterns using algorithms like cross-correlation and behavioral clustering, (4) Synthesize a reflex proposal, (5) A/B test the proposal against your behavior, (6) You approve or reject, (7) If approved, deploy and monitor. The loop repeats continuously, with each cycle improving the system's capabilities.

*See also:* Demonstration, Observation, Proposal, A/B Test

---

**LittleFS** — *Category: Firmware*  
A lightweight file system used on the limb node's flash memory to store reflex JSON files, compiled bytecode, and configuration data. LittleFS is designed for embedded systems — it handles power loss gracefully (no corruption) and wears evenly across flash memory cells to maximize lifespan.

*See also:* Configuration JSON, Bytecode, Universal Binary

---

**MOSFET** — *Category: Hardware*  
A type of electronic switch used in NEXUS to control actuators (motors, pumps, valves, relays) from the limb node's low-voltage GPIO pins. MOSFETs can switch large currents (amps) using tiny control signals (millivolts). In NEXUS, all MOSFET gates have 10K-ohm pull-down resistors, so if the limb node loses power, the MOSFETs default to the off (safe) position.

*See also:* Polyfuse, Safe-State

---

**MQTT** — *Category: Protocol*  
A lightweight messaging protocol used for telemetry data between the Jetson and external systems (cloud, dashboards, mobile apps). MQTT organizes messages into "topics" (e.g., `nexus/telemetry/node1`) and supports quality-of-service levels from "fire and forget" to "guaranteed delivery." Safety-critical messages like override commands use the highest delivery guarantee.

*See also:* Telemetry, Cloud Offloading

---

**Node** — *Category: Hardware*  
Short for "limb node." An ESP32-S3 microcontroller board that directly interfaces with sensors and actuators. Each node runs the NEXUS universal firmware and executes reflex bytecode independently. A typical NEXUS installation has 1-8 nodes, each controlling a specific subsystem (steering, throttle, bilge, etc.).

*See also:* Jetson, Limb Node, Hot-Swap

---

**Observation** — *Category: Learning*  
A single timestamped record of all sensor readings and actuator states, captured at a specific moment. At 100 Hz (100 times per second), the system generates 8,640,000 observations per day per node. Observations are stored in the observation buffer and periodically analyzed by the pattern discovery engine.

*See also:* Observation Session, Demonstration, Delta Encoding

---

**Override** — *Category: Operations*  
A manual intervention where you take control of a subsystem away from the system. Overrides can be triggered by voice ("take over steering"), by grabbing a physical control (turning the steering wheel), or by pressing a dashboard button. Every override is logged and affects the trust score. Frequent overrides signal that the system is not performing well enough for its current autonomy level.

*See also:* Gated Trust, Trust Score, Command

---

**PID Controller** — *Category: Firmware*  
A standard control algorithm (Proportional-Integral-Derivative) used in many NEXUS reflexes. A PID controller calculates how much correction to apply based on three factors: the size of the error (P), how long the error has persisted (I), and how fast the error is changing (D). For example, a steering PID controller applies more rudder when the heading error is large, adds correction for persistent drift, and dampens oscillation.

*See also:* Reflex, Bytecode VM

---

**Polyfuse** — *Category: Hardware*  
A self-resetting fuse that protects actuator circuits from overcurrent damage. Unlike a traditional fuse that blows permanently, a polyfuse increases its resistance when too much current flows, then resets itself after cooling down. In NEXUS, polyfuses are wired in series with each actuator output as a hardware-level backup to the software current monitoring.

*See also:* MOSFET, Overcurrent, Safe-State

---

**Proposal** — *Category: Learning*  
A suggested reflex behavior generated by the AI based on pattern analysis of your observation data. Proposals are presented in plain language with supporting evidence (number of observations, consistency percentage, expected improvement). You can approve, reject, test (A/B), or ask for more details. Approved proposals become active reflexes.

*See also:* Demonstration, A/B Test, Reflex Deploy

---

**PSRAM** — *Category: Hardware*  
External memory chip on the ESP32-S3 limb node, providing 8 MB of additional storage. PSRAM is used primarily for the observation buffer (recording sensor data at high speed). The limb node's internal SRAM (512 KB) is too small for long observation sessions, so PSRAM acts as a fast, temporary data store.

*See also:* Node, Observation

---

**Reflex** — *Category: System*  
An automated behavior that runs on a limb node. Reflexes are compiled from your demonstrations and approved proposals into bytecode, then deployed to the limb node's VM. A reflex reads sensor registers, performs calculations (using the bytecode VM), and writes actuator registers at a fixed rate (typically 10-100 Hz). Reflexes are deterministic, safe, and independent of the Jetson — they continue running even if the brain goes offline.

*See also:* Bytecode VM, Reflex Deploy, Learning Loop

---

**Reflex Deploy** — *Category: System*  
The process of sending a compiled bytecode reflex from the Jetson to a limb node. The Jetson compiles the JSON state machine into bytecode, validates it (checking for safety violations, memory bounds, and timing constraints), signs it with a cryptographic signature, and sends it over the serial link. The limb node verifies the signature, stores the bytecode in LittleFS, and begins executing it.

*See also:* Bytecode, LittleFS, JSON State Machine

---

**Role Assignment** — *Category: System*  
The process by which the Jetson tells each limb node what it should do. A role includes the node's I/O pin configuration, sensor types, actuator types, safety limits, telemetry settings, and the reflexes it should run. Roles are assigned at boot and can be changed at runtime. This is how NEXUS achieves hot-swap: a replacement node receives the same role as the failed one.

*See also:* Configuration JSON, Hot-Swap, Provisioning Wizard

---

**RS-422** — *Category: Hardware*  
A robust serial communication standard used for the data link between the Jetson and limb nodes. RS-422 uses differential signaling (two wires carrying opposite-polarity signals) which makes it highly resistant to electrical noise — ideal for the marine and industrial environments where NEXUS operates. Standard Cat-5e Ethernet cables with RJ-45 connectors are used. Maximum cable length: 10 meters at 921,600 baud.

*See also:* UART, COBS, Heartbeat

---

**Safe-State** — *Category: Safety*  
A condition where all actuators are in their safest positions and all reflexes are suspended. Safe-state is entered when the kill switch is pressed, when heartbeat communication is lost, when a safety event is detected, or when you command it. In safe-state, the system monitors sensors and communicates but does not move anything. The system never auto-resumes from safe-state — you must explicitly restore operation.

*See also:* Kill Switch, E-Stop, HALT

---

**Safety Event** — *Category: Safety*  
Any occurrence that triggers a safety response: kill switch activation, heartbeat loss, overcurrent detection, sensor failure, watchdog timeout, or a reflex safety violation. All safety events are logged with high precision timestamps and full diagnostic data. Serious safety events cause an immediate autonomy level reduction.

*See also:* Safe-State, Watchdog, Trust Score

---

**Self-Test** — *Category: Firmware*  
A series of automated checks that each limb node runs at boot to verify its hardware is functioning correctly. Tests include flash memory read/write verification, I2C bus scan, ADC calibration check, GPIO continuity test, and PSRAM integrity check. Self-test results are reported to the Jetson. If any test fails, the node enters a fault state and does not accept reflex deployments.

*See also:* Boot Sequence, Role Assignment

---

**Sensor Register** — *Category: Firmware*  
A numbered slot in the limb node's memory where a sensor reading value is stored. Before each reflex execution tick, the firmware reads all connected sensors and writes their values into the sensor registers. The bytecode VM reads from these registers to make decisions. There are 64 sensor registers per node.

*See also:* Actuator Register, Bytecode VM

---

**SYSCALL** — *Category: Firmware*  
A pseudo-instruction in the bytecode VM that performs extended operations beyond basic math and I/O. SYSCALLs include CALL/RET (function calls), PID_COMPUTE (PID controller step), RECORD_SNAPSHOT (save diagnostic data), and EMIT_EVENT (send a message to the Jetson). SYSCALLs are implemented by setting the SYSCALL flag in the flags byte of a NOP instruction.

*See also:* Bytecode VM, Bytecode

---

**Telemetry** — *Category: System*  
The stream of sensor data sent from limb nodes to the Jetson for monitoring, display, and analysis. Telemetry is sent at a configurable rate (typically 10-100 Hz per sensor) using a compact binary format that is 8x smaller than JSON. Telemetry data is used for the dashboard display, cloud sync, and observation recording.

*See also:* MQTT, Observation, Sensor Register

---

**Trust Score** — *Category: Safety & Autonomy*  
A continuous number between 0.0 and 1.0 that measures how reliably the system has performed for a specific subsystem. Trust is gained slowly (0.002 per evaluation) and lost quickly (0.05 per bad event) — a 25:1 loss-to-gain ratio that matches human psychology. The trust score determines when a subsystem is eligible for advancement to the next autonomy level. Safety-critical subsystems use a higher loss ratio than low-risk ones.

*See also:* Autonomy Level, INCREMENTS, Gated Trust

---

**UART** (Universal Asynchronous Receiver-Transmitter) — *Category: Hardware*  
The serial communication hardware on the ESP32-S3 used for the RS-422 data link to the Jetson. UART converts parallel data from the processor into a serial bit stream for transmission and vice versa. NEXUS uses UART at 921,200 bits per second (configurable down to 115,200 for longer cable runs).

*See also:* RS-422, COBS, Serial Link

---

**Universal Binary** — *Category: Firmware*  
The single firmware file (~320 KB) that runs on every ESP32-S3 limb node in the system. All nodes run identical firmware; their specific behavior is determined by the role assignment received from the Jetson at boot. This design enables hot-swap (any node can replace any other) and simplifies inventory management (one firmware image to test and sign).

*See also:* Node, Role Assignment, Hot-Swap

---

**VM** (Virtual Machine) — *Category: Firmware*  
See: Bytecode VM.

---

**Watchdog** — *Category: Safety*  
A hardware timer (MAX6818 chip) that resets the limb node's processor if the firmware stops responding. The watchdog must be "kicked" (reset) by the firmware every 200 milliseconds using an alternating 0x55/0xAA pattern. If the firmware gets stuck in an infinite loop, crashes, or the processor hangs, the watchdog will not be kicked and will reset the processor within 1 second. This is a hardware-level safety mechanism that the software cannot disable.

*See also:* Safe-State, Heartbeat, E-Stop
